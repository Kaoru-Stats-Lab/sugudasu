## 0. 最上位の参照元：SUGUDASU の約束

本機能は https://sugudasu.com/statements に定義された「SUGUDASU の約束（7つの設計原則）」の**下位仕様**として設計する。矛盾が生じた場合、statementsの原則が常に優先される。

### 7つの設計原則（既存・引用）

1. 登録不要 — 開いたら、すぐ使えます
2. 入力データは、原則ブラウザ内で処理します
3. 静的配信 — 重いサーバーに頼りません
4. 実務3分 — 1 URL・1用途
5. Copy-first — 結果はSlack・Excel・PDFへ
6. 帳票と証跡 — 印刷・PDFに価値を置きます
7. 過剰な断定はしません

### ⚠️ Roomがstatementsと緊張関係を持つ点（要・明示的整理）

既存の「あえてやらないこと」には次の記載がある。

> **アカウント必須のクラウド同期** — 班分け等はセッションJSON + 同期メモ（Google Keep等）でPC→スマホ

Roomは有料版Syncの機能であり、複数人・複数端末でのリアルタイム同期を必須要件とする。これは一見、上記の「あえてやらないこと」と矛盾する。**この矛盾は以下の理由で解消可能と判断する。**

| 論点 | 整理 |
|---|---|
| 「アカウント」とは何か | statementsが拒否しているのは、メール認証・OAuth・SSOのような**身元確認を伴う会員登録**（原則1）。Roomは、URL #fragmentに埋め込まれたSecret Key（1Password式2SKD）による**知識ベースの一時参加**であり、身元登録を一切要求しない。よって原則1には抵触しない |
| 「クラウド保存」とは何か | statementsが拒否しているのは、名簿等の**恒久的なクラウド保存**。Roomが保存するのは、Room生存中のみ存在し、全員退出で即削除される**一時的な暗号化Blob**。「保存しないことをデフォルトとする」という設計思想（第2原則の精神）とは整合する |
| 帳票・証跡（第6原則）との関係 | Roomは意図的に「何も残さない」ことを価値とする、statements上の他ツールとは異なるカテゴリの機能である。この違いをRoomのUI上でも明示し、「これは請求書や証跡PDFを作るツールではない」と誤解を防ぐ |

**Cursorへの指示**：この整理をRoomのランディングページ・ヘルプ文言に**必ず反映**すること。「アカウント不要」「保存しない」を謳う際は、上記の定義（身元登録なし／恒久保存なし）に基づいた表現に統一し、statementsの「過剰な断定はしません」（原則7）に従い、「完全に安全」「100%痕跡が残らない」のような言い切りは使わない。

### 正直な但し書き（statements準拠）

Roomは以下の通信が発生する。これをRoom内のヘルプ文言でも明示すること（statementsの「正直な但し書き」の精神を踏襲）。

- ページ配信（HTML/CSS/JS）
- WebRTCのシグナリング（Supabase経由、接続確立のためだけの通信）
- 暗号化済みBlobのUPSERT（Sync契約時のみ、内容はサーバーで復号不可）
- （TURN中継を使う場合）暗号化された通信の中継。中継サーバーも復号鍵を持たない

## 1. プロダクト名

**SUGUDASU 付箋ルーム**

英語表記併記が必要な箇所のみ `SUGUDASU Room` を使用可とする。

### 却下した案（再検討不要）

- 「SUGUDASU Room」単独：一般語すぎてビデオ会議サービスと混同されやすい
- 「消えるホワイトボード」：v1に自由描画機能がなく、実体と乖離した過大な期待を生む
- 「ブレストルーム」：用途がブレストに限定され、座席配置・配属検討等を裏切る
- 「付箋会議」：意味不明瞭で離脱要因になる

備考：「Room」は一般語のためSEO上の弱さがある。ページタイトル・ドメイン表記では必ず「SUGUDASU」を前置する運用を徹底する。

---

## 2. プロダクト思想

> **付箋ルームは、複数人でその場に集まり、付箋を配置しながら合意形成するための、記録を残さないツール。**

### 既存クラウドサービスとの違い

一般的なSaaSは「入力→保存→履歴→検索→蓄積」で価値を作る。SUGUDASU Roomは逆で「集まる→考える→配置する→合意する→閉じる」を価値とする。終われば部屋は消える。**必要なのは記録ではなく合意**。

補足整理：

- 成果物は「決定事項の文書」ではなく「配置された付箋の最終状態（合意の形）」
- Slack＝時系列、Notion＝文書、付箋ルーム＝**位置**、という整理をUI・コピーの判断基準とする
- 「共通認識で終わる」ユースケースと「合意で終わる」ユースケースの両方を許容し、無理に一方へ寄せない

### キャッチコピー（開発思想・社内向け）

> 保存するクラウドではない。考えるためのクラウド。

### デザイン原則

Roomはチャットではない。Docsでもない。Dropboxでもない。目的は「会議室のホワイトボード」をWebへ持ってくること。ファイル管理機能は作らない。

## 4. 非ゴール（作らないもの）

Slack／Teams／Google Docs／Dropbox／Notion／画像共有／PDF共有／動画共有／音声通話／画面共有／永続ストレージ／ファイル転送全般。

加えて、v1では以下を非ゴールとして明示する。

- 付箋間の関係線・矢印・親子階層（MindMap的機能全般）
- チャットスレッド・メッセージログ・既読機能

理由：statements第3原則（静的配信・重いサーバーに頼らない）と、Room固有の「引き算優先」の設計原則に基づく。

## 5. システム構成

```
Browser
  ↓ WebRTC DataChannel
P2P同期（カード位置・テキスト・カーソル等）
  ↓
Supabase（Signalingのみ、暗号化Blobの一時UPSERT）
```

サーバー（Supabase）の役割は「暗号化された塊を右から左に流すだけの土管」に徹する。TTL管理・シグナリング以上のロジックは持たせない。

## 6. 技術方針：保存対象

### 同期する（○）

```json
{
  "cardId": "string",
  "x": "number",
  "y": "number",
  "text": "string",
  "color": "string",
  "updatedAt": "number (unix timestamp ms)"
}
```

加えて：色、カーソル位置、Presence（誰が今Roomにいるか）。

### 同期しない（×）

画像、PDF、動画、音声、Officeファイル、添付ファイル全般。

理由：SUGUDASUの「軽さ」という価値と、statements第3原則（静的配信・重いサーバーに頼らない）に基づく。WebRTC DataChannelでのファイル転送はチャンク分割・再送制御の実装コストが高く、投資対効果が低い。

---

## 7. 🔴 技術的に必ず解決すべき2点（未解決のまま実装しない）

### 7-1. TURNサーバー方針

企業ネットワーク（プロキシ・ファイアウォール配下）では、STUNだけでのP2P接続が失敗するケースが構造的に多い。これはRoomの主要ユースケース（会議室での配属会議等）と正面から衝突する。

**v1の方針**：TURN未対応と割り切る。接続失敗時は「一部の厳格な社内ネットワークでは接続できない場合があります」とUI上で明示し、正直に免責する（statements第7原則「過剰な断定はしません」に整合）。

**v2以降の検討**：安価な従量課金TURN（Cloudflare Calls等）の導入をロードマップに明記する。無料版では提供しない前提とする。

**Cursorへの指示**：接続失敗時のフォールバックUI（エラーメッセージ＋原因説明）は**v1のMUST要件**とする。「なぜか繋がらない」という無言の失敗を絶対に作らないこと。

### 7-2. 同時編集の競合解決

2人が同時に同じカードを操作した場合の挙動を明確に定義する。

**採用方式：Last-Write-Wins（LWW）**。`updatedAt`（クライアントのタイムスタンプ、ミリ秒）が新しい更新を採用する。CRDT（Y.js等）は本要件に対してオーバースペックと判断し、**採用しない**（第8章参照）。

```
受信した更新のupdatedAt > ローカル保持しているカードのupdatedAt
  → 受信データで上書き
それ以外
  → 無視（ローカルを維持）
```

**既知の限界**：クライアント間の時計ズレがある場合、正確なLWWにならない可能性がある。v1では許容する（数秒〜数十秒規模の同時操作の頻度が低いユースケースのため）。v2で単調増加カウンター方式への移行を検討する。

## 8. 技術スタック

| 技術 | 採用可否 | 理由 |
|---|---|---|
| React + TypeScript + Vite | 採用 | 既存の開発環境との親和性 |
| Konva.js（Canvas描画） | 採用 | 生Canvas APIよりドラッグ実装のハマりどころを吸収できる。ただしバンドルサイズの目安（初期ロード合計 **250KB以内（gzip後）** を目標値とする）を守ること |
| **Y.js（CRDT）** | **不採用** | 本要件（単純な座標・テキストの状態同期）に対してオーバースペック。LWW（第7-2章）で要件を満たせるため、「引き算優先」の設計原則に反する複雑性を持ち込まない |
| WebRTC DataChannel | 採用 | P2P同期の中核 |
| WebCrypto（AES-GCM） | 採用 | E2E暗号化 |
| Supabase | 採用（Signaling専用） | シグナリング・TTL管理・暗号化Blobの一時UPSERTのみ。それ以上のロジックを持たせない |

## 9. セキュリティ思想・脅威モデル

### 目標は「コピー禁止」ではなく「情報漏えいの抑止」

ブラウザ技術で完全なコピー防止は不可能。目標は「抑止」であり、「絶対に防げる」とは謳わない（statements第7原則に整合）。

### 脅威モデル（守る対象 / 守れないもの）

| 守る対象（○） | 守れないもの（×） |
|---|---|
| 誤コピー・誤共有 | スマホでの画面撮影 |
| プロジェクター投影中の覗き見（離席時等） | HDMIキャプチャ・OSレベルキャプチャ |
| URLの意図しない拡散 | 悪意ある参加者によるメモ書き・盗撮 |

この線引きをRoomのヘルプページ上でも必ず明示する。

### 「主な脅威」の明文化

Roomが対策すべき主な脅威は以下の2点と定義する（レビューを依頼する際・実装判断に迷った際は、常にこの2点に立ち返る）。

1. Room URLの意図しない拡散
2. 参加者による無断スクリーンショット・離席中の覗き見

サーバー運営者（開発者自身）による覗き見の懸念への対策は、E2E暗号化という技術設計そのもので既に担保されているため、UI上で過剰に強調しない（「運営者が信用できないから」という訴求は避ける。statementsの「過剰な断定はしません」の精神に反する）。

## 10. UX思想

コピーさせない、ではなく**コピーしたくならないUI**を作る。ホワイトボード・カード・付箋・ドラッグ中心。長文を書くUIは作らない。

### MindMap要素・関係性表現（確定）

- v1では実装しない。v2でも「線」は原則不採用
- 採用検討可（v2）：ゆるい囲み（グルーピング枠）。半透明・色分け・タイトル任意
- 不採用：付箋間を結ぶ明示的な線・矢印・親子階層・自動接続

理由：自由配置（KJ法的なあいまいな近さ）の価値は発散思考を支えることにあり、関係線は早すぎる構造化を強制するため。

### チャット機能の扱い（確定）

独立したチャット機能は実装しない。マーケティングでも「セキュアチャット」を主訴求にしない。

- 許可するのは事実記述のみ（例：「入力内容はサーバーに残らず、全員退出で削除」）
- 「チャットとして使える」「密談用」など、チャット連想を誘導する訴求コピーはNG
- v2検討対象は、付箋に対する短いコメント（1行メモ）まで。時系列ログ/既読/メンションは対象外

## 11. Trust Level

Room作成時に選択する、用途に応じたUX調整の仕組み。**「セキュリティ機能を全部ON」という発想ではない**点に注意。

| Trust Level | 想定用途 | 機能 |
|---|---|---|
| Open | 一般用途 | ウォーターマークなし |
| Private | 社内会議向け | ウォーターマーク、一定時間で画面ロック |
| Confidential | 重要会議 | ウォーターマーク、Privacy Lock（後述）、一定時間操作なしでマスク、画面復帰はクリック |

### Privacy Lock（旧称「Panic Mode」からの改名・機能再定義）

**旧称「Panic Mode」（Escキー連打で画面を白くする）は採用しない。** 「痕跡を瞬時に消す」という文脈を強く示唆し、正当な離席保護と誤解されるリスクがあるため。

代わりに、**離席時の覗き見防止（OSのスクリーンロックと同種の正当な用途）**として再定義する。

- 一定時間（Trust Levelごとに設定可能）操作がない場合、自動的に画面をマスク
- 復帰はクリック／タップのみ（緊急性を煽る操作は設けない）
- 名称は「Privacy Lock」に統一し、「隠す」ではなく「保護する」という文言でUIコピーを統一する

## 13. 実装原則（Cursor向け設計原則）

### 引き算を最優先

迷ったら、機能を追加するのではなく削る。

### 状態同期のみ

送信するのは第6章のデータ構造程度に限定する。

### 永続ストレージは禁止

Room終了後、以下を全て削除する：暗号鍵、暗号化Blob（サーバー側）、LocalStorage、SessionStorage、IndexedDB。

### UIより速度

美しいアニメーションより、ドラッグ操作が軽いことを優先する。

### 思想を守る

Google DocsやSlackを目指さない。「一緒に考えて、終わったら片付ける」という体験を最優先にする。

## 14. SUGUDASU Room 憲法（実装中の判断基準）

Cursorが実装中に機能追加を提案してきた場合、以下の原則との整合性を先に確認し、反する場合は理由を添えて却下すること。

1. **ブラウザファースト** — ブラウザ標準APIを最大限活用し、ネイティブアプリを前提にしない
2. **データ最小化** — 保存しないことをデフォルトとし、保存が必要な場合のみ明示的に行う
3. **引き算の設計** — 機能追加よりも、シンプルさと軽さを優先する
4. **共同思考ファースト** — ドキュメント作成ではなく、リアルタイムの合意形成を支援する
5. **誠実なセキュリティ** — 「完全に防げる」とは言わない。「漏えいしにくく、残りにくい」設計を追求する
6. **接続の限界を隠さない** — WebRTC P2Pが接続できない環境（一部の企業ネットワーク等）が存在することを前提とし、「必ず繋がる」とは謳わない

## 15. Cursor向け実装ガードレール（厳守）

```
# 実装ガードレール

## 絶対にやらないこと
- Redux/Zustand/Recoil等の状態管理ライブラリの導入禁止。
  useState/useRef + WebRTC DataChannelのイベントハンドラで完結させる
- Y.js等のCRDTライブラリを導入しない（第8章参照）。LWW方式で実装する
- カスタムフックの新設は「2箇所以上で使い回す場合」のみ許可
- 「将来の拡張のため」を理由にした型・interfaceの多重定義禁止
- 単一責任原則を盾にした過剰なファイル分割禁止
- try/catchは発生しうる箇所（getUserMedia、RTCPeerConnection確立、
  WebCrypto操作）のみ最小限で囲む。全行に防御的try/catchを付けない
- 未依頼のテストコード・Storybook・E2Eテスト設定を生成しない
- 「念のため」の多重防御・過剰なnullチェックをしない
- Panic Mode / 「痕跡を消す」ことを示唆する名称・機能・UIコピーを
  一切実装しない（第11章：Privacy Lockに統一）
- ファイル転送・画像アップロード関連のコード、UIボタン、
  受け入れ用のinput要素を一切実装しない（v1は絶対に作らないもの、第12章）
- 「完全に安全」「100%痕跡が残らない」「絶対に繋がる」等の
  断定的な文言をUIコピー・コメント・READMEに含めない
  （statements第7原則、本仕様書第14章原則5・6）
- 付箋(カード)間を結ぶ「線」「矢印」「親子関係」を一切実装しない
- グルーピング枠を実装する場合、意味ラベルを強制しない
  （タイトルは任意入力、未入力でも成立）
- チャットスレッド、メッセージログ、既読、メンションを実装しない
- 「チャットとして使える」「密談用」などの訴求コピーを実装しない

## TURN未対応時のフォールバック（MUST・省略禁止）
- ICE接続がfailedになった場合、無言で失敗させない
- 「一部の社内ネットワークでは接続できない場合があります」という
  趣旨のエラーメッセージを必ず表示する

## バンドルサイズ
- 初期ロードの合計サイズ（gzip後）250KB以内を目標とする
- 依存関係を追加する場合、bundle sizeへの影響を一言明記する

## パフォーマンス（厳守）
- ドラッグ中に全体再描画を禁止
- React Contextの全面更新を禁止
- 差分更新のみ（O(1)更新を維持）
- Map/Setを使った状態管理を基本とする
- requestAnimationFrameを利用し、移動はtransformで行う
- top/leftの連続書き換えを禁止
- UndoはCommand Patternで管理する
```
