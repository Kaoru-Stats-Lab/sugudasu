# SUGUDASU Image Trim — 最終仕様書（確定版 v4）

対象: `image-trim`（指定サイズへの画像一発トリマー）
本書はプロダクト仕様のみを記載する。実装言語・ライブラリ・出力形式（HTML/Tailwind等）の指示は本書に含めない。それらは別途の実装依頼文で指示する。

---

## 関連ツールとの導線

- `mask`（スクショの機密消し）とは想定ワークフローが直列に繋がる: スクショを撮る → mask で機密を隠す → image-trim でサイズを整える → 資料に貼る。両ツールは処理内容が異なる（maskは編集、image-trimは切り出し）ため機能統合はしないが、互いの結果画面から相手ツールへの導線（テキストリンク程度）を設けることが望ましい
- 上記の理由により、DENYの「フィルター・図形描画・レイヤー」は編集領域としてmaskの担当範囲であることが明確になった。image-trim側にこれらを持ち込まない判断を維持する

---

## Why（この仕様の理由）

SNSカバー画像・アバター等、プラットフォームが要求する「指定ピクセルサイズ」を満たすことだけが目的。画像編集ツールを作るのではなく、「指定サイズに収める」という単一ジョブを最速で終わらせる道具である。

**代表ユースケースの追加（v3）**: 「PowerPoint専用」ではなく、Google Slides/Keynote/Notion/Confluence等を含む「資料（プレゼン/ドキュメント）に画像を貼る」全般を対象とする。特に「Webページ等のスクリーンショットを資料に貼る前に、ちょうど良い比率へ整える」という用途は、営業・マーケターに限らずカスタマーサクセス・PM/エンジニア・人事総務まで職種横断で発生する実務摩擦であり、本ツールの中心的な価値提案の一つとして扱う。ただし「画像編集ツール」への拡張ではなく、あくまで既存の枠固定モデルの適用範囲を広げるだけである。

## Philosophy

- 操作対象は常に1つ（画像）。枠は動かさない。
- 余白（背景の虚無）を一切許容しない。常に枠は画像で覆われている。
- 機能を増やす提案が来たら、まず削れないかを疑う。

---

## MUST

- 枠（出力サイズ）は画面中央に固定し、ユーザーは移動・リサイズできない
- 画像はドラッグで移動、スライダーで拡大縮小できる
- 画像は「枠を完全に覆う最小サイズ」を拡大縮小の下限とする（余白防止のガードレール）。ドラッグの移動量も、画像が常に枠を覆う範囲内にクランプする
- プリセットは4種（1500x500 / 256x256 / 16:9 / 4:3）。切り替え時は現在の画像に対して自動的にcover-fit・中央寄せに再計算する
- 16:9・4:3の資料向けプリセットは、SNS系プリセット（プラットフォーム規約による厳密なピクセル指定）とは性質が異なり、本質的には「比率」が重要である。ただし出力はピクセル画像として書き出す必要があるため、一般的なプレゼン解像度を目安にデフォルト値を設定する（例: 16:9→1920x1080、4:3→1600x1200程度。数値は実装時に一般的なプレゼンツールの標準解像度を確認して決定する）
- クリップボードからの貼り付け（Ctrl+V / Cmd+V）に対応する。ユーザー操作起点のpasteイベントで画像データを取得し、D&D・ファイル選択と同格の入力経路として扱う
- 画像読み込み時にEXIFのOrientation情報を反映し、正しい向きで表示・書き出しする
- 長辺が一定サイズ（目安2048px）を超える画像は、読み込み直後に内部作業用画像としてダウンスケールする。元の高解像度データは保持しない。プレビュー・書き出しの両方をこのダウンスケール後の画像から行う（プレビュー用と書き出し用でロジックを分岐させない）
- プレビュー上の枠と、実際にトリミングされる範囲は完全に一致する（別々の座標系・別ロジックで計算しない。単一の状態［拡大率・オフセット］から両方を導出する）
- ドラッグ中の再描画は`requestAnimationFrame`等でスロットリングし、フレーム落ちを防ぐ
- マウス・タッチ双方の入力に同一のイベント処理で対応する（Pointer Events相当の統一的な扱い）
- リセットボタンを設置し、押下で初期のcover-fit・中央寄せへ戻せる
- 非対応形式（HEIC等、ブラウザがデコードできない画像）やデコード失敗時は、エラーメッセージを表示し処理を止める。ここで自動変換は行わない
- 画像未読み込み時・空状態でのボタン操作を無効化する

---

## SHOULD

- 現在の拡大率を100%基準の数値（例: 150%）で小さく表示する
- ダブルクリック/ダブルタップで画像を中央（初期位置）に戻す
- マウスホイール／トラックパッドでのズームに対応する（スライダーは残す。ホイールは追加の操作手段であり、スライダーの代替ではない）
- プリセットは「SNS」「資料」の2グループでグルーピング表示する（4種になったため実施する）
- 貼り付け（Ctrl+V）が可能であることを、ドロップゾーンの案内文言に明記する（D&Dより優先度を上げた専用UIは設けない。あくまで文言レベルの案内に留める）

---

## DENY（明確にやらないこと）

- 回転・フィルター・文字入れ・図形描画・レイヤー・編集履歴・テンプレート・AI画像生成/補正
- 複数画像の一括処理
- 枠のサイズ変更・自由比率のトリミング
- HEIC等非対応形式の自動変換
- クロップ位置のサーバー保存・共有機能
- URL入力によるWebページ自動取得・スクリーンショット化（Puppeteer等のサーバー処理、html2canvas等のブラウザ内擬似キャプチャ、ブラウザ内ブラウザを含め、いかなる方式でも非対応。ユーザーが自分でスクリーンショットを撮り、貼り付ける前提を崩さない）

## 判断ログ（検討したが不採用にした案とその理由）

- **「枠を動かせる逃げ道を残すべき」という案**: 見送り。枠固定は本ツールの差別化の核であり、これを崩すと「Canvaの縮小版」になってしまう。縦長画像がうまく収まらないケースがあることは事実だが、それは「対応しきれない入力がある」という制約として受け入れる。回避策を用意するのではなく、対応できない旨をUI上で静かに伝える方針とする（エラーではなく、単に「思ったより被写体が大きく見える」程度の許容範囲とする）。
- **プリセットのカテゴリ分け（SNS/YouTube/ブログ等）を初期実装から入れる案**: 見送り。現状2プリセットのため過剰設計。3種類目を追加する際に再検討する。
- **スライダーを廃止しホイール操作のみにする案**: 見送り。トラックパッド／ホイール非搭載のポインティングデバイス利用者を排除するリスクがあるため、スライダーはMUSTとして残し、ホイールは追加の利便性（SHOULD）とする。
- **（v3）資料向けプリセットを6種（16:9, 4:3, Word横長, note, Xヘッダー, LinkedIn）一括追加する案**: 一部のみ採用。16:9・4:3の2種のみ追加し、残り4種は将来に回す。v2の判断ログで「プリセットのカテゴリ分けは3種類目を追加するタイミングで検討する」と決めており、今回がそのタイミングにあたる。一度に大量追加すると同じ判断ログの想定していたリスク（選択肢過多による迷い）を招くため、SNS/資料の2グループ・計4種に絞り、グルーピングUIを合わせて導入する。
- **（v3）Ctrl+Vを前面に出す専用UI（強調ボタン等）を設ける案**: 見送り。ペースト対応自体はMUSTとして採用するが、UI上の優先順位づけは実利用データがない段階で決め打ちすべきでないため、ドロップゾーンの案内文言に追記するに留める（SHOULD）。

## エッジケース対応表（実装ガードレールとして厳守）

| リスク | 対応方針 |
|---|---|
| EXIF Orientation未反映で画像が横倒しになる | 画像デコード時にオリエンテーション情報を反映してから以降の処理に渡す |
| 巨大画像（4000px超）でのメモリ・描画負荷 | 読み込み直後に長辺2048px程度へダウンスケールした作業用画像を作り、以降は全てそれを参照する |
| プレビューの枠と実際の切り出し範囲がズレる | 単一の状態（拡大率・オフセット・枠サイズ）から、プレビュー描画と書き出し処理の両方を導出する。別々の計算式を持たない |
| ドラッグで枠外に余白ができる | オフセットの可動範囲を「画像が常に枠を覆う」条件で毎フレームクランプする |
| ドラッグ中のフレーム落ち | 再描画をアニメーションフレームに同期させる |
| 非対応形式・破損ファイル | デコード失敗を検知し、処理を止めてエラーメッセージを表示する |

**① 実装プロンプト（1つ目）**

```
添付の仕様書（SUGUDASU Image Trim 最終仕様書）に基づいて実装してください。

技術制約:
- Vanilla JavaScript + HTML5 Canvas API のみを使用する
- フレームワーク（React/Vue等）、ビルドツール、状態管理ライブラリは一切追加しない
- 外部の画像処理ライブラリ（Cropper.js等）は使用しない
- 出力は1ファイルの完結したHTML（Tailwind CSSはCDN経由で読み込み、JSも同ファイル内に記述する）
- ファイル分割はしない。parser.js等への切り出しは不要
- Ctrl+V貼り付けの実装は、既存の sugudasu.com/mask で稼働しているCtrl+Vパターンを踏襲してください（同一プロダクト内でのUX一貫性のため）。

実装ルール:
- 仕様書のMUST項目は全て満たすこと
- SHOULD項目は余力があれば実装し、余力がなければ未実装のまま報告すること
- DENY項目は絶対に実装しないこと
- プレビューに表示する枠と、実際にトリミングされる範囲は、単一の状態（拡大率・オフセット・枠サイズ）から導出すること。プレビュー用と書き出し用で別々の座標計算を作らないこと
- コメントは最小限とし、関数名で意図が伝わるようにすること
- 「将来の拡張性のため」の抽象化やオプション化は行わない。仕様書に書かれている範囲だけを実装する
```
