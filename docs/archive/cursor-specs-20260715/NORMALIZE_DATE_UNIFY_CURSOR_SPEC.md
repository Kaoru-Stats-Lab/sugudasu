# SUGUDASU normalize — 日付表記統一プリセット追加 仕様書（確定版 v1）

対象: 既存id `normalize`（sugudasu.com/normalize）への機能追加。新規idは作らない。
現状: normalizeは既存プリセット3つ（sql_in / comma_join / name_trim）＋トグル3つ（並べ替え / 重複を消す / 行をしぼる）の計6機能構成。
追加内容: 7つ目のプリセットとして「日付表記統一」を追加する。

本書はプロダクト仕様のみを記載する。実装言語・出力形式の指示は別途の実装依頼文で行う。

---

## Why

バラバラな日付表記（`2026/7/11` `26/7/11` `2026.7.11` `R8/7/11` `2026-07-01` 等）が1つの列に混在した状態で他部署・他社から届き、Excelの`TEXT`/`DATEVALUE`関数がエラーを起こし、結局手作業で1件ずつ打ち直す、という実務摩擦は実在する。

ただし、これを独立プロダクトとして切り出すと「データ整形ツール」という大きすぎるカテゴリの入口になり、電話番号・郵便番号・住所・氏名等への拡張要求を呼び込む（OpenRefine/Power Queryのような汎用データクレンジング化のリスク）。normalizeは既に「1列の文字列を整えて次工程（Excel/CSV）に渡す」という役割で運用されており、日付表記統一はその役割の範囲内にある一機能として扱う。

## Philosophy

- 「データを綺麗にする」のではなく「次のソフト（Excel/CSV）が読める形に整える」に徹する
- 曖昧な入力は推測で変換せず、警告を出して人間の判断に委ねる
- 完璧なパーサーを目指さない。対応できないものは対応できないと明示する

---

## MUST

- 既存プリセット（sql_in / comma_join / name_trim / 並べ替え / 重複を消す / 行をしぼる）の並びに「日付表記統一」を7つ目として追加する
- 入力: 複数行のテキスト（1行1件の日付データを想定）。既存の入力欄・処理フローをそのまま使う
- 出力形式は `YYYY/MM/DD` 固定（スラッシュ区切り）とする。将来的にハイフン区切り等の選択が必要になった時点で検討する（現時点では選ばせない）
- 対応する区切り文字: スラッシュ（`/`）・ハイフン（`-`）・ドット（`.`）・「年」「月」「日」による区切り
- 前後の半角・全角スペースはトリムしてから解析する
- 全角数字（`２０２６`等）は解析前に半角へ変換する
- 4桁年（`2026`）を含む表記は解析対象とする
- パース不能な行はエラー表示とし、その行は変換せずそのまま出力する（他のnormalize機能と同じ「継続処理・個別行エラー表示」の挙動を踏襲する）

---

## SHOULD

- 元号表記（`R8.7.11` `令和8年7月11日`等）を西暦へ変換する。対応は令和のみとし、平成以前は対象外とする（対応表は固定の単純なルックアップで実装できる範囲に留める）
- 2桁年（`26/7/11`）は2000年代として解釈するが、`warning`表示を出し「2026年の意味で解釈しました」等、解釈内容をユーザーが確認できるようにする
- 年を含まない表記（`7/11`のみ等）は、実行時点の年として補完しつつ`warning`表示を出す

---

## DENY

- Excelシリアル値（`46210`のような数値のみの表記）の自動日付判定。理由: 正当な数値データ（金額・ID・件数等）を誤って日付として書き換えるリスクが、対応できないことによる不便さを上回るため。ユーザーが明示的に「これは日付シリアル値です」と指定する手段は将来検討するが、自動検出はしない
- 電話番号・郵便番号・住所・氏名・メールアドレス等、日付以外のデータ種別への対応拡張
- 出力フォーマットの選択UI（ハイフン区切り・和暦出力等の切り替え）
- 平成・昭和等、令和以外の元号への対応
- 日付の妥当性検証を超えた処理（曜日の自動付与、祝日判定等）

## エッジケース対応表

| リスク | 対応方針 |
|---|---|
| `2026/7/1` と `2026/07/01` のような桁数違いの混在 | 両方とも正しく解析し、出力は`2026/07/01`のようにゼロ埋めで統一する |
| 2桁年の解釈間違い（`26`が2026年か1926年か） | 2000年代として解釈しつつ`warning`を出す。断定はしない |
| 元号と西暦が同じ列に混在 | 行ごとに個別解析するため問題ない（既存のparseLine的な行単位処理を踏襲） |
| 全角・半角・元号・区切り文字が複合的に混ざる（`Ｒ８．７．１１`等） | 全角→半角変換を最初に行ってから、区切り文字・元号の解析に進む処理順序とする |
| 日付ではない数値・文字列（`ABC123`、金額`¥46,210`等）を誤って日付と解析してしまう | 4桁年相当の数値や日付らしい区切り文字パターンが検出できない場合は`error`とし、変換しない。数値のみの文字列は対象外（DENY参照） |

## 判断ログ

- **独立id `data-cleanse` / `data-cleanse-date` として新規開発する案**: 不採用。normalizeが既に持つ「1列のテキストを整えて次工程に渡す」役割の範囲内であり、新規HTML・新規入力フロー・新規Undo等の基盤を重複して作る必要がない
- **「Paste Fix」等への名称統合・リブランド案**: 不採用。normalizeが既にその位置（全角半角・空白・改行等の統一）を占めており、名称変更は既存ユーザーの認知を分断するだけで得るものがない
- **Excelシリアル値の自動日付判定**: 不採用。技術的な実装難易度以上に、正当な数値データを誤って書き換える誤検出リスクの方が実害が大きいと判断
- **電話番号・郵便番号・住所等への拡張**: 不採用。「データ整形」という言葉を掲げた瞬間に汎用データクレンジングツール化の要求を呼び込むため、日付のみに範囲を固定する


Cursorに順番に貼る用です。normalizeも既に公開済みの実装への追記なので、mask拡張の時と同じく「まず読ませる」ステップを挟んでいます。

**② 実装プロンプト（2つ目・①の報告を確認してから貼る）**

```
添付の仕様書（SUGUDASU normalize — 日付表記統一プリセット追加 仕様書）に基づいて、normalize.htmlに7つ目のプリセット「日付表記統一」を追加してください。

技術制約:
- 新しいファイルは作らない。normalize.html 1ファイルへの追記とする
- 新しい状態管理の仕組みや処理パイプラインを作らない。既存のプリセット切り替え・行ごとの処理・エラー表示の仕組みにそのまま乗せる
- 日付ライブラリ（day.js、date-fns等）は追加しない。標準JSで完結させる
- 外部ライブラリ・フレームワークは追加しない

実装ルール:
- 仕様書のMUST項目は全て満たすこと
- SHOULD項目は既存の実装パターンに大きな変更を必要としない範囲でのみ実装し、コストが高い場合は未実装のまま報告すること
- DENY項目は絶対に実装しないこと。特にExcelシリアル値（数値のみの文字列）を日付として自動判定する処理は、どのような形であれ実装しないこと
- 日付のパース処理は1関数（例: parseDate(line: string)）に集約し、正規表現をあちこちに分散させないこと
- 全角→半角変換は、既存機能内に同種の処理があればそれを再利用し、重複実装しないこと
- パース不能な行は変換せずそのまま出力し、エラー表示すること。推測で埋めないこと
- コメントは最小限とし、関数名で意図が伝わるようにすること
- 「将来の拡張性のため」の抽象化（例: 日付以外のデータ種別にも対応できる汎用パーサー）は作らない。日付のみを直接実装する
```
