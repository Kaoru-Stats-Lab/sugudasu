# SUGUDASU mask — 注釈モード追加 仕様書（確定版 v1）

対象: 既存id `mask`（sugudasu.com/mask）への機能追加。新規idは作らない。
現状: mask.htmlは「ぼかし・黒塗り・同色塗り・スタンプ」の4モードを持つCanvasベースの非送信画像加工ツール（Ctrl+V貼り付け対応・Undo/Redo・PNG保存/コピー実装済み）。
追加内容: 5つ目のモードとして「注釈（矢印・四角）」を追加する。

本書はプロダクト仕様のみを記載する。実装言語・出力形式の指示は別途の実装依頼文で行う。

---

## Why

Skitch（旧Evernote買収の注釈ツール）が支持されていた本質は「色や太さの選択」ではなく、**「何も考えずに"ここ"を指せること」**にある。バグ報告・デザインレビュー・障害報告・社内問い合わせで、「スクショに矢印を書いてTeams/Slackに貼る」という5〜15秒の作業を、ペイントソフト起動やCanva/Miroのような重量ツールを介さずに終わらせる。

既にmask.htmlが持つ「機密を隠す」と、今回追加する「意図を指す」は、同じスクショ加工ワークフローの中で両方使われる可能性が高い（機密を隠してから矢印で指す、またはその逆）。別プロダクトに分離せず、同一ツール内のモードとして提供する。

## Philosophy

- ツールを選ばせない。色を選ばせない。太さを選ばせない。
- 「最初に何か図形が置いてある」状態を作らない。ユーザーの操作だけで図形が生まれる。
- モード自体を増やしすぎない（Skitchモード/シンプルモード等のプリセット概念は導入しない）。

---

## MUST

- 既存のモード切り替えUI（ぼかし/黒塗り/同色塗り/スタンプ）の並びに「注釈」を5つ目として追加する
- 注釈モード内で使える図形は**矢印・四角（角丸枠）の2種のみ**
- 初期状態では画面に何も図形を自動配置しない。画像上をドラッグした瞬間に矢印が1本生成される（ドラッグ＝矢印がデフォルト操作）
- 四角を描きたい場合は、モード内の小さな切り替えボタン（矢印⇄四角の2択のみ）で切り替える
- 色は選択不可。マゼンタ系固定色＋白フチ（縁取り）で統一する。太さも固定値とする
- 四角は塗りつぶさない。線のみとし、下の画像内容を隠さない（塗りつぶしが必要な用途は既存の黒塗り/同色塗りモードが担当するため、注釈モードでは扱わない）
- 複数の矢印・四角を重ねて描画できる（1個だけに制限しない。1枚のスクショに複数箇所を指すケースは実務上頻出する）
- 描いた図形は個別に選択して削除できる
- 既存のUndo/Redo・PNG保存・コピー機構をそのまま流用する（注釈モード専用の別機構を作らない）

---

## SHOULD

- 矢印の始点・終点、四角のサイズは自由にドラッグで調整できる（太さ・色は固定のまま、形状のみ可変）
- キーボードショートカット（例: `A`=矢印、`R`=四角、`Delete`=選択中の図形を削除）は実装コストが低ければ入れる。必須ではない

---

## DENY

- 色選択・太さ選択・フォント選択のUI
- テキスト注釈（IME入力対応はCanvas実装上のコストが高いため今回は扱わない）
- 円・フリーハンド線・その他の図形
- 「Skitchモード」「シンプル枠モード」等、モードをさらに分岐させるプリセット概念
- 画像貼り付け時の図形自動配置（触った/触らなかったで図形が消えるような隠れ状態を持つ挙動も含む）
- レイヤー機能、操作履歴パネル、複数矢印のグループ化・整列機能
- 独立した新規id化（`mask`の一部として維持する）

---

## エッジケース対応表

| リスク | 対応方針 |
|---|---|
| 白フチ付き矢印の描画品質 | 白い太めのストロークを先に描き、その上にマゼンタの細いストロークを重ねる順序で描画する（縁取りを別レイヤーとして重ねて処理しない） |
| 細い矢印線へのクリック当たり判定 | 図形選択時のヒット判定は、実際の線の太さより広めの当たり判定領域を持たせる（線ぴったりだとクリックで選択できない） |
| 既存Undo/Redo機構との整合性 | ぼかし/黒塗り等の既存操作と注釈の描画・削除を、同一のUndo/Redoスタックで扱う。モードが違っても操作履歴は1本にする |
| Retina環境での線のにじみ | 既存モード（ぼかし等）で採用している解像度対応方針をそのまま踏襲する |
| モード切り替え時に描画中の図形が残る | 他モードへ切り替えても、既に描画済みの矢印・四角はキャンバス上に残る（消去されない） |

---

## 判断ログ

- **独立プロダクト化（Image Markup等）案**: 不採用。`mask.html`が既に持つ入力（Ctrl+V等）・Canvas基盤・Undo/Redo・出力機構をそのまま流用でき、実装コストが小さい。新規HTMLは四半期1id・既存id拡張優先の原則にも合致する。
- **画像貼り付け時に矢印・枠を自動配置する案（ツイン・プリセット）**: 不採用。「触ったかどうか」という隠れ状態を持つ設計であり、複数箇所を指したい実務ケース（矢印で指しつつ別箇所を枠で囲む等）に対応できない。ドラッグ操作だけで図形が生まれる方式に統一し、隠れ状態そのものをなくした。
- **テキスト注釈**: 不採用（MVP時点）。Canvas上でのIME対応は実装コストが高く、他の図形（矢印・四角）と実装難易度が大きく異なる。将来、需要が明確になった時点で再検討する。
- **モード切り替え（Skitchモード/シンプル枠モード）案**: 不採用。「選択の儀式を排除する」という本仕様の前提と矛盾するため。
- **色の完全固定・変更不可という運用方針**: 維持する。ただし「例外は絶対に認めない」と強く宣言する必要はなく、「現時点では1色のみ」という状態で留める。




Cursorに順番に貼る用です。今回は既存ファイルへの追記なので、①の前に「まず読ませる」ステップを挟んでいます。

**② 実装プロンプト（2つ目・①の報告を確認してから貼る）**

```
添付の仕様書（SUGUDASU mask — 注釈モード追加 仕様書）に基づいて、mask.htmlに5つ目のモード「注釈」を追加してください。

技術制約:
- 新しいファイルは作らない。mask.html 1ファイルへの追記とする
- 新しい状態管理の仕組みや描画パイプラインを作らない。既存のモード切り替え・Undo/Redo・Canvas描画の仕組みにそのまま乗せる
- 外部ライブラリは追加しない
- フレームワークの導入はしない

実装ルール:
- 仕様書のMUST項目は全て満たすこと
- SHOULD項目は既存の実装パターンに大きな変更を必要としない範囲でのみ実装し、コストが高い場合は未実装のまま報告すること
- DENY項目は絶対に実装しないこと
- 矢印・四角の描画ロジックは、既存のぼかし/黒塗り等の矩形選択ロジックと同じ「ドラッグで座標を取る」仕組みを流用し、別の入力処理系を新設しないこと
- Undo/Redoは、既存のモードでの操作と同じスタックに積むこと。注釈専用の別履歴を作らないこと
- コメントは最小限とし、関数名で意図が伝わるようにすること
- 「将来の拡張性のため」の抽象化（例: 図形の種類を増やせる汎用Shapeクラス等）は作らない。矢印と四角の2種類だけを直接実装する
```
