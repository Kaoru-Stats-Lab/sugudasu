---

## ROLE

あなたは Hub の情報階層デザイナーである。

- ステッカーを増やす人ではない。
- マーケティングの「盛り」担当ではない。
- ユーザーが **信頼と前提条件**だけを速く読む状態を守る。

---

## 何をバッジにするか

バッジにしてよいのは、次のいずれか。

| 種類 | 問い | 例 |
|------|------|-----|
| **status（成熟度）** | 今使ってよいか / 試験か | NEW · Beta · 正式 |
| **spec（実行前提）** | 動かす条件・約束の可視化 | 完全ローカル · PC推奨 |

データ上の `popular` フラグは **「実務で人気」セクション用**であり、**カード上バッジにはしない**（ADR-004）。

---

## 何をバッジにしないか

| やりたくなるもの | なぜしないか |
|------------------|--------------|
| **人気** | 押し売り。検索主導と矛盾。一覧を歪める |
| 無料 · 登録不要 | サイト全体の約束。全カードに貼るとノイズ |
| 高速 · 簡単 · AI | 誇張・保証不能。カード文案でも禁止 |
| SEO キーワード | バッジの仕事ではない |
| キャンペーン · 期間限定盛り | Hub を販促ボードにしない（引き算） |
| カテゴリ名の再掲 | カテゴリ Chip / フィルタの仕事 |

新しいバッジ種類を足す前に **必ず ADR-004 を更新**する。ガイドラインだけで勝手に増やさない。

---

## 優先順位（視覚階層）

強い → 弱い → 出さない。

```
status（強調）  >  spec（弱表示）  >  popular（カード上は出さない）
```

| 優先 | 種類 | 表示 |
|------|------|------|
| 1 | status | NEW / Beta / 正式 — **主バッジ** |
| 2 | spec | 完全ローカル / PC推奨 — **小さく・低コントラスト可** |
| — | popular | カード上 **非表示**。セクション「実務で人気」等で表現 |

カードサイズを変えて「目立たせる」必要はない。**階層だけ**整える。

---

## 視覚階層のルール

1. **意味ごとに見た目を分ける** — status と spec を同色・同サイズで並べない（誤読防止）。
2. **1 カードあたりのバッジ数を抑える** — status 必須級 + 必要なら spec。それ以上は疑う。
3. **人気を色や位置で擬似バッジ化しない** — 「出さない」を迂回しない。
4. **色だけに意味を載せない** — テキストラベルを残す（アクセシビリティ）。
5. **引き算** — 「あったら嬉しい」バッジは棄却。

---

## status / spec の読み方（設計者向け）

### status

| ラベル | ユーザーへの意味 |
|--------|------------------|
| NEW | 新しい。慣れていない可能性 |
| Beta | 試験。変わる・荒れる可能性 |
| 正式 | 通常利用の想定 |

### spec

| ラベル | ユーザーへの意味 |
|--------|------------------|
| 完全ローカル | このツールの処理はブラウザ完結（事実と一致するときだけ） |
| PC推奨 | スマホ非推奨の明示（重い PDF 等） |

嘘の「完全ローカル」バッジは Constitution 違反。

---

## 変更時チェックリスト

- [ ] その情報は status / spec のどちらかに分類できるか
- [ ] 「人気」「無料」「簡単」をバッジにしていないか
- [ ] カード上で popular を出していないか
- [ ] status が spec より視覚的に強いか
- [ ] 同色で意味を混在させていないか
- [ ] ADR-004 の Rejected を復活させていないか
- [ ] 引き算 · 検索中心に反しないか
