## 構成（H1〜CTA）

### ヘッダー

- **カテゴリラベル:** `制作・進行管理 · 更新 YYYY-MM-DD`
- **H1:** 制作スケジュール表をExcelで回すときの落とし穴
- **リード（100字以内で誰向けか）:**  
  WEB制作の進行管理は、チーム内は Asana、クライアント向けは Excel という**二極化**がよくあります。問題はツールの良し悪しではなく、**クライアント確認が1日遅れた瞬間に後続が全部ずれる**のに、版を再送し続ける運用が残っていることです。ディレクター向けに、よくある失敗と Excel が向く場面を整理します。

### コールアウト「この記事でわかること」（bullet 5）

1. 制作スケジュール表で版ズレ・手動再計算が起きる典型パターン  
2. Asana/スプレッドシート/Excel — それぞれが向く層（二極化の整理）  
3. Excel / スプレッドシートが向いている工程（企画・初版・提出 PDF）  
4. クライアント遅延1日で後続をずらすときのチェックリスト  
5. 共有 URL が効く場面と、要員管理まで求める場合は別ツール向きであること（§4 正直線）

### H2: なぜ「Asana でもスプレッドシートでも」足りないことがあるか（§6）

**本文 200〜300字 · 礼賛しない**

- **Asana / Monday 側:** チーム内タスク管理には向くが、クライアント提示用の工程表・版管理とは別物になりがち。稟議・習得コストもある。  
- **スプレッドシート共有側:** 手軽だが、**先行タスク1行の変更で後続が連動しない**（または式が壊れる）。  
- **我々の立ち位置（記事内では一般論）:** 「ガント SaaS の代替」ではなく、**Excel 正本の上に乗る連動再計算 + 閲覧 URL** の話。  
- **避けるKWは主語にしない:** 「Asana 代替」「ガントチャート 無料」は使わない。

### H2: 制作進行表でよくある3つの失敗

**各 H3 + 本文 150〜200字**

#### H3-1: 版が3つになる（最新はどれ？）

- 症状: `制作スケジュール_v3_最終.xlsx` · Slack に PDF · クライアントが古い版で返信  
- 原因: ファイル再送文化 · 閲覧専用リンクの未整備  
- キーワード自然投入: `案件 進行管理 シート` `制作スケジュール表 エクセル`

#### H3-2: クライアント確認1日遅れ → 後続を手でずらす（§6 核心ペイン）

- 症状: デザイン初稿確認が金曜→月曜に。**コーディング着手・入稿・公開日**を1行ずつ手修正 · 最新版 PDF をメール再送  
- 原因: 日数連動の式が壊れる · 結合セル · 担当が別シート  
- キーワード: `スケジュール 逆算 表` `WEBディレクター スケジュール 管理`

#### H3-3: 外注だけ古い納期で動いている

- 症状: コーダーは木曜納品のつもり · ディレクターは水曜に変更済み  
- 原因: チャット口頭 · スプレッドシートの通知オフ  
- キーワード: `外注 スケジュール` `デザイン 進行管理`

### H2: FAQ（構造化データ候補 · Q5）

1. **制作スケジュール表のエクセルテンプレートはどこまで使えばよいですか？**  
   → 提案・初版・提出 PDF まで。制作中の動的更新は別の「最新1本」を用意。

2. **WEBディレクターのスケジュール管理で一番つらいのは何ですか？**  
   → 版ズレと、遅延時の手動ずらし（本文 H3-1, H3-2 へ内部リンク）。

3. **クライアントに工程表を共有するとき、Excel のまま送るべきですか？**  
   → 初回・正式提出は PDF/Excel 可。更新が週次以上なら閲覧 URL の検討（Sync LP へ CTA · 未公開時は一般論のみ）。

4. **クライアント確認が1日遅れたとき、後続タスクはどうずらしますか？**  
   → 影響日数を先に合意し、公開日など固定マイルストーンだけ錨にする（表B 参照）。

5. **Asana と Excel、制作会社はどう使い分ければよいですか？**  
   → チーム内は PM ツール · クライアント提示と外注周知は表計算 — **二重管理を減らす**のが目的（§6）。
