# SUGUDASU Tool Card Writing Guideline

**正本** · Agent 必読（Hub カード文言）  
**Version:** 1.0 · **更新:** 2026-07-20  
**適用:** `conceptName` · `hub-cards.json` の `blurb`（description）を書く・直すとき  
**実装先:** `data/tool-registry.json` · `data/hub-cards.json` → `npm run validate:hub-ia`  
**関連:** `HUB_IA_REFRESH_V2.md` · `TOOL_NAMING_AGENT_PLAYBOOK.md` §1.5

Hub カード文案について、この文書以外のルールを新造しない。品質チェック表をすべて通してから JSON に書く。

---

# Purpose

SUGUDASUは「登録不要・ブラウザだけで3分以内に終わる実務ツール集」である。

ツールカードはSEOのためではなく、

> 「一覧を見た瞬間に、自分の課題と一致する」

ことを目的とする。

カードは**検索キーワードではなく、利用シーン（JTBD）を説明するUI**である。

---

# ROLE

あなたは

- UXライター
- 情報設計(IA)デザイナー
- 日本語コピーライター
- 実務Webツールのプロダクトマネージャー

として振る舞う。

マーケティングコピーではない。

実務ユーザーが

「これ、自分のことだ」

と思える文章を書くこと。

---

# Output

各カードは以下のみを出力する。

```json
{
  "toolId": "",
  "conceptName": "",
  "description": ""
}
```

descriptionは

**2文固定**

---

# description構造

必ず

```
○○したい時に。
○○します。
```

の2文構成。

例

```
PDFの中から画像だけを取り出したい時に。
画質を落とさず抽出します。
```

---

# 1文目のルール

1文目は

「いつ使うか」

を書く。

書くのは

ユーザーの状況

である。

## 良い例

```
画像形式を変更したい時に。
```

```
会議のメモを議事録にまとめたい時に。
```

```
QRコードをパソコンで読み取りたい時に。
```

---

悪い例

```
画像を変換できます。
```

```
便利なツールです。
```

```
高速に変換できます。
```

---

# 2文目のルール

2文目は

ツールが何をするか

だけを書く。

例

```
WebPをJPGへ変換します。
```

```
画像を抽出します。
```

```
議事録を整理します。
```

性能は書かない。

---

# 基本思想

カードは

「何ができるか」

ではなく

「どんな時に使うか」

を書く。

---

# 文体

常体は禁止。

必ず

```
〜したい時に。
〜します。
```

で終える。

---

# 長さ

descriptionは

40〜70文字程度。

長く説明しない。

---

# conceptName

conceptNameは

ユーザーが一瞬で理解できる一般名称。

なるべく

2〜8文字程度。

例

良い

```
差分チェック
```

```
画像透かし
```

```
議事録整形
```

悪い

```
AI文章差分比較システム
```

```
画像ウォーターマーク付与
```

---

# 品質チェック

作成後に以下を確認する。

□ 2文構成か

□ 1文目が利用シーンか

□ 2文目が機能説明か

□ 「〜したい時に」で終わるか

□ 「〜します」で終わるか

□ 誇張がないか

□ 保証できない性能を書いていないか

□ UI説明を書いていないか

□ ブランド用語に統一されているか

□ 60文字前後に収まっているか

```

```

```

決定事項やタスクを整理します。
```

# 最終目標

カード一覧を見たユーザーが

「このツールで何ができるか」

ではなく

「これは今の自分のためのツールだ」

と思えること。

そのために

**機能説明ではなく、利用シーンを書く。**
