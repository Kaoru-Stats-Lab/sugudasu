# SUGUDASU Sync — タイムライン保持・削除ポリシー

**更新:** 2026-06-25（提督確定 — 無期限保持禁止）  
**関連:** `SUGUDASU_SYNC_LINE.md` §3-2 要素 F/H · `SYNC_S1_ARCHITECTURE.md`

---

## 1. 原則

| 原則 | 内容 |
|------|------|
| **無期限クラウド保存はしない** | イベント単位の使い捨て mental model · DB 肥大・コスト防衛 |
| **課金でも同時ルーム上限** | Supabase 行数 cap · 詳細は `SYNC_STORAGE_QUOTAS.md` |
| **長期ログはユーザー JSON** | エクスポート DL · クラウドは短期バッファ |
| **削除は一等市民** | オーナーがいつでもルームごと消せる（S1 必須） |
| **自動パージ** | `retain_until` 経過後にバッチ削除（S2 以降の cron） |
| **アーカイブ ≠ 永久保存** | エクスポート or テンプレ取込 · クラウドに残し続けない |

**コア無料**は従来どおり localStorage — ユーザーがブラウザデータを消すだけ（サーバ負荷ゼロ）。

---

## 2. 削除の種類

| 種類 | 誰 | いつ | 実装 |
|------|-----|------|------|
| **手動削除** | ルームオーナー | いつでも | `DELETE sync_rooms` → `sync_room_states` CASCADE |
| **自動パージ** | システム | `retain_until < now()` | pg_cron / 日次 Edge（S2+） |
| **エクスポート後削除** | オーナー | イベント終了後 | JSON DL → 手動削除（要素 F · `SYNC_STORAGE_QUOTAS.md` §4） |

閲覧 URL はルーム削除と同時に **無効**（404）。

---

## 3. `retain_until`（保持期限）

`sync_rooms.retain_until` — **必ず設定** · NULL 禁止（DB デフォルトあり）。

| 課金・状態 | 保持目安（案） | 根拠 |
|------------|----------------|------|
| **trial** | 最終更新 + **14 日** または作成 + **30 日**（短い方） | 試用 · 放置ルーム掃除 |
| **イベント単位課金**（S3） | `event_date` + **7 日**（§4 課金モデルと整合） | 当日〜翌週の振り返りのみ |
| **延長購入**（将来） | チェックアウトで `retain_until` を延長 | 例外は課金で明示 |

**更新ルール:** 編集者が保存するたびに trial は「最終更新 + 14 日」に **延長しない**（放置検知のため）。有料イベント枠のみイベント終了日基準。

### 3-1. 有料 → 無料（ダウングレード）— 提督 Q&A · 2026-06-26

**結論: 解約・失効の瞬間にデータを消さない。** 業界標準は **③段階** — **権利停止 → 読み取り専用の猶予 → `retain_until` 到達でパージ**。

| 段階 | いつ | データ | ユーザーができること |
|------|------|--------|----------------------|
| **A. 有効** | 課金中 · チケット消費済 | クラウド保持 | 編集 · 同期 · 新規ルーム（枠内） |
| **B. 猶予（grace）** | Webhook で `expired` 等 · またはサブスク期間終了後 | **削除しない** | **閲覧 · JSON エクスポート · 手動削除** のみ |
| **C. パージ** | `retain_until < now()` | 自動削除 | —（事前に UI / メールで警告） |

**即時削除は採用しない理由:** 幹事はイベント直後にエクスポート・振り返りする · 誤解約 · カード更新漏れが多い · 法務・信頼コストが大きい。

#### 権利（entitlement）と保持（retain_until）は別軸

| 軸 | ダウングレード時の動き |
|----|------------------------|
| **`user_entitlements`** | Stripe Webhook で失効 · 新規 Checkout / 新規 active 化は不可 |
| **`sync_rooms.entitlement`** | `active` → **`expired`**（新規保存 · 同期 · 端末 cap 超過は拒否） |
| **`retain_until`** | **原則短縮しない**（既に払った保持期間は尊重） |

**イベント単位課金（¥980/件）:** 決済時に付与した `retain_until`（`event_date + 7日`）は **失効後もそのまま**。無料に「戻った」＝新イベント用の権利が無いだけで、**過去に買ったルームは期限まで残る**。

**月額サブスク（Schedule 等 · 将来）:**

- Stripe の **`cancel_at_period_end`** — **支払い済み期間の末日まで** は A（有効）のまま（即ダウンしない）
- 期間終了 Webhook 後 → B（猶予）へ
- 猶予中の `retain_until` = **`max(既存 retain_until, expired_at + 14日)`**（エクスポート窓 · 調整可）
- 同時ルームが trial 上限（1）を超える場合 → **古いルームを即削除しない** · **超過分は読み取り専用** · 新規作成はブロック（quota は新規 INSERT のみ）

#### 業界のふつう（参考）

| パターン | 例 | Sync との対応 |
|----------|-----|----------------|
| 期間末まで有効 | Stripe Subscription | 採用 |
| 読み取り専用 grace | Notion · 多くの B2B SaaS（7〜30日） | **14日** を初期案 |
| エクスポート促し | Slack export · Figma | JSON DL + 削除確認 |
| 即時全消し | ほぼ採用しない（悪評 · 規約明示が必要） | **不採用** |
| 無制限クラウド残留 | Dropbox 型 | **不採用**（`retain_until` で必ずパージ） |

**実装メモ（S3）:** `customer.subscription.deleted` / `invoice.payment_failed`（最終）→ entitlement `expired` · RLS/Function で **INSERT/UPDATE 拒否** · SELECT は `retain_until` まで許可 · パージは既存 CF Cron。

### 3-1b. ダウングレード後の「ページ」— 何を見せるか（提督 Q&A · 2026-06-26）

**前提:** アカウント（`auth.users.id`）は残る · **退会（DELETE）とは別**。

| 画面 / URL | ダウングレード後（`retain_until` 前） | パージ後 |
|------------|--------------------------------------|----------|
| **ログイン** `/timeline/app/` | **可** — Auth は課金と独立 | **可**（アカウント存続時） |
| **ルーム一覧・中身の閲覧** | **可**（読み取り専用） | 該当ルーム行なし · 空一覧 |
| **保存・編集・新規ルーム** | **不可**（`expired` · quota） | 不可（trial 枠で新規は可） |
| **JSON エクスポート** | **可** | 不可（データ消滅） |
| **閲覧 URL** `/e/{id}`（S2） | **可** — **最終保存版の表示のみ** · ライブ同期は止める | **404** |
| **LP** `sync.sugudasu.com/` | 常時公開（課金無関係） | 同左 |

**業界慣行（リサーチ要約）:**

- **ログインは維持** — 解約＝アカウント消去ではない（MicroPIM · Kinde · StackBE 等）
- **期間末までフルアクセス** — Stripe `cancel_at_period_end` が標準
- **失効後は read-only grace** — 書き込み停止 · 閲覧・エクスポート可（7〜30日が多い · Sync は 14日案）
- **即ログイン拒否は避ける** — エクスポート・再課金導線が切れる

**Sync 固有:** 製品価値の大半は **当日のライブ同期**。`expired` では **閲覧 URL の Push/Pull を止め**、オーナー画面は読み取り専用。司会が古い版を「生きた同期」と誤認しないよう UI で区別（S2 実装時）。

**S1 現状:** RLS は entitlement 未チェック（オーナーなら読書き可）· **S3 で §3-1 の INSERT/UPDATE 拒否を実装**。

---

**§3-1 の grace · 枠 · バッファ性は運営・実装の正本。** 表立った FAQ や LP で **オペレーションの内訳を見せない**。

| 層 | 載せる | 載せない（運営内規のみ） |
|----|--------|--------------------------|
| **法務**（利用規約 · プライバシー） | 短期バッファ · 非永続の要約 · 削除権 | grace 日数 · quota 定数 · Webhook 段階 |
| **製品 UI**（文脈内） | ルームごとの **保持期限**（日付）· 削除確認 · ブロック時の **短い理由**（例: 保存できませんでした） | 「解約してもすぐ消えません」等の **ポリシー説教** |
| **LP / FAQ** | 価値（同期 · 共有 · 当日変更）· コア無料との差 · 返金は Stripe 規約に委ねる程度 | 同時1ルーム · 14日猶予 · JSON 自己保管の **運用マニュアル化** |
| **運営ドキュメント**（本書 · `SYNC_STORAGE_QUOTAS.md`） | 上記すべて | — |

**トーン:** ユーザーには **事実（日付・できない操作）** だけ見せ、**裏のオペレーション設計は見せない**。パージ前警告は **そのルームの期限が近いとき** のコンテキスト通知に留める（グローバル FAQ 化しない）。

## 4. 「アーカイブ」（要素 H）の再定義

| 誤解 | 正しい意味（Sync） |
|------|-------------------|
| クラウドに何年も残す | **×** |
| 翌年のテンプレとして使う | 終了前に **JSON エクスポート** → コア or 新規ルームへ取込 |
| Pro の履歴一覧 | **削除前のスナップショット数件**（メタのみ · payload は期限付き）— S3+ 検討 · 無期限は不可 |

## 6. コスト防衛

- `payload` jsonb は **ルームあたり1行** — ルーム削除でまとめて消える
-  revision 履歴テーブルは **S2 では作らない**（最新1版のみ · `sync_room_states`）
- 過去版が必要なら **エクスポート**（クライアント）— サーバに版歴を積まない
