## 2. 提督の意思決定（2026-06-17 FIX）

1. **対外キャンペーン = メイン / 社内イベント = サブ**（セグメントで分岐）
2. **証跡PDFに景品名・金額を載せる（一択）** — 監査・後追い説明のため。ローカル生成・外部送信なしを明記
3. **法令本文を読ませない** — e-Govリンクは載せるが、判断はツールのカード＋数値入力＋黄/赤フラグ
4. **LLMで合法/違法を判定しない** — ルール表（JSON）＋IF のみ。ChatGPT代替の「説明可能な一次スクリーニング」
5. **「合法です」「違法です」と断定しない** — 要確認 / リスク高 / 専門家確認推奨
6. 昔の社内揉め事の個別エピソードは提督も覚えていない → **公表事例ベースの汎用パターン（P01〜P15）** で十分

## 4. UI構成（2フェーズ・2セグメント）

### セグメント L1（`sg-segment`）

| 値 | ラベル |
|----|--------|
| `check` | 1. 景品リスク判定（推奨） |
| `draw` | 2. 公平抽選（名簿投入） |

**入口（案C · 2026-06-19）:** 同一 `fair-draw.html`。Hub から `?tab=check` / `?tab=draw` で初期タブを分岐。`URLSearchParams` で `initial` を決定し、パネル表示を同期。

**Out of scope（製品境界）:** 会場上映・ドラムロール・全画面プロジェクター演出（旧「宴会モード」）。公平抽選の結果は監査PDF/JSONで渡し、上映は PowerPoint・社内チャット画面共有・各社の粧飾に任せる。

**In scope（現場オペレーション層）:** 宴会演出ではなく **チャット・発表用テキストのワンタップコピー**（`warikan.html` の清算テキストと同型）。箱・抽選用紙の準備代替はここで完結する。

### 対象者（Phase 0 · Step 0 から自動判定）

UIタブは置かない。Step 0「キャンペーンの型」で **`internal_event`（社内ビンゴ・慰労会）** を選ぶと `internal`、それ以外は `consumer`。

| 値 | きっかけ |
|----|----------|
| `consumer` | デフォルト · 一般向けの型を選択 |
| `internal` | Step 0 で社内ビンゴ・慰労会 |

### Phase 0 — 景品チェック

```
Step 1  揉め事カード（P01〜P15）複数選択 — 法律用語なし
Step 2  数値入力（取引最高額・景品最高額・景品総額・当選人数・有償/無償・景品名）
Step 3  結果（黄/赤）＋ やさしい説明 ＋ 法令リンク ＋ Slack貼り付け用1段落
Step 4  証跡ダウンロード（PDF + JSON）
```

**PDFに必ず含める（提督FIX）:**

- 実施日時（JST）
- ツール名・バージョン（例 `SUGUDASU Lottery v1.0.0`）
- 入力した **景品名・金額・当選人数・キャンペーン概要**
- 選択したパターンID（P01…）
- 計算に使ったルール版日付（`prize-law-rules.json` の `version`）
- 乱数方式（Phase 1 実施時）
- 参加者リスト SHA-256（氏名を載せないオプションも将来可。v1は提督指示で氏名もPDF可）
- 黄/赤フラグと推奨アクション（専門家確認等）
- 免責全文

**赤フラグ残存時:** Phase 1「抽選」はデフォルト無効。「自己責任で続行」チェック＋ログに記録で解除。

### Phase 1 — 公平抽選

- 名簿テキスト貼付（1行1名 · SNS懸賞は `@ユーザー名` が典型）
- **Connpass CSV（Phase 1b · Q3 拡張）** — 下記 §4-1b
- **名簿の取得はスコープ外** — Xフォロー・リプライ等の応募者リストは幹事が手動コピー or 別ツールで作成し、fair-draw は貼り付け後の公平シャッフル＋監査PDFを担当（FAQ・名簿入力ヒント参照）
- 当選人数 · 重複当選可否 · 連続抽選（既当選除外）
- シード: 自動（`crypto.getRandomValues`）＋画面表示 · v2で手入力
- アルゴリズム: Fisher-Yates + Web Crypto
- 出力: 結果一覧 · 監査PDF · 検証JSON · **チャット発表用テキスト（コピー）** · v2で検証タブ

#### チャット・発表用テキスト（Phase 1 オペレーション · MVP）

- 公平抽選完了後、`warikan.html` と同型の **ワンクリックコピー**（Slack / Teams / LINE）
- 含める: イベント名（任意）· 当選者一覧 · 方式 · 名簿指紋（SHA-256 先頭）· ローカル完結の注記
- **含めない（v1）:** 検証URL（検証タブ P2-2 実装後に追記可）· 賞帯の自動整形（単一当選リスト。複数帯は幹事が手で見出しを足す）
- FAQ: 「大画面で発表したい」→ コピー文を社内チャットに投稿し画面共有（Wit 可）
- FAQ: 「Xで応募者IDはどう集める？」→ 本ツール外（手動 `@` コピー or 応募管理ツール）。名簿指紋は作成方法までは証明しない

#### SNS懸賞の名簿作成（製品境界 · Out of scope）

| 段階 | 誰がやる | 典型手段 |
|------|----------|----------|
| 応募収集 | 幹事 | 返信欄から `@` 手動コピー（小規模）· 懸賞管理SaaS / X API（中規模以上） |
| フォロー確認 | 幹事 or 別ツール | 手動は件数増で破綻しやすい · ツールのフォロー確認機能 |
| 公平抽選＋証跡 | **fair-draw** | 名簿貼付 → Fisher-Yates → 監査PDF |

X公式UIに返信者CSVエクスポートはほぼない。fair-draw は X ログイン・返信スクレイピングを **意図的に実装しない**（データ非送信・統制システムではない境界）。

#### §4-1b Connpass CSV 取込（Phase 1b · 2026 Q3 · GO拡張）

**正本:** [`data-finance-priority-gemini-RESULT.md`](data-finance-priority-gemini-RESULT.md) P0-2 · `BACKLOG.md` §1-15-6

| 項目 | 仕様 |
|------|------|
| 入力 | CSV **ドラッグ＆ドロップ** + 既存テキスト貼付 **併存** |
| 列選択 | ヘッダー解析 → ドロップダウン。**初期推奨: `表示名`** |
| 典型列（要実機確認） | `ユーザー名` `表示名` `メールアドレス` `受付番号` `ステータス` `アンケート回答内容` |
| 投影表示 | **メールアドレスは非推奨**（プロジェクター誤表示リスク） |
| 文字コード | UTF-8 優先 · 失敗時 Shift-JIS 手動トグル |
| 上限 | **5,000行** |
| 乱数 | `crypto.getRandomValues` + Fisher-Yates（**現行維持**） |

**証跡（新規実装ではない）**

監査 PDF · JSON · 名簿 txt は **既存**。CSV 経路追加時は以下を追記:

- `sourceFileName`（またはハッシュ）
- `selectedColumn`（抽選に使った列名）
- `rowCount`（有効行数）

**演出（Out / 代替）**

- ドラムロール · 全画面プロジェクター演出 — **Out**（§4 製品境界）
- 代替 UX（幹事が場を盛り上げる）:
  1. カウントダウン（3…2…1…）→ 一括表示
  2. 超高速テキストシャッフル数秒 → 停止
  3. **めくるカード** — 抽選は即確定 · 幹事がクリック/スペースで1人ずつ公開

**景表法 FAQ（断定弱め）**

> Connpass イベントで景品を提供する際、参加条件・抽選方法・景品価額によっては景品表示法の規制対象となる場合があります。本ツールは CSV からランダム抽出する機能のみを提供し、イベントの合法性や景表法適合性を判断しません。専門家確認・消費者庁ガイドラインを参照してください。

**差別化（Wheel of Names 等）**

- 完全非送信 · Connpass 列自動認識 · **既存3点証跡** · 日本幹事 UI

| 論点 | 実装メモ |
