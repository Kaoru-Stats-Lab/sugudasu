### 本文

スプレッドシートから1列コピーして、どこかの **全角半角変換サイト** に貼り、また Excel に戻す — この作業、地味ですが頻度は高いはずです。私も EC 登録や名簿整理で何度もやりました。経験があるほど **中身を見ずにコピー** しがちで、気づいたときには行がずれていたり、型番の長音がハイフンになっていたりします。

地味なプロダクトですが、**おっちょこ防止** に特化した文字列正規化ツール [SUGUDASU 正規化](https://sugudasu.com/normalize?utm_source=zenn&utm_medium=social&utm_campaign=article_12_normalize_oops) を個人開発しました。入力処理はブラウザ内のみで、名簿テキストを当方サーバーへ POST する API はありません（[プライバシーポリシー](https://sugudasu.com/privacy)）。

この記事では、**変換サイトで起きがちな3つの事故** のほか、よく聞かれる **「Excelの関数でもできるのに？」「なぜ500行？」「なぜ用途を3つに分けた？」** に答えます。EC 担当者向けの手順と、エンジニア向けの設計判断を同じ記事に載せます。

## 三大事故タイプ — あるあるの正体

Gemini に業務事故のパターンを洗い出させたところ、だいたい次の3型に集約できました（事例 ID は社内メモ `SUGUDASU_OOPS_GUARDRAILS`）。

| 型 | 典型 | 気づくタイミング |
|----|------|------------------|
| **① 行ズレ** | 空行削除 · セル内改行 · 分割貼付 | Excel に戻したあと、行数が合わない |
| **② データ変質** | NFKC一括 · 長音→ハイフン · 先頭ゼロ消失 | 登録エラー · 型番が壊れる |
| **③ クリップボードミス** | 変換前をコピー · 変換忘れ · 履歴ソフト | 貼った先で「さっきと違う」 |

「正規化」という言葉より、**行がずれない・意図しない変換をしない・コピー時に最新結果を出す** — この3つが実務の本丸です。

## 一般的なWeb変換ツールで起きがちなこと

特定サービス名は出しません。カテゴリとしてよくあるのは次です。

- テキストを **サーバーに送る**（機密列・会員ID列の心理的ハードル）
- **NFKC を一括** 適用し、ルール細部をユーザーに任せる
- 変換と同時に **クリップボードを上書き** し、確認チャンスがない
- 行数が変わっても **画面上で気づけない**
- 500行超を **無制限** に受け、ブラウザが固まる

どれも「悪い」とは言いません。用途で選ぶ話です。ただ、名簿1列を何度も往復する現場では、上のどれかに当たると **説明できないミス** になりやすいです。

ここで言う「行数不変」とは、正規化ルール上 **意図的に行を増減しない** 設計です。Excel 側の貼り付け形式やセル結合など、ツール外要因までは保証しません（[免責事項](https://sugudasu.com/disclaimer)）。

### 補足: NFKC とは？

**NFKC** は Unicode の互換正規化の一種で、見た目が似た文字を別の形に寄せます。便利な場面もありますが、**丸数字・記号・型番の長音** などを意図せず変えることがあります。normalize では全体適用を避け、英数字の全角→半角など **帯域を限定** しています。

## Excelの関数でもできる — ではなぜプロダクトにするのか

率直に言うと、**Excel だけで十分な場面はあります**。`ASC` / `WIDE` / `TRIM` / `SUBSTITUTE`、あるいは GAS の `replaceAll` で、シート内の補助列に変換式を置けば、多くのケースは処理できます。normalize は Excel の **完全代替** ではありません。ポジションは **Excel 列コピーの前段** — 貼付 → 変換 → コピーの **3秒ワンタップ** です。

それでもプロダクトにした理由は、次の4点です。

1. **往復の相手が Excel ではない** — EC 管理画面・モール登録フォーム・別 PC のスプシへ、**1列だけ** クリップボードで運ぶ場面。補助列を増やすほど「どの列が正本か」がブレます。  
2. **行数が見えない** — 関数の結果をコピーしても、貼る前に **入力 N 行 → 出力 M 行** を大きく出す UI は Excel 標準にはありません。おっちょこ防止の芯はここです。  
3. **ルールを毎回思い出さない** — 「ハイフン統一は ON？ 長音は守る？ 先頭ゼロは？」を毎回組み立てると、経験者ほど **うっかり強い設定** を選びがちです。  
4. **外部の変換サイトに貼らない** — ブラウザ内だけで完結し、名簿列を当方サーバーへ POST しません（[プライバシーポリシー](https://sugudasu.com/privacy)）。

Excel が得意なのは **シートの中で完結する変換** です。normalize が得意なのは **クリップボード往復の途中で、行数とルールを固定する** ことです。どちらか一方が正解、ではなく、**失敗の仕方と運用コストの置き場所** が違います。

## なぜ 500 行 cap なのか

無制限にすると、ブラウザ内 JavaScript だけで処理する設計と相性が悪いです。数万行の貼付は **固まり** の原因になり、行数チェックの意味も薄れます。

上限は **500行** に固定しました（改行 `\n` でカウント · 空行も1行）。理由は次のとおりです。

- **説明がブレない** — バイト数（32KB 等）だと、タブ入り列コピーで「何行まで？」の答えが変わり、FAQ が書きにくい。  
- **現場の1列コピーに合う** — Excel から1列まるごと貼る量なら、だいたい 500 行以内に収まる想定です。  
- **分割を促す** — 501行以上は変換不可にし、「2回に分けて貼ってください」と明示。巨大貼付のうっかりを止めます。

500行は「性能の限界ぎりぎり」ではなく、**軽量・説明可能・事故りにくい** 境界として置いています。数千行の列操作は、将来の CSV クレンザー（T06）の領域です。

## なぜ EC登録 / CSV名簿 / 全角英数 の3択なのか

正規表現やチェックボックス10個を並べると、読者は **ロジックを組み立てる人** になってしまいます。normalize ではユーザーが決めるのは **貼り先（用途）** だけです。裏のパイプライン順序はプロダクト固定です。

| プリセット | 想定貼り先 | ハイフン統一 | 意図 |
|------------|------------|--------------|------|
| **EC・フォーム登録用** | モール/自社ECの半角英数字段 | **OFF** | 型番・コードを壊さない。長音 `ー` も保護 |
| **CSV・名簿用** | スプシ再貼付 · 班分け/抽選の前処理 | **ON** | 行頭行末 trim · 行内スペースは維持 |
| **全角英数** | 帳票・見栄え | OFF | 半角→全角の **逆方向**（登録用とは別用途） |

分けた理由はシンプルです。**同じ「半角化」でも、貼り先によって壊れるものが違う** からです。

- EC 登録では、ハイフン統一 ON のまま貼ると **型番の `ー` や商品コード** が意図せず変わる（B02 事故）。  
- 名簿 CSV では、行内の姓名スペースを潰すと **列位置がずれる**（B03 事故）。逆に、記号ゆれのハイフンは揃えたい。  
- 全角英数は **登録用とは逆方向** なので、別プリセットにしないと毎回トグルを間違えます。

Outcome-first — **ルール10個より用途3択** — は、Excel の関数でも「毎回式を組み立てる」負担を、プロダクト側で肩代わりする設計です。

コピー時は **行数が一致しないとゲート** がかかります。一致していれば `Copied!` と先頭1行プレビューを出し、貼り付け先で「変な行数になっていないか」を最後にもう一度確認できます。

## 地味なツールを横展開する理由

normalize だけの話で終わらせません。行数表示やコピー契約は `sg-copy-feedback` として、割り勘・請求書・抽選など他ツールにも同型で載せています（[見積→請求の記事](https://sugudasu.com/invoice?utm_source=zenn&utm_medium=social&utm_campaign=article_12_normalize_oops) と同じ非送信の流れ、と捉えてもらえれば十分です）。

派手さはありません。ただ、**同じおっちょこを繰り返さない** ための共通 UI です。

## やらないこと

- 特定コンバータの名指し批判
- 「100%安全」「完全に同じ行数保証」（Excel 側要因は除く）
- 全文 diff · AI 判定 · CSV 列操作（別ツール T06 の領域）
- Phase B 未実装（差分ハイライト等）を **できる** と書く

## まとめ

全角半角変換は地味です。Excel の関数でも多くはできます。でも現場では、地味な作業ほど **行ズレとデータ変質** で止まります。normalize は流行語の正規化ツールではなく、**クリップボード往復の途中で、行数が見える · 用途でルールが固定される · コピーが最新結果になる** ためのおっちょこ防止 UI です。

名簿を外部に送りたくない・Excel 往復でミスを減らしたい — その線引きで使ってもらえれば十分です。
