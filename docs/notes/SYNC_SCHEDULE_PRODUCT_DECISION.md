## 0. なぜ Schedule を作るか（動機 · 提督）

**一言:** 書類と Excel の **コピペ転記** を減らす。監督が **新ツールを勉強する時間はない**。

### 0-1. 既存手段が解けない理由

| 手段 | なぜダメか |
|------|------------|
| **Notion（汎用）** | 汎用性が高いゆえ **作り込みが難しい**。現場・提出・週間・行政様式まで一気に揃えるのは監督の仕事ではない |
| **一般ガント SaaS / Excel ガント** | **書類が増えるだけ** — 提出用・現場用・週間・重機表が別ファイルのまま。**省力化になっていない** |
| **転記地獄** | 同じ工期・工種・日付が **コピペで何度も** 別フォーマットへ。データは似ているが **正本がない** |
| **フォーマット多様** | 役所指定 · 監理者の好み · 監督独自 · 工事種別（土木/建築/設備）で **様式がバラバラ** — 1つの固定テンプレでは足りない |

### 0-2. 我々が狙うこと

| 原則 | 製品への落とし込み |
|------|-------------------|
| **共通データの使い回し** | 1現場1 payload（工程表 DB）· 入力は1回 · **Export / preset** で提出・週間・現場を出し分け |
| **様式は Export 側** | 監理 A3 · 週間縦表 · xlsx — **正本は DB**、紙/PDF は書き出し（§3-2.1） |
| **学習コスト最小** | **Notion タイムライン（左表+右棒）** に寄せる — ゼロからの汎用 DB 構築ではない · 新概念を増やさない（依存は既定 OFF 等） |
| **監督は勉強しない** | Excel より複雑な操作を v1 に入れない · ゲスト・下請は **リスト/PDF** で足りる面がある（週間リスト設計） |
| **完全自動化は狙わない** | 行政様式・打合せ簿は硬い — **転記削減** が現実的ゴール（Gemini H4 · Pain Horizon） |

### 0-3. だから v3 は「Notion クローン」ではなく「Notion **タイムライン画面** の代替」

- Notion **ワークスペース全体** は作らない（作り込み地獄の再現を避ける）
- **表+棒+プロパティ+ビュー出し分け** だけに絞り、**書類枚数と転記回数** を減らす
- ガント単体アプリではなく **工程表の正本** — チャートは編集 UI、配布物は別 Export（週間リスト等）

### 0-4. 汎用ガントは選べない · 業界特化がやりやすい

| 観点 | Web / IT プロジェクト向けガント | 建設現場監督 |
|------|--------------------------------|--------------|
| **市場** | 製品が多すぎて **選定自体が負荷** | 同左 · かつ **誰も比較選定してくれない** |
| **情報の高低差** | フラット — 「載せたくないものは載せない」で回る | **提出 / 現場 / 下請 / 行政** で層が固定 · ユーザーが設計できない |
| **データの使い回し** | 自己設定 · チーム内に PM がいる | **IT リテラシー前提にできない** — 上手く使い回せない |
| **日常ツール** | ブラウザ SaaS · チケット | **CAD と Excel しか使わない**（不断） |
| **ナレッジ** | 社内にデジタル担当 · ペアで育成 | **教えてくれる人がいない** · 少数監督がペアで **タコツボ化** · ナレッジが組織に貯まらない |

**示唆:** 汎用ガントの「柔軟なフィルタ・カスタムフィールド設計」は **監督には渡せない**。  
**業界特化** = 提出/現場/週間の **preset・列・Export を製品側が決める** · 監督は **Excel に近い表** で触るだけ。

### 0-5. ペルソナ制約（設計の前提 · 提督）

| 前提 | 製品ルール |
|------|------------|
| IT リテラシー低 · 独学不可 | **オンボーディング最小** · 設定画面を増やさない · 初回から「現場1枚」 |
| 使えるのは Excel 程度 | **左表が主 UI** · 棒は補助 · 複雑なガント操作は v1 に入れない |
| データ設計はできない | **schema / preset は製品固定** — ユーザーがビューを「作る」Notion にはしない |
| ナレッジが個人に閉じる | **組織10席** で横展開 · 同じ Export ボタン · ドキュメントは Agent/SSOT 側 |
| ガント製品比較 fatigue | **名乗りは「工程表」** — ガント SaaS 比較表に乗せない（§1 名乗り） |
| **UI 文案** | **現場語は維持 · IT 語だけ隠す** — [`SCHEDULE_UI_COPY.md`](SCHEDULE_UI_COPY.md) D0〜D3 |

### 0-7. ブランドとペルソナの切り分け（提督 2026-06-29）

**コア SUGUDASU（IT リテラシー高め）と Schedule（現場監督）は、意図的に別ペルソナ。統一感より勝ち筋を取る。**

| | **コア `sugudasu.com`** | **Sync Schedule** |
|---|-------------------------|-------------------|
| **IT リテラシー** | 高め（Notion / スプレッドシート日常）— [`DESIGN_GUIDELINE.md`](../DESIGN_GUIDELINE.md) §1.1 | **低め** · CAD+Excel · 独学不可（§0-4〜5） |
| **スケジューラー** | 既に各自の好きなツールを持っている | **スコープ外** — 取りに行かない |
| **Schedule の勝ち筋** | — | 転記削減 · 業界 preset · **提督自身が使う**（dogfood 正本） |
| **ブランド統一** | 無料 · 登録不要 · 軽快 | **トーン・文案は別** — 色トークン `--sg-*` / Sync 節は共有（ADAPT） |
| **名乗り** | SUGUDASU + 各ツール | **工程表** — ガント SaaS 比較に乗せない |

**提督判断:** IT 高層向けに Schedule を寄せると **既存 Scheduler 競合の真っ只中** で勝てない。統一感が下がることは **許容** — 「私が使いたいから作る」が正当性。

**Agent:** コア hub のトーンを Schedule に流用しない · Schedule UI は [`SCHEDULE_UI_COPY.md`](SCHEDULE_UI_COPY.md) 優先。
