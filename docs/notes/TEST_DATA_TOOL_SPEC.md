## 1. Pain · ポジション

外部CSVダウンロード禁止 · Excel手打ち · 本番コピー禁止 — の現場で、**テスト・マニュアル用の整合済みダミー**をブラウザ内だけで作りたい。

**NOT:** 本番DBマスキング · 大規模TDM · 実在口座/マイナンバー  
**IS:** 顧客マスタ/取引明細風CSV · シード再現 · 社内ルール（ID/ドメイン）· 地雷混入

### 1.1 プロダクトアイデンティティ — 閉域 × ブラウザのみ（FAQ正本 · SEO）

**なぜ存在するか（1文）:** 閉域・持ち込み禁止・本番コピー禁止の現場で、**外部サービスにも怪しいCSV DLにも頼らず**、整合済みテストデータを**今日**作るため。

**強みの型（C1–C6 · [`TEST_DATA_CLOSED_ENV_RESEARCH.md`](TEST_DATA_CLOSED_ENV_RESEARCH.md)）**

| 制約 | 従来の困りごと | test-data の答え |
|------|----------------|------------------|
| **C2** 外部ファイル持ち込み禁止 | ネットのダミーCSVを落とせない | **ブラウザ内生成** — 入力データはサーバーへ送らない |
| **C3** 本番・個人情報の持ち出し禁止 | マスキング export も審査 | **架空データのみ** — 本番の代用ではない |
| **C4** クラウドSaaS禁止 | Mockaroo 等が使えない | **単一HTML + JS** — アカウント不要 · API不要 |
| **C5** 監査・再現性 | 「同じ手順で再テスト」 | **シード固定** — 同一設定＝同一CSV |
| **C1/C6** 閉域網・顧客VPN内 | オフラインでもページが開ければ動く | **初回ロード後は生成処理がオフライン完結**（CDN要確認 · `statements` 参照） |

**SEO・文案で使う語（自然に）:** 閉域環境 · テストデータ · ダミーデータ · ブラウザのみ · 非送信 · 社員マスタ CSV · 給与テスト · インポート検証

**画面FAQ（`tools/test-data.html`）が正本。** 仕様は上記の骨子のみ保持。競合比較は FAQ「他のダミーデータ・テストデータ生成ツールと何が違いますか？」（5行比較表）。固有名は Mockaroo のみ（1回）。

## 6. データセット依存あるある — 取り込み判断

人事・給与系開発で「**テストの成否がデータの作り方に依存する**」パターン。  
SUGUDASU は **50型ビルダー全面競合はしない** — 楔（労務SMB）で **よく事故る・説明しやすい・単一HTMLで届く** ものだけ入れる。

**判断軸（4つ）**

| 軸 | 入れる |
|----|--------|
| **頻度** | 受託SI・社労士・情シス兼務が月1回以上踏む |
| **事故率** | 本番コピー禁止・Excel手打ち・インポート失敗で定時が溶ける |
| **再現** | シード＋地雷で「同じバグを説明できる」（C5） |
| **実装** | 単一HTML · コンプラ安全 · 過度なドメイン専門知識不要 |

**凡例:** ✅ 済 · 🔜 次 · 📋 ドキュメントのみ · ❌ 意図的に入れない

### 6.6 全角半角整え（`normalize`）との切り分け — **2026-07-03 FIX**

**結論:** test-data と normalize は **別ツール・別ペルソナ**。大規模データセット（§7–§8）と normalize を **1本のパイプラインとして売らない**。

| 観点 | test-data | 全角半角整え（`normalize`） |
|------|-----------|------------------------------|
| **主用途** | インポート試験 · ページング · 件数スケール | Excel **1列コピー**のクリップボード往復の安全装置 |
| **規模** | 100〜**250,000**（社員マスタ · CSV DL） | **500行/回**（`LINE_LIMIT` · 仕様は `NORMALIZE_TEXT_TOOL_SPEC.md`） |
| **出力の行き先** | SaaS / DB / バッチへ **CSV直投入** | スプシ再貼付 · EC登録 · 班分け前の**手作業** |
| **想定ユーザー** | 受託SI · 情シス · QA | 総務 · EC登録 · 名簿整備（非エンジニア寄り） |

**なぜ相性が悪いか**

- 25万件を normalize に通すには **500行×500回** の分割が必要 — 設計外。
- エンジニアの大規模試験では、全半角変換は **生成時に仕様どおり出す**か、**自前スクリプト/ETL** で足りる。normalize handoff は本筋の助けにならない。
- normalize の 500行 cap は性能限界ではなく **説明可能・事故りにくい境界**（Zenn #12）。数千行以上の列クレンジは将来 **CSVクレンザー（T06）** の領域。

**製品方針**

| やる | やらない |
|------|----------|
| **≤500行**の社員サンプルを事務職が触るときだけ「全角半角整えで開く」（handoff · 既存） | 大規模CSV（5,000件超 · 特に25万）を normalize 連携前提にしない |
| FAQ / UI で **切り分けを明記** | 「test-data → normalize で25万を整える」系の文案 |
| **生成時オプション**でフリガナ半角・日付統一を足す（バックログ優先 · §6.5-4） | normalize の cap を test-data の都合で引き上げない |

**FAQ文案（`test-data` 画面 · 正本）**

- 「全角半角整えへ渡せる？」→ **500行まで**（ヘッダー含む）。**大規模エクスポート（2.5万〜25万）は CSV ダウンロードが正** — normalize とは連携しない。
- 「フリガナを半角にしたい（25万件）」→ 生成オプション（将来）またはインポート先の変換ルールで対応。normalize 一括は想定外。
- 「班分け前に名簿を整えたい」→ **100〜500件**を test-data で作り handoff、または Excel 列を normalize に直接貼る — **こちらが normalize の本線**。

#### 6.7.3 人材派遣 — 3層モデル（仕様メモ）

派遣領域を扱うとき、**社員マスタ1枚に「所属会社」列を足すだけでは足りない**。主体が3つに分かれる。

```text
【派遣会社のシステム】
  ① スタッフマスタ（人）     … 氏名 · 社保 · 給与 · 雇用形態 · 自社支店
  ② 派遣先マスタ（法人）     … 客先企業 · 就業場所 · 契約条件
  ③ 配属・契約（トランザクション） … スタッフID × 派遣先ID × 期間

【派遣先（ユーザー企業）の人事】
  ・自社社員マスタ           … 正社員 · パート
  ・派遣者一覧（別カテゴリが多い） … 派遣元会社名 · 配置部署 · 契約期間
  ・取引先マスタ             … 派遣会社を仕入先として持つ場合あり（調達レーン · 人事CSVとは別）
```

| 「所属会社」を入れたい意味 | 置き場所 | 楔での優先 |
|--------------------------|----------|------------|
| 派遣会社の **自社支店・営業所** | スタッフマスタの列 | 中（一般の事業所と同型） |
| 派遣先の **配置部署** | 派遣先企業の人事 · 配属 | 中 |
| **派遣元会社名**（誰の派遣か） | 派遣先側の派遣者一覧 · または1列オプション | 低（ペルソナ要ヒアリング） |
| **派遣先法人** | **別マスタ**（②） | 低〜別プリセット |
| **配属期間・契約** | **別CSV**（③） | v0.6+ 候補 |

**社労士・給与SaaS楔（P1）でMustなのは:** 雇用形態に **「派遣」** があること（現行 `派遣スタッフ` ラベル）と給与分岐。**派遣先法人・配属全体**までは通常スコープ外。

**現行実装（v0.5）:** `EMPLOYMENT_TYPES` に `派遣スタッフ`（約9%）のみ。**派遣先 · 配属 · 所属会社列は未実装** — 一般企業の給与インポート試験としては意図的に浅い。

### 6.8 閉域利用運用 — ホワイトリスト · 広告 · 社内ミラー — **2026-07-03**

**結論:** ダミーCSVを「作れる」だけでは閉域現場は通らない。**承認 → 初回取得 → 生成 → 取込 → 廃棄**まで含めた運用設計が必要。ホワイトリストは **貴社情シスが決める** もので、SUGUDASU が代行承認するものではない。

#### 6.8.1 エンジニア側に「型」はあるか

**ある現場ではある · 全員が頭に入っているわけではない。**

| 層 | 典型の頭の中 | 非エンジニア現場 |
|----|--------------|------------------|
| **受託SI · 社内情シス** | URL申請 · プロキシ例外 · DevTools確認 · 社内ミラー | 申請書の書き方がわからない · 「非送信」と「外部通信なし」を混同 |
| **小規模事務所** | ほぼなし | 「閉域OK」の文言だけでは稟議が通らない |

エンジニアがやることの**よくある順序**（製品がサポートするのは **④の根拠提示** まで）:

1. **規程確認** — 外部Webツール可否 · 未分類ドメイン · USB/持ち込み
2. **申請** — URL · 業務目的 · **入力データをPOSTしない** · 確認手順（`statements#for-it-security`）
3. **ドメイン許可** — プロキシ/ファイアウォールで **ホスト単位**（広告カテゴリは別判定になりがち）
4. **技術確認** — DevTools Network で生成時 POST なし · 必要なら機内モードで再実行
5. **代替** — エアギャップなら `dist/` の **社内静的ホスト**（§6.8.4）

#### 6.8.2 広告とホワイトリストの関係

| 事実 | 説明 |
|------|------|
| **コア（sugudasu.com）** | 無料運営のため **広告枠あり**（現状はプレースホルダ · AdSense 本番タグは未設置） |
| **Sync（sync.sugudasu.com）** | **広告なし**（別ライン · 登録 · 有料）— test-data 楔とは別判断 |
| **レイヤ分離** | 広告は [`statements#verify-layers`](../../tools/statements.html) の **③**。**① 生成処理には不要** |
| **ブロック時** | 広告ドメインだけ止まっても **CSV生成は動く** 想定（枠が空 · フォント欠けはUI劣化のみ） |
| **将来** | AdSense 本番化で `googlesyndication.com` 等が増える · **Pro（広告非表示）** で申請が楽になる余地 |

**情シスに伝えるときの芯:** 「テストデータ生成に広告通信は**必須ではない**。ページ本体（`sugudasu.com`）の静的配信が許可されれば足りる。広告カテゴリは組織ポリシーでブロック可。」

#### 6.8.3 申請メモ用 — 本番ビルドで触りうる外部先

`npm run build:pages` 後の **本番 dist** 想定（開発用 HTML の Tailwind CDN は **含まない**）。

| 優先 | ホスト | 用途 | 生成に必須？ |
|------|--------|------|--------------|
| **Must** | `sugudasu.com` | HTML / JS / CSS / 辞書JSON | はい（初回取得） |
| **Should** | `fonts.googleapis.com` · `fonts.gstatic.com` | 見出しフォント | いいえ（崩れても生成可） |
| **Optional** | AdSense 系（承認後） | 広告表示 | **いいえ** |
| **Optional** | アクセス解析（導入時） | 集計 | **いいえ** |

貼り付け用の要約は [`statements#for-it-security`](../../tools/statements.html) の blockquote を正本とする。

#### 6.8.4 社内ミラー（エアギャップ向け）

インターネット自体が使えない現場では **ホワイトリストではなく配布** が現実解。

```text
リポジトリ clone（社外）→ npm run build:pages → dist/ を社内Webサーバ or 検証用PCへ
→ ブラウザで https://intranet.example/test-data/ を開く → 以降は社内のみ
```

- 広告枠はミラーに **載せない**（静的ファイルのみ）
- CSV の **社内保管・廃棄** は利用者責任（§6.4）
- 手順の詳細は情シスと合意 · 本仕様は**方針のみ**

#### 6.8.5 FAQ 正本（`test-data` 画面）

- 「ホワイトリストに載せられますか？」→ **貴社申請**。上記 §6.8.3 · `statements` 申請メモを添付。
- 「広告があると使えませんか？」→ **生成には不要**。広告だけブロックされても可 · 全面ブロックなら社内ミラー。
- 「エンジニアは何をすればいい？」→ §6.8.1 の 1〜5 · DevTools 確認。

## 7. 件数スケール — 3層上限とドメイン根拠

**問題:** 現行 `MAX_ROWS = 5000` が「1回の出力上限」と「データセット全体の上限」の両方に見える。  
**方針:** 層を分離する。組合加入率は長期下降（令和7年 **16.0%**）しており、**歴史ピークを超えて急伸する想定はしない**。統計の正本は **労働組合基礎調査**（最新概況）とする。

**統計 SSOT（優先順）**

| 資料 | 用途 |
|------|------|
| [**労働組合基礎調査 概況（令和5年度版 PDF）**](https://www.mhlw.go.jp/toukei/itiran/roudou/roushi/kiso/25/dl/gaikyou.pdf) | **全国**の組合数・組合員数・推定組織率の推移（各年6月30日）— **現代の正本** |
| [主要団体別労働組合員数 附表（平成11–12年）](https://www.mhlw.go.jp/www2/kisya/daijin/20001219_02_d/20001219_02_d_fhyou.html) | **業界別単産連合**ごとの員数（単位：千人）— 単一**団体**規模のアンカー |

### 7.1 三層モデル

| 層 | 定数名（案） | 現行 | 目標 | 意味 |
|----|--------------|------|------|------|
| **A** 1チャンク | `CHUNK_MAX` | **5,000** | 5,000〜50,000 | プレビュー · メモリ · UI応答（1回の生成） |
| **B** 1ダウンロード | `DOWNLOAD_MAX` | = A | 論理上限まで | チャンク連結 · IDオフセット · Blobストリーム |
| **C** 論理上限 | `DOMAIN_MAX` | 未定義 | 下表 | プリセット別「日本の労務・給与であり得る最大」 |

**鉄則**

- 画面の「5,000件」は **A だけ** を指す（[test-data](https://sugudasu.com/test-data) FAQ で明記予定）。
- 「あと5,000件ユニークに」は **B**（オフセット + シード分離）。
- **C** を超える要求はスコープ外（全国組合員**合計**約1,000万人の**1CSV社員マスタ**など · §7.2）。

### 7.2 全国規模の統計（基礎調査 — 現代）

[概況 PDF](https://www.mhlw.go.jp/toukei/itiran/roudou/roushi/kiso/25/dl/gaikyou.pdf)「労働組合種類別…推移」より（**単一労働組合**の組合員数 · 各年6月30日）:

| 年 | 単一労働組合員数 | 推定組織率 | メモ |
|----|------------------|------------|------|
| **令和7年（2025）** | **9,926,971人**（≈993万） | **16.0%** | 最新 |
| 令和元年（2019） | 10,087,915人（≈1,009万） | 16.7% | 近年ピーク |
| 平成6年（1994） | **12,698,847人**（≈1,270万） | 24.1% | **調査系列上の最大** |
| 昭和22年（1947） | — | 45.3% | 組織率は以降長期低下 |

**読み方（テストデータ設計）**

- 全国の単一組合員 **合計≈1,000万人** は、**2万超の組合の統計合計**であり **1使用者・1インポート用の社員マスタではない**。
- 推定組織率は **45%→16%** と長期低下 — **1994年≈1,270万人を統計上の天井**とすれば足りる。
- 連合（総連）加盟 **約700万人** は中央組織の話。**全国≈1,000万**（基礎調査）の方が現代の全体像として正確。

### 7.3 論理上限 `DOMAIN_MAX`（プリセット別）

**単一使用者（1給与テナント）の実務天井:** **JP労組**（日本郵政グループ労働組合）— 国内最大の**単一労働組合** · 組合員 **約21万〜24万人** · 日本郵政グループ関連の正社員中心 · **正規社員の加入率9割超**。これより高い組織率・単一組合規模は想定しない → **社員マスタの実用上限はここ**。

| プリセット | `DOMAIN_MAX` | 根拠（アンカー） |
|------------|--------------|------------------|
| **社員マスタ**（単一使用者 · 1テナント） | **250,000** | **JP労組** 約21万〜24万人 + 実装・試験用の丸め余裕（v0.5 `DOMAIN_MAX`） |
| **社員マスタ**（業界単産連合・横断団体） | **1,000,000** | 平成12年附表 **自治労 1,007千人** 等 — **1社インポートではなく**業界連合単位の参考値。[附表](https://www.mhlw.go.jp/www2/kisya/daijin/20001219_02_d/20001219_02_d_fhyou.html) |
| **社員マスタ**（全国統計合計・横断1ファイル） | **対象外**（参考天井 **≈1,270万**） | [基礎調査](https://www.mhlw.go.jp/toukei/itiran/roudou/roushi/kiso/25/dl/gaikyou.pdf) 単一労働組合員数系列最大（1994年）— **1CSVではない** |
| **給与明細（行）** | **10,000,000** | 行＝社員×月。25万人×12ヶ月×3年≈900万行 — **v0.5スコープ外**（社員マスタ先行） |
| **顧客マスタ** | **1,000,000** | 金融楔 · 件数試験の余裕値 |
| **取引明細** | **10,000,000** | 明細系ソート · 集計 · インポート上限試験 |

**注（附表・千人単位）**

- 附表は原則 **1万人（10千人）以上** の組合のみ（附表注1）。**単一団体**の規模試験用。
- 現行コードは `MAX_ROWS = CHUNK_MAX = 5000` のみ実装。`DOMAIN_MAX` は **仕様正本**（実装は §7.5）。

### 7.4 ユーザーが5,000件のあとに欲しがるもの（B層 Must）

| 機能 | 理由 |
|------|------|
| **IDオフセット** | 2バッチ目を `5001` から — 社員番号ユニーク |
| **シード分離** | 氏名・給与の中身はバッチごとに変える |
| **進捗表示** | 例: `5,000 / 120,000` |
| **結合DL** | 1 CSV または `part-001.csv` 連番 |

シードだけ変えて同じ `EMP-2026-0001` を繰り返すのは **ユニーク制約テストに使えない**（現状の罠）。

### 7.5 実装ロードマップ

| 版 | 届く規模 | 内容 |
|----|----------|------|
| **v0.4.x** | 5,000 | 現行 · `CHUNK_MAX` |
| **v0.5** | **250,000（社員マスタのみ）** | IDオフセット · 5,000×50チャンク結合 · **UTF-8 BOM CSV** 一括DL（§8） |
| **v0.6** | 250,000+ | Web Worker · Blobストリーム · ZIP · 給与明細大規模は別版 |
| **v1.0** | `DOMAIN_MAX` 全体 | 中断再開 · プリセット別上限の完全実装 |

**エンジン定数（将来）:** `CHUNK_MAX` · `DOMAIN_MAX_BY_PRESET` を `test-data-engine.js` に明示 export。

### 7.6 FAQ / 文案（追記予定）

- 「10,000件欲しい」→ チャンク連続生成（v0.5+）· 今はプレフィックス変更で2ファイル
- 「100万人の社員マスタ」→ **1テナントならJP労組規模（上限25万）** · 100万は業界単産連合の参考値
- 「1,000万人」→ 全国組合員の**統計合計**（基礎調査）の話で **1社インポート用マスタではない**
- 「組合員はこれから増える？」→ 推定組織率は長期 **低下**（令和7年16.0%）— 1994年≈1,270万人を統計天井とする
- 「全角半角整えと併用？」→ **§6.6** — 大規模CSVは normalize 非連携 · handoff は500行サンプルのみ

## 8. v0.5 — 大規模社員マスタ（25万件）

### 8.1 Must（スコープ固定）

| 項目 | 内容 |
|------|------|
| **対象プリセット** | **社員マスタのみ** |
| **対象外** | 給与明細25万×3ヶ月＝75万行（v0.6以降） |
| **件数** | 最大 **250,000**（`DOMAIN_MAX` · §7.3） |
| **生成** | `CHUNK_MAX` 5,000 · **IDオフセット**（`startIndex`）で社員番号ユニーク |
| **RNG** | 行番号ベース（バッチ分割でも行 `i` の中身が一意に決まる） |
| **プレビュー** | 先頭チャンク10件のみ（25万行はRAMに載せない） |
| **非送信** | 現行どおりブラウザ内完結 |

### 8.2 アウトプット形式（エンジニア向け）

労務・給与の**一括インポート試験**が主用途。拡張子より **列名・日付・郵便番号クォート・シード再現** が価値の中心。

| 優先 | 形式 | 件数 | 用途 |
|------|------|------|------|
| **Must** | **CSV（UTF-8 BOM）** 1ファイル | 〜25万 | 給与SaaS · DB · バッチのインポート試験 · diff/CI |
| **Should** | **ZIP**（中身は上記CSV、または `part-001.csv` 連番） | 〜25万 | 添付上限 · 部分再取込 · 「ファイルが大きい」対策 |
| **Later** | **xlsx** | **≤5,000** のみ | 目視確認 · マニュアル用サンプル（§6.3） |
| **Out** | JSON · SQL · Parquet | — | 一部開発者向けだが楔の主戦場外 |

**区切り文字:** カンマ（現行維持）。TSV要望が複数来たらテンプレ追加で検討。

**結合DL:** 5,000行×50チャンクを **ヘッダー1回 + データ行追記** で1 CSV にまとめる（§7.4）。

### 8.3 Excelで開ける限界（参考 · FAQ用）

エンジニアは多くの場合知っているが、**25万件をExcel前提にしない**ため明記する。

| 区分 | 目安 |
|------|------|
| **Excelの理論上限** | **1,048,576 行/シート**（Excel 2007以降）· 列は 16,384 列（XFD列）まで |
| **実務上の快適ゾーン** | **〜数万行** — それ以上は起動・スクロール・フィルタ・保存が著しく重い |
| **25万行をExcelで開く** | 理論上は可能だが **非推奨** — メモリ・待ち時間・フリーズのリスク。目視したいなら **5,000件以下** を生成するかCSVの先頭だけ切り出す |
| **CSVをダブルクリックでExcel** | 文字コード・郵便番号先頭ゼロ・日付化けのリスク — **インポート試験はSaaS/DBへCSV直投入**が本筋。Excelで見る場合は「データの取得」からUTF-8指定（§6.4） |

**SUGUDASUの方針:** v0.5の25万は **CSV一括DL** を正とする。xlsxは将来 **CHUNK_MAX以下** のサンプル用途に限定する。
