### 本文

プログラミングを仕事にしているわけではありません。HTML でウェブサイトを組んだ経験や、WordPress で **できあがったサイトをメンテする** 程度はあります。流行りの Python も少しかじりましたが、**プロダクトは何も作らず** でした。

営業・マーケの立場では、同じフロアの隣の島にいるフロントエンドやバックエンドのエンジニアと、サバ缶を突き合わせて話すことが多くありました。担当プロダクトの **本質 · 仕組み · FAQ · 刺さるポイント · オペレーション** は現場から熟知していたので、「こういう機能が欲しいが、何がネックなの？」「DB ってどう動いているの？」「なら、こういうことはできる？」と聞けました。エンジニアの考え方を知ること自体が、営業・マーケとして有意義な経験でした。**書くのは向こう、要件と制約を引き出すのはこちら** — その経験がなければ、今の個人開発はかなり難しかったと思います。

そのうえで、事務の現場で「また同じ書類を打ち直す」と感じたのが、実務ツール [SUGUDASU](https://sugudasu.com/?utm_source=zenn&utm_medium=social&utm_campaign=article_02_hub_devstory) を個人開発したきっかけです。今回は **エンジニアの隣に座っていた側が、自分で書く側に回った** 、という話です。

「プログラミング未経験が一夜にアプリを作った」という話ではありません。**ゼロではないが、プロダクト開発は別物** — そこに AI を **誰が何を書くか** 分けて並走した、という話です。

| 経験としてある | 今回の SUGUDASU で初めて深くやった |
|----------------|-----------------------------------|
| 営業・マーケとしてエンジニアと要件のすり合わせ | 自分の手でコードを書いて出荷する側 |
| HTML · 静的ページの感覚 | `build-pages.mjs` · 複数ツールの一貫ビルド |
| WordPress 完成サイトのメンテ | バニラ JS · ブラウザ内処理の設計 |
| Python を少しかじった（成果物なし） | changelog 運用 · Zenn 連載とコードの突合 |

最初の記事では見積の下書きを請求書に転用する手順（invoice）を書きました。そのあと正規化 · 班分け · 非送信設計 · 抽選など、**ツール別の実務記事** を出しています。今回はその裏側 — ツール群の配信と、改善の回し方 — です。非送信の設計詳細は別記事（#6）に任せ、ここでは **AI の分担と運用** に絞ります。

## AI の使い分け — 企画と実装を混ぜない

私が使っているのは、ざっくり次の分担です。

| 役割 | 向く作業 | 向かない作業 |
|------|----------|--------------|
| **Gemini（企画セッション）** | Zenn テーマ表 · 3ヶ月スケジュール · 比較表 | 記事本文 · HTML/JS の実装 |
| **Cursor / Claude（実装セッション）** | コード · changelog 突合 · ドラフト執筆 | 礼賛リライト · 未実装機能の宣伝 |
| **別ツール（監査）** | タイトル短文化 · 冗長削除 | 事実の追加 |

Zenn 連載のテーマは、Gemini に **「表と見出し案だけ」** 出させ、本文は Cursor 側で書いています。企画プロンプトには「記事本文の執筆は禁止」と明記しています。一度、企画と実装を同じチャットに混ぜたとき、存在しない機能まで記事に載りかけたので、役割を分けました。

「魔法のプロンプト1本」より、**出力フォーマット（§1〜§7 の表だけ、など）** と **禁止リスト（礼賛 · 競合名指し · 未公開数字）** の方が効きました。

## Cloudflare Pages を選んだ理由 — サーバーより「預けない」

SUGUDASU は **静的サイト** です。`build-pages.mjs` が HTML・`sitemap.xml`・`robots.txt` を `dist/` に生成し、Cloudflare Pages が配信します。見積ツールの入力は **ブラウザ内の JavaScript** が処理し、下書きは `.json` ファイルとして手元に保存します。

### 補足: 静的サイト（SSG）とは？

ページを **アクセスのたびにサーバーで組み立てる** のではなく、**公開前に HTML を焼いておく** 方式です。個人開発では、サーバー代を抑える以前に、**ユーザーの会社名や金額を自社 DB に預けない** 設計を優先しました。現状、Pages Functions（サーバーレス API）は使っていません。機能追加はまず「ブラウザで完結するか」で判断しています。ブラウザの通信レイヤや DevTools の見方は、非送信設計の記事で詳述しています。

## Form → changelog — 要望を製品に戻す

改善の窓口は [更新履歴ページ](https://sugudasu.com/updates) の Google Form です。届いた要望を `data/changelog.json` に載せ、ビルドでサイトへ反映します。

たとえば 2026-06-17 には、見積で保存した下書きを納品書・請求書に転用できる旨を、画面上・FAQ・読み込み後の案内に追記しました。ユーザーから「転用のやり方が分かりにくい」という声があれば、まず changelog に事実として残し、必要なら Zenn や X で補足します。

有料のプレスリリース配信は、収益化前の段階では見送りました。代わりに **Zenn（設計・実務の話）** と **X（短い課題フック）** で認知を積み上げています。派手さより、**changelog が動いていること** が信頼になると考えています。

```json
{
  "date": "2026-06-17",
  "type": "improve",
  "title": "見積・請求書：下書きの納品・請求への転用を案内",
  "body": "見積で保存した下書きデータを読み込み、書類種別タブで納品書・請求書に切り替えて転用できる旨を追記しました。",
  "tools": ["invoice.html"]
}
```

## まとめ

エンジニアではない立場で AI と並走するコツは、AI に全部任せないことです。**Gemini には企画表、Cursor には実装と記事、監査は別セッション** — かつてエンジニアに「何がネックか」を聞いていたのと同じく、**誰に何を聞くか** を分けています。HTML や WordPress の経験に加え、プロダクトを現場から知っていたことが土台になりました。完成品ではなく、実務で使いながら直しているポータルです。ツール別の話はこれまでの連載、設計の骨格は #6、今回はその **作り方のループ** です。
