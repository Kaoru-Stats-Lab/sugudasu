## 1. 3層（対になる構造）

| 層 | 画面 | 時間 | ユーザーが知りたいこと |
|----|------|------|------------------------|
| **更新履歴** | `/updates` | **過去**（Shipped） | 何が変わった？信頼できる？ |

## 2. 更新履歴（changelog）の粒度

### 2層データ（MECE 台帳 + public pickup）

| 層 | `audience` | 載せ方 | 画面 |
|----|------------|--------|------|
| **MECE 開発台帳** | `internal` | コミット・ファイル・実装詳細を漏れなく追記 | 非公開（JSON のみ） |
| **ユーザー向け pickup** | `public` | 体感単位 · registry tool id のみ | `/updates` |

- **原則:** デプロイごとに `internal` を必ず追記（MECE）。ユーザーが気づく変更だけ `public` を追加または同日バンドル。
- **`rollup`:** internal 行がどの public 要約に束ねられたか（public 側の `id` を参照）。
- **`id`:** public 要約用の安定キー（例: `layout-refresh-20260703`）。
- **検証:** `npm run validate:changelog` — `audience: public` のみ粒度ルールを強制。`audience` 未設定は legacy として WARN。

### 載せる（public pickup）

- 新ツール · 新プリセット · 計算ロジックの変更
- ユーザーが**画面で気づく** UX 改善 · 不具合修正
- 帳票・印刷・コピー挙動の変更
- 料金 · 制限値（件数 cap 等）の変更

### 載せない（Too Much）

| NG 例 | 理由 |
|--------|------|
| `sg-main-shell` · `sg-info-page` 等の **CSS クラス名** | 実装詳細。ユーザーには無意味 |
| `sugudasu.css` · `*.md` 単体 | ファイル名の羅列 |
| Agent 用 SSOT 追加のみ | 開発者向け |
| デプロイ台帳 · CI | 運用内部 |
| 同一日の細切れコミット | **1デプロイ＝原則1エントリ**にバンドル |

### タイトル · 本文の書き方

- **タイトル:** ツールの概念名 + 何が変わったか（例: 「全角半角整え — SQL IN 句プリセット」）
- **本文:** 1〜3文 · **体感**（誰のどんな作業が楽になるか）
- **`type`:** `feature`（新機能 · 新設・α）· `fix` · `improve`。**`feat` は禁止**（本番で「改善」表示になる）
- **`tools` 配列:** registry の **tool id** のみ（`normalize` · `shift`）。`*.html` · `*.css` · パス禁止
- 横断リリースは **1エントリにまとめる**（例: 「サイト全体のレイアウト改善」）

### 悪い例 → よい例

| 悪い | よい |
|------|------|
| `sg-main-shell` を全ツールに適用 · `PAGE_LAYOUT_SELECTOR.md` 追加 | ツールページとガイドの幅・余白を統一し、読みやすくしました |
| `mask-app.js` オーバーレイ確定 | マスク — 黒塗りが確定しない不具合を修正 |
| 12件を同日に細切れ | その日の本番反映を **機能単位で最大3件** まで（それ以上は週次まとめ） |

## 3. ロードマップ（予定 · 検討中）の粒度

### 載せる

- **予定（scheduled）:** 次の開発サイクルに入れた機能 · ツール横断 OK
- **検討中（considering）:** FB で優先度が上がりうる候補
- 各項目に安定 **`id`**（アンカー `roadmap.html#id`）

### 載せない

- 実装タスク（「CSS 変数追加」）
- 日付の確約（「7月中必須」）
- 個人名 · 却下された個別要望

### リリースしたら

1. `roadmap.json` から該当 `id` を **削除**（または `shipped` にし JSON から外す）
2. `changelog.json` に **ユーザー向け1エントリ** を追加
3. 必要なら roadmap 項目に一時 `shippedChangelogDate` は使わず、changelog のみ正本

## 4. 対象外レーンの粒度

### 載せる

- **プロダクト非目標**（病院24/365看護 · 多店舗勤怠 · 労基自動判定）
- **業態名 · 用途名** を明示（SEO · 誠実線）
- **代替**（ジョブカン · 専用 SaaS 等）を1行

### 載せない

- 特定ユーザーへの言及
- 「永遠にやらない」と断言しすぎる法務的拘束（「意図的にスコープ外」で十分）

## 5. Agent · 提督の作業チェックリスト

**changelog 追記前**

- [ ] MECE 用に `audience: "internal"` 行を追記したか？（ファイル名・実装詳細 OK）
- [ ] ユーザーが画面で気づく変更は `audience: "public"`（または同日バンドル）を追加したか？
- [ ] public 行から CSS クラス · ファイルパス · 内部 doc 名を除いたか？
- [ ] public の `tools` は registry id のみか？
- [ ] internal に `rollup` がある場合、public 側 `id` は存在するか？

**roadmap 追記前**

- [ ] `id` は kebab-case で安定か？
- [ ] 予定 / 検討中 / 対象外 のどれか明確か？
- [ ] 対象外は**領域**単位か（個別 FB ではないか）？

**デプロイ後**

- [ ] shipped した roadmap 項目を JSON から外したか？
- [ ] changelog 1件追加したか（ユーザー向け文言）？
