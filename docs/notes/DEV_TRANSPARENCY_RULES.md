## 0. 一言（changelog vs /updates）

| 置き場 | 何を載せるか | 誰が見るか |
|--------|--------------|------------|
| **`data/changelog.json`** | **すべての変更**（MECE 台帳）。CSS・SEO・FAQ幅・ファイル名・デプロイも OK | Agent · 提督 · 開発ログ |
| **`/updates`** | そのうち **`audience: "public"` だけ** | エンドユーザー |

**結論:** 全部載せるのは JSON。ユーザー画面には「仕事が変わるもの」だけ。片方を削って片方に寄せない。

## 1. 3層（対になる構造）

| 層 | 画面 | 時間 | ユーザーが知りたいこと |
|----|------|------|------------------------|
| **更新履歴** | `/updates` | **過去**（Shipped） | 何が変わった？信頼できる？ |
| **予定 · 検討中** | `/roadmap`（コア · Sync レーン） | **未来** | 開発スケジュールに入っている？ |
| **対象外** | `/roadmap`（対象外レーン） | **方針** | やらないと決めた領域は？ |

**対ナビ:** 両ページ共通の `.sg-dev-transparency-nav`（過去 | これから）。

**載せないもの（別台帳）:** 個別 FB の却下理由 · スプシ生データ → `FEEDBACK_TRIAGE.md` のみ。

## 2. 更新履歴（changelog）の粒度

### 2層データ（MECE 台帳 + public pickup）

| 層 | `audience` | 載せ方 | 画面 |
|----|------------|--------|------|
| **MECE 開発台帳** | `internal` | コミット・ファイル・実装詳細を漏れなく追記 | **JSON のみ**（`/updates` に出さない） |
| **ユーザー向け pickup** | `public` | **仕事が変わる**単位 · registry tool id のみ | `/updates` |

- **原則:** デプロイごとに `internal` を必ず追記（MECE）。**ユーザー価値テスト**に通る変更だけ、別エントリで `public` を追加（または同日バンドル）。
- **禁則:** 「全部 public にする」「全部 internal にして `/updates` を空にする」— MECE と pickup は両立させる。
- **`rollup`:** internal 行がどの public 要約に束ねられたか（public 側の `id` を参照）。束ね先が無いインフラのみの変更は `rollup` なしの `internal` でよい。
- **`id`:** public 要約用の安定キー（例: `planning-poker-20260709`）。
- **検証:** `npm run validate:changelog` — `audience: public` のみ粒度ルールを強制。`/updates` は `audience === "public"` のみ表示。

### ユーザー価値テスト（public にする前に必ず）

次の **どれか1つ以上** に Yes なら public。**すべて No** なら `internal` のみ。

1. **新しい道具** — 新ツール · 新モード · 新プリセット（例: 見積会議 α · 表変換 · SQL IN 句）
2. **計算・結果が変わる** — 税率 · 端数 · 抽選 · 班分けなど、同じ入力で出力が変わる
3. **手元の作業手順が変わる** — コピー/印刷/共有/取込のやり方が変わる · 名前が変わって探し方が変わる
4. **壊れていたものが直った** — ユーザーが再現できる不具合の修正

**迷ったら internal。** 「きれいになった」「SEO」「レイアウト統一」だけでは public にしない。

### 載せる（public pickup）

| 優先 | 例 |
|------|-----|
| **必須** | 新ツール · α/β 公開 · 計算ロジック変更 · 重大バグ修正 |
| **可** | ユーザーが名前で探す表示名の変更 · 件数 cap · 料金 |
| **原則不可** | FAQ 背景幅 · 設定パネルのスクロール · canonical · robots · ビルドパイプライン |

### 載せない（Too Much）— public 禁止

| NG 例 | 理由 | 置き場 |
|--------|------|--------|
| FAQ レイアウト統一 · 設定パネル縦表示 | 見た目の微調整。仕事は変わらない | `internal` |
| canonical · og:url · `robots` · GSC · 内部 JSON 除外 | 検索エンジン向け。画面操作に無関係 | `internal` |
| `sg-main-shell` · CSS クラス名 | 実装詳細 | `internal` |
| `sugudasu.css` · `*.md` · Agent SSOT | ファイル名の羅列 | `internal` |
| デプロイ台帳 · CI · Pages 予算 | 運用内部 | `internal` |
| 「単体テスト追加」だけの追記 | 開発者向け | `internal`（または本文から削除） |
| 同一日の細切れコミット | ノイズ | **機能単位で最大3件/日** · それ以上は週次まとめ |

### タイトル · 本文の書き方

- **タイトル:** **概念名**（navLabel）+ 何が変わったか。slug（`planning-poker` · `time-calc`）は括弧補足まで · 主役にしない
  - よい: 「見積会議 α — 同時 Reveal で認識差を可視化」
  - 悪い: 「FAQレイアウト統一・設定パネルの縦表示改善」
- **本文:** 1〜3文 · **誰のどんな作業が楽になるか**（技術語 · ファイル名禁止）
- **`type`:** `feature`（新機能 · 新設・α）· `fix` · `improve`。**`feat` は禁止**
- **`tools` 配列:** registry の **tool id** のみ。`*.html` · `*.css` · パス禁止
- 横断リリースは **ユーザー価値があるものだけ** 1エントリにまとめる（インフラ横断は `internal`）

### 悪い例 → よい例

| 悪い（public にしない） | よい |
|------|------|
| 検索向けURL整理（canonical · 内部JSON除外） | `internal` のみ。表示名変更があれば別 public「時給計算に改名」 |
| FAQレイアウト統一・設定パネルの縦表示改善 | `internal` のみ |
| `sg-main-shell` を全ツールに適用 | （読みやすさ改善が体感できるなら）「ツールページの幅・余白をそろえ読みやすくしました」 |
| `mask-app.js` オーバーレイ確定 | マスク — 黒塗りが確定しない不具合を修正 |
| 見積会議を improve で埋もれさせる | **feature** で独立エントリ（新ツールは必ず目立たせる） |

## 3. ロードマップ（予定 · 検討中）の粒度

### 載せる

- **予定（scheduled）:** 次の開発サイクルに入れた機能 · ツール横断 OK
- **検討中（considering）:** FB で優先度が上がりうる候補
- 各項目に安定 **`id`**（アンカー `roadmap.html#id`）

### 載せない

- 実装タスク（「CSS 変数追加」）
- 日付の確約（「7月中必須」）
- 個人名 · 却下された個別要望

### リリースしたら

1. `roadmap.json` から該当 `id` を **削除**（または `shipped` にし JSON から外す）
2. `changelog.json` に **ユーザー向け1エントリ** を追加
3. 必要なら roadmap 項目に一時 `shippedChangelogDate` は使わず、changelog のみ正本

## 4. 対象外レーンの粒度

### 載せる

- **プロダクト非目標**（病院24/365看護 · 多店舗勤怠 · 労基自動判定）
- **業態名 · 用途名** を明示（SEO · 誠実線）
- **代替**（ジョブカン · 専用 SaaS 等）を1行

### 載せない

- 特定ユーザーへの言及
- 「永遠にやらない」と断言しすぎる法務的拘束（「意図的にスコープ外」で十分）

## 5. Agent · 提督の作業チェックリスト

**changelog 追記前**

- [ ] MECE 用に `audience: "internal"` 行を追記したか？（ファイル名・実装詳細 OK）
- [ ] **ユーザー価値テスト**（新道具 / 計算変化 / 手順変化 / バグ修正）に Yes があるか？なければ public にしない
- [ ] 新ツール・α は `type: "feature"` で独立エントリか？（インフラと同梱しない）
- [ ] public 行から CSS クラス · ファイルパス · 内部 doc 名・slug 先頭タイトルを除いたか？
- [ ] public の `tools` は registry id のみか？
- [ ] internal に `rollup` がある場合、public 側 `id` は存在するか？

**roadmap 追記前**

- [ ] `id` は kebab-case で安定か？
- [ ] 予定 / 検討中 / 対象外 のどれか明確か？
- [ ] 対象外は**領域**単位か（個別 FB ではないか）？

**デプロイ後**

- [ ] shipped した roadmap 項目を JSON から外したか？
- [ ] changelog 1件追加したか（ユーザー向け文言）？
