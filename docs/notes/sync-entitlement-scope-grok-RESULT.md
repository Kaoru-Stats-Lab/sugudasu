## §1 総評（200字 · 賛否を明示）

GeminiのS-Dハイブリッドは紙面上バランスを取っているが、イベント当日幹事視点では複雑化の弊害が深刻。trial全体1＋active製品別は理論的だが、製品切り替えや枠確認の認知負荷が本番進行表操作中にパニックを誘発する。DB/トリガー改修中程度でも、個人開発のβ→S3移行でバグ連鎖リスクが高く、SSOT過剰投資に見合うリターンが不明。S-A簡素維持を推奨し、S-Dは却下寄り。**（178字）**

## §6 載せすぎリスク Top 3

- 製品別残り枠表示 → 幹事が枠管理に飛び進行中断
- Trial 製品切替フロー → データ消失懸念で操作停止
- トリガー/Webhook 不整合 → 突然作成不可 · システム不信

## §7 実装時の落とし穴

- `sync_count_active_rooms` の product_type 移行でカウント不整合
- Webhook 着地失敗 → entitlement ずれ → ルームロック連鎖
- ルーム作成 JOIN 増 → 当日保存遅延
- Trial→Active 移行の payload 引き継ぎ漏れ
- 製品追加時の関数/インデックス改修肥大

## §8 最終推奨

**S-A 維持 ＋ `product_type` 参照列のみ追加（数え方変更なし）。** Gemini S-D は却下。

| 項目 | Grok 推奨 |
|------|-----------|
| trial | アカウント横断 **1**（変更なし） |
| active | アカウント **プール 3**（製品横断） |
| `product_type` | ルーティング · Stripe 着地 · 共有 URL 用 **のみ** |
| トリガー | **現行維持**（`owner_id` + `retain_until`） |
| UI 枠表示 | **単一** `残り X/Y` |

### 推奨フェーズ分割（Agent · 提督承認用）

| フェーズ | スコープ | 根拠 |
|----------|----------|------|
| **β〜2製品目 UI** | **S-A'（Grok）** — 数え方不変 · `product_type` 列だけ | バグリスク最小 · 1製品稼働中は実質同じ |
| **S3 課金接続時** | 製品別 SKU が **価格差別化に必須**なら S-D を **再評価** | Gemini のビジネス論点は S3 で初めて効く |
| **UI** | 単一 `残り X/Y` + 超過時アーカイブ提案（Grok §5） | 当日パニック対策 |

### Grok が正しく突いた Gemini の弱点

1. 製品別枠 UI は `sync-account-page-content-grok-RESULT` の「分散パニック」と同型
2. トリガー改修は E2E 未完了の S1 に **連鎖リスク**
3. `ON CONFLICT` / webhook 着地は S3 問題 — β で SSOT 深掘りは過剰

### 却下しない Gemini 要素（S3 に温存）

- `user_entitlements` に `product_type` · `stripe_price_id` · `status`
- Stripe metadata · Webhook 行の形（§3-6）
- active 製品別は **月額＋アドオン販売が確定した時点**のオプション
