### 1-2. 提督実体験 · 2モード（2026-06-20）

同じ「班分け」でも **現場要件が真逆** になる。機能優先度の判断軸。

| モード | シーン | 欠席 | 制約 | ツールの役割 | スイッチャー |
|--------|--------|------|------|--------------|--------------|
| **バッファ型** | 納会 · 円卓シャッフル（事業部×職種） | 料理・グラスで吸収 · **再編ほぼ不要** | 分散（混ざればOK） | 事前に1回 · TSV/告知 | **不要** |
| **ゼロバッファ型** | ハッカソン（オンライン） | エンジニア0人 → **即再編** | **各組必須**（職能 hard） | 欠席のたび再実行 · 差分共有 | **必須**（BR · Slack） |
| **緩和型** | **アイディアソン** | 欠席はある程度吸収可 | **職能 hard にしない** | 分散 · 均等が主 | オンラインなら任意 |

**ハッカソン体験の本質:** 班を決めるだけでは終わらない。**BR への案内 · Slack の部屋割当 · 欠席時の Zoom 手動移動**がセット。会議画面に属性が出ないため、**班番号 ↔ BR名 ↔ Slack** の対応表を持った **スイッチャー** がボトルネックになる。

**アイディアソン（提督）:** 班分け自体は大きな問題にならなかった。**エンジニアリング概念のない参加者**への役割説明・期待値調整は班分けツールの外（名簿の職能ラベル定義 · 事前ブリーフが先）。

## 2. 設計原則

### 2-1. Outcome-first · **2次利用前提（提督FIX · 2026-06-19）**

本ツールの成果物は **画面で見せて終わり** ではない。幹事・人事が **Slack / Excel / LINE / 社内Wiki / 印刷** に **そのまま流す** のが主用途。

| 原則 | 内容 |
|------|------|
| **Copy-first** | 各形式 **1クリックコピー**（`sg-copy-feedback` · fair-draw TSV と同型） |
| **Excel 最優先** | **TSV/CSV（UTF-8 BOM）** · 列 `グループ番号, 氏名` が正本 |
| **チャット向け** | Slack/Markdown · プレーンテキスト（LINE）を **別ボタン** |
| **再実行・監査** | JSON ダウンロード（シード · 制約 · 班構成） |
| **避ける** | PDF だけ · 画像スクショだけ · コピー不能な Canvas 演出 |

ユーザーが決めるのは **「何人1組か / 何組か」** と **名簿**。アルゴリズム名（Fisher-Yates）は UI に出さない。

| 避ける | 採用 |
|--------|------|
| ルーレット · Magic Hat 演出 | **表形式の結果** · TSVコピー |
| 会場座席図ドラッグ | **グループ1 · グループ2 …** のリスト |
| サーバー保存 · 共同編集 | **1端末完結** · 結果 JSON ダウンロード（任意） |
| 「100%公平保証」 | **シード + 名簿指紋** · 再実行で同条件なら同結果 |

### 2-2. 軽量・上限

| 項目 | 値 |
|------|-----|
| 名簿最大 | **250名 FIX**（Phase C · 複数属性ソルバ · 研修規模） |
| ソフト警告 | **200名超** → Excel TSV 推奨（典型ブレイクアウト規模） |
| グループ数 | **2〜50**（「M グループ」モード）· 「N 人1組」は自動算出（最大500組相当） |
| 1グループ最小人数 | **2**（端数はラウンドロビン · 差最大1） |
| 処理 | ブラウザ内 · **POST なし** · Phase B/C 制約も O(n) ヒューリスティック想定（250名規模） |

### 2-3. 公平性の定義（Phase A · v1）

1. 名簿を **シード付きシャッフル**（`prize-law-eval.js` の `createSeededRandomFn` を **共通化抽出 or import**）
2. シャッフル順に **ラウンドロビン** でグループへ割当（人数差最大1）
3. メタ出力: `seedHex` · `rosterSha256`（先頭16桁表示可）· `splitAtJst` · `groupCount` · `mode`

**seed · 名簿指紋と復元（v1.2.3）**

| 項目 | 役割 | 復元に使えるか |
|------|------|----------------|
| `seedHex` | 同じ名簿 · 同じ条件で再実行したとき同じ班にする乱数の種 | **単体では不可** — セッションJSONに含まれる |
| `rosterSha256` | 実行時名簿テキストの SHA-256 | **単体では不可** — 照合（同じ名簿か確認）用 |
| **セッションJSON** | 名簿 · 条件 · seed を一式保存 | **復元の正攻法** — UI「読み込んで再実行」 |

手順: 実行直後にセッションJSONをコピー/保存 → 別端末で貼り付け再実行。名簿指紋が一致すれば同じ名簿で再現できている。

### 2-4. 定員・均等・制約 — **提督FIX · 2026-06-19**

**Pain:** 均等に分けたいが、**属性や制約のせいで「目標定員より多い班」が必要**になることがある（ハッカソン · 研修で実体験）。

#### 用語（UI と SSOT で統一）

| 用語 | 意味 | 例 |
|------|------|-----|
| **目標定員** | 幹事が望む 1 グループの人数（Phase A の「4人1組」） | 4 |
| **定員超過** | 目標より **多い** メンバーが 1 グループに載る状態 | 固定3名班 + α → 5人組 |
| **下限** | グループごとに **最低** 必要な人数・役割 | 各チームにメンター1名以上 |
| **固定班（bundle）** | **分割不可** のまとまり（1ユニットとして割当） | 事前に決まった3人チーム |
| **上限（hard max）** | Phase B 任意 · 絶対に超えてはいけない人数 | 卓の物理席 6 |

#### 優先順位（アルゴリズム · Phase B 以降）

1. **硬い制約を満たす**（固定班 · 固定メンバー · 離すペア · 各組下限）
2. そのうえで **目標定員からのズレを最小化**（`|人数 − 目標|` の総和を小さく）
3. **定員超過が必要なら許容** — ただし **黙って均等に見せない**
4. 制約どおりに割当 **不可能** なときだけ **赤旗**（例: 離すペアが多すぎて M グループに収まらない）

> **思想:** 「きれいに均等」より **「ルールを破らない」**。超過は **バグではなくトレードオフ** として画面に出す。

#### 定員超過が起きる典型例

| シーン | 原因 | 結果例（目標4人） |
|--------|------|-------------------|
| ハッカソン | **3人でエントリー済み** の班を分割しない | その組 alone で **3人** → 残りを足すと **5〜6人** |
| 研修 | **ファシリ1名が各組必須** + ランダム4名 | 実質5人組 |
| 研修 | **リーダー候補が少ない** · 1組に2リーダー必要 | 偏り · 一部組だけ +1 |
| 人事 | **同一部署を分散**（タグ） | 完全均等は不可能なことも · 差を最小化 |

#### UI 要件（Phase B · 超過あり得るとき）

- 実行前: **「制約を満たすため、目標定員を超えるグループがあります」**（制約入力時のみ）
- 実行後:
  - 各グループ見出しに **`4名（目標+1）`** のようなバッジ
  - **超過理由 1 行**（例: 「固定班3名を分割しないため」）
  - **人数サマリ:** `最小3 · 最大5 · 目標4 · 差2`
- JSON に `targetSize` · `actualSizes[]` · `overflowReasons[]` · `constraintsApplied`

#### Phase との対応

| Phase | 均等 | 制約 · 定員超過 |
|-------|------|----------------|
| **A** | ◎ のみ（ラウンドロビン · 差最大1） | なし |
| **B1** | 制約後にズレ最小化 | 固定班 · 固定メンバー · 離すペア |
| **B2** | 同上 | 2列目タグ · 各組「属性1名以上」等 |
| **C** | 同上 | Excel複数列 · **属性2〜3** · 列ごとに分散/各組必須 |

## 3. Phase A MVP（v1 · 最初に ship する範囲）

### 3-1. 入力

- テキストエリア **1行1名**（`normalize` と同型）
- モード **1択**（sg-segment）:
  - **「N 人1組」** — 例: 4人1組 → グループ数は自動（ceil(人数/N)）
  - **「M グループ」** — 例: 5グループ → 各グループ人数は自動均等

### 3-2. 出力（2次利用 · Phase A 必須）

**UI:** 結果直下に **出力形式タブ or セグメント** — デフォルトは **Excel用TSV**。

| ID | 形式 | 用途 | 仕様メモ |
|----|------|------|----------|
| **O1** | **TSV（ロング）** | Excel · Googleスプレッドシート · ピボット | ヘッダ: `グループ,氏名` · LF · コピー primary |
| **O2** | **CSV（UTF-8 BOM）** | Windows Excel 文字化け回避 | O1 と同列 · ファイルDLボタンも可 |
| **O3** | **Slack用** | Slack / Teams 投稿 | `*グループ1*` + 箇条書き · mrkdwn 控えめ |
| **O4** | **LINE/プレーン** | LINE · メール本文 | `【グループ1】田中、佐藤、…` 1行または改行 |
| **O5** | **告知ブロック** | 研修冒頭・ハッカソンDiscord | セッション名 + 全グループ一覧 · そのまま貼付 |
| **O6** | **JSON** | 再現 · 監査 · 次回編集 | `seedHex` · `groups[]` · `sessionLabel` · Phase B で制約メタ |
| **O7** | **TSV（ワイド）** | 横並び一覧（人数差≤1 の Phase A のみ） | 列=グループ1…M · 行=席次 — **定員超過 Phase B では非表示 or 警告** |
| **O8** | **スイッチャー対応表**（**P1 · M13 未実装**） | Zoom BR · Slack 案内の当日オペ | 列: `グループ, BR名, Slack, 氏名` — 幹事が BR/Slack 列を事前入力 · 班員と結合して TSV/Slack コピー · **API 連携なし** |

- コピー成功: fair-draw / normalize と同じ **緑フラッシュ · 2sロック**
