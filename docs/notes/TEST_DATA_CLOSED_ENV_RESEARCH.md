# 閉域環境 × テストデータ — 業種横断リサーチ

**更新:** 2026-07-02  
**目的:** 金融以外にも「閉域でテストデータが欲しい」現場を洗い出し、**業種固有のネック（詰まり）**を記録する。  
**関連:** [`TEST_DATA_TOOL_SPEC.md`](TEST_DATA_TOOL_SPEC.md) · [`test-data` ツール](../../tools/test-data.html) · 競合参考 [SnowFakery](https://media.zenet-web.co.jp/entry/2021/07/19/100002) · [Handy Dev Tools](https://handy-dev.tools/test-data-generator) · [Qiita: chiyoyo](https://qiita.com/chiyoyo/items/d0ee467a0229a465efcd)

## 0. 提督用 · 3行結論

1. **閉域で困るのは金融だけではない。** 官公庁・医療・インフラ・製造・人材・小売でも、**本番コピー禁止・外部DL禁止・USB禁止**は日常茶飯事。
2. 業種ごとに欲しいデータの**形（列・コード体系・整合ルール）**は違うが、**ネックの型は横断で似ている**（持ち込み・再現性・整合性・説明責任）。
3. SUGUDASU `test-data` の伸ばし方は **「50型ビルダー」ではなく、業種プリセット＋非送信説明＋地雷＋シード** の垂直追加が自然。

## 1. 分類軸（MECE）

### A. なぜ閉域か（制約の種類）

| コード | 制約 | 典型の一言 |
|--------|------|------------|
| **C1** | インターネット遮断（エアギャップ / 閉域網） | 「開発PCは外に出せない」 |
| **C2** | 外部ファイル持ち込み禁止 | 「怪しいCSVを落とすな」 |
| **C3** | 本番・個人情報の持ち出し禁止 | 「マスキング export も審査」 |
| **C4** | クラウドSaaS利用禁止 | 「Mockaroo も Google も NG」 |
| **C5** | 監査・再現性要求 | 「同じデータで再テストせよ」 |
| **C6** | ベンダー境界（顧客側だけ閉域） | 「SIは顧客VPNの中だけ」 |

※ 多くの現場は **C1+C2+C3** の複合。

### B. 何が欲しいか（データの層）

| 層 | 例 |
|----|-----|
| **D1 マスタ** | 顧客 · 患者 · 社員 · 設備 · 商品 |
| **D2 トランザクション** | 取引 · 受診 · 検針 · 出荷 · 勤怠 |
| **D3 帳票・提出** | 請求 · レセプト風 · 工程表 · 届出 |
| **D4 ログ・監査** | 操作ログ · 承認履歴 · アクセス記録 |

### C. 誰が困るか（組織の下位レイヤー）

エンタープライズTDM（Delphix / SnowFakery / 社内基盤）に届かない層：

- 受託SIの **開発・テスト・マニュアル担当**
- 顧客社内の **情シス1〜2名＋業務担当**
- **プリセール・導入支援**（デモ用データを今日欲しい）
- **研修・オンボーディング**（本番見せられない）

## 2. 業種別 — 固有データ · 固有ネック

### 2-1. 金融（銀行 · 保険 · 証券 · 決済）

**欲しいデータ（D1/D2）**

- 顧客ID · 口座番号風 · カナ氏名 · 取引明細 · ステータス · 与信区分
- 法人顧客 · 振込先 · 摘要 · 通貨 · マイナス残高

**固有ネック**

| # | ネック | 詳細 |
|---|--------|------|
| F1 | **口座・顧客番号の書式規則** | 桁数 · 支店 · チェックディジット（社内ルール） |
| F2 | **整合トリオ** | 口座種別 × 取引種別 × 残高符号 |
| F3 | **本番マスキング politics** | 仮名化 export も情シス審査・監査ログ |
| F4 | **マニュアルと結合試験** | 画面スクショ用に「それっぽい残高」が要る |
| F5 | **受託SIの二重閉域** | 顧客閉域の中で、さらにベンダーPCは持ち込み不可 |

**オンライン生成ツールが刺さらない理由:** C2+C4。Handy 等は機能十分だが **「外部サービスに触れない」説明**が弱い。

**SUGUDASU適合:** **高** — 現行 `customer` / `transaction` プリセットの正本。

### 2-2. 官公庁 · 自治体 · 公共

**欲しいデータ**

- 住民票風マスタ（※実在自治体コード不要レベルのダミー）
- 申請番号 · 届出ステータス · 担当課コード · 文書番号
- 操作ログ（職員ID · 端末 · 処理種別）

**固有ネック**

| # | ネック | 詳細 |
|---|--------|------|
| G1 | **持ち込み媒体制限** | USB禁止 · メール添付上限 · 個人端末禁止 |
| G2 | **ガイドライン準拠の説明** | 政府情報システム · ゼロトラスト · クラウド利用方針 |
| G3 | **組織コード体系** | 省庁コード · 課係 · 年度 · 予算科目 |
| G4 | **デモと本番の混同恐怖** | 「本物っぽすぎる」データも逆に怖い |
| G5 | **ベンダー入札後の短期導入** | スキーマは決まったがテストデータ未準備 |

**SUGUDASU適合:** **中** — `申請番号` `課コード` プリセット追加で刺さる。マイナンバー風は **生成しない／別列で明示ダミー** が安全。

### 2-3. 医療 · 介護 · 薬局 · 医療機器ベンダー

**欲しいデータ**

- 患者ID · カナ氏名 · 生年月日 · 診療科 · 保険組合せ
- レセプト風明細 · 処方 · 検査コード · 入院日数
- DPC/標準病名コード風（形式だけ · 実コード不要）

**固有ネック**

| # | ネック | 詳細 |
|---|--------|------|
| H1 | **実患者データは原則不可** | 同意取得困難 · 仮名加工情報は契約・加工手続が重い ([厚労省ブック](https://www.mhlw.go.jp/content/001507527.pdf)) |
| H2 | **コード体系の壁** | 病名 · 医薬品 · 診療行為 — 病院ごとに異なる |
| H3 | **ベンダー受託開発** | 病院閉域内で動作確認 · マニュアル用スクショ |
| H4 | **合成データ期待と現実のギャップ** | エンタープライズは合成データ研究 ([NEC等](https://www.innervision.co.jp/products/release/20260506)) · 下位レイヤーはExcel |
| H5 | **外字・長い患者名** | レイアウト崩れ · カルテ連携の境界値 |

**SUGUDASU適合:** **中〜高** — `患者マスタ` `受診明細` プリセット · 地雷（外字・超長名）· `mask` 併用。レセプト完全準拠は **v3以降**。

### 2-4. インフラ（電力 · ガス · 通信 · 交通）

**欲しいデータ**

- 需要家ID · 契約番号 · 検針値 · 料金プラン · 区間
- 設備ID · 点検履歴 · 異常アラーム · 地理コード
- 24h時系列（グラフ・閾値テスト）

**固有ネック**

| # | ネック | 詳細 |
|---|--------|------|
| I1 | **OT/IT分離** | 制御系は完全隔離 · 業務系だけテスト用PC |
| I2 | **時系列の整合** | 検針日時 · 欠測 · スパイク — ランダムCSVでは破綻 |
| I3 | **設備点数スケール** | 数万点マスタ — ブラウザ5,000件では足りない場面あり |
| I4 | **ベンダー長期保守** | 10年システム · 担当者交代 · 再現データなし |
| I5 | **障害訓練・BCP** | 特定パターンの異常値を **意図的に** 混ぜたい |

**SUGUDASU適合:** **中** — `検針履歴` `設備マスタ` · 時系列は **シード＋連番日付** で部分対応。大規模負荷は対象外。

## 6. やらないこと（明示）

## 9. 用語

| 語 | 意味 |
|----|------|
| **閉域** | インターネット未接続 or 社内LANのみ |
| **下位レイヤー** | エンタープライズTDMを持たない開発・テスト・マニュアル現場 |
| **ネック** | その業種で **特に詰まる** 制約・作業 · 他業種には少ないもの |
| **形式風** | 見た目・桁数は似せるが、規格準拠・実在性は担保しない |
