## 記事の核心

**「ランダムに分ける」だけならルーレットサイトでも足りる。** 差は **名簿を外部に送らない** · **制約付き編成** · **Slack/Excel にそのまま貼れるテキスト**。

| | Excel RAND | Online Stopwatch 系 | SUGUDASU group-split |
|---|------------|---------------------|----------------------|
| 名簿の行き先 | ローカル/Drive | **運営サーバーへ送信** | **ブラウザ内のみ** |
| 再現性 | F9 で変わる | 証跡なし | **シード値で再現** |
| 2次利用 | 手動整形 | 画面演出中心 | **TSV · Slack · 告知文** |
| 制約 | マクロ必須 | 均等分割のみ | 固定班 · 離すペア · **属性各組必須**（Phase C） |
| 演出 | なし | ルーレット豊富 | **事務作業向け（演出なし）** |

### 記事に入れる一文（コピペ可）

> 研修名簿を Online Stopwatch のグループ生成に貼ると、**参加者名が海外のサーバーに送られます**。F12 で POST が出ない静的ツールなら、人事・研修担当は「載せたくない名簿」を載せずに班分けできます。

## 提督実体験ネタ（2026-06-20 · 記事の差別化柱）

**書き方:** 冒頭 or H2-1 で「3つのイベントで班分けの重さが違った」と体験談1段落 → 表で整理。**断定しすぎず「幹事の判断が変わる」** トーン。

### 3モード比較（コピペ可）

| | 納会・円卓 | アイディアソン | ハッカソン（オンライン） |
|---|------------|----------------|--------------------------|
| 班分けの難度 | 低（料理バッファ） | **低** | **高** |
| 制約 | 所属・職種を分散 | **職能 hard にしない** | **各組にエンジニア必須** |
| 欠席 | 再編ほぼ不要 | ある程度吸収 | エンジニア0人 → 即再編 |
| ツールの仕事 | 事前1回 · TSV | 分散・均等で十分 | 各組必須 · 欠席タップ再構成 |
| 当日の主役 | 幹事 | ファシリ | **スイッチャー**（BR · Slack） |

### エピソード（そのまま使える）

1. **納会** — 事業部・職種で円卓シャッフル。欠席しても料理・グラスにバッファがあり、班の組み直しはほぼ問題にならなかった。
2. **ハッカソン** — エンジニアが絶対必要。欠席に応じて組み直し。オンラインでは **BR への案内 · Slack の部屋割当** が必須で、**スイッチャー**（Zoom で人を動かす人）が最重要。Zoom 画面に属性は出ないので、**班番号 ↔ BR名** の対応表を持っていないと詰む。
3. **アイディアソン** — **班分け自体は問題にならなかった**。一方で **エンジニアリングの概念がない参加者** がいて、班に入ったあとの **役割説明・期待値調整** がつらかった — これは班分けツールの前（名簿の職能ラベル · 事前ブリーフ）の話。

### 記事に入れる一文（ツール境界）

> 班分けツールが解くのは **「誰をどの組に」まで**。ハッカソンで「非エンジニアの班での役割」をどう伝えるかは、**名簿の職能列をどう定義するか** と **開始前の1分ブリーフ** の領域です。

### ハッカソン幹事向け小ネタ（H2 or コラム）

**名簿の職能列、こう書くと当日つらくない**

| 避ける | 代わりに |
|--------|----------|
| 「ITに詳しい」「なんとなく触れる」 | **エンジニア / UI / 企画 / その他** を幹事が事前に確定 |
| エントリー時の自由記述だけ | Excel 1列に **幹事が正規化**（`normalize` 経由可） |
| 各組必須をかけない曖昧名簿 | 「エンジニア」人数 < 組数なら **緩和モード** か組数を減らす判断を先に |

**アイディアソン向け1行:** 各組必須は使わない。所属・部署の **分散** だけでよい（緩和型）。

### スイッチャー向け（記事後半 · 実務コラム · 箇条書き5つ）

Zoom API は使わない前提で、幹事が事前に揃えるもの:

1. **班 ↔ BR 対応表**（`G1 = BR-A`）
2. **班 ↔ Slack**（`#team-01`）
3. **班分け正本**（TSV + シード）
4. 欠席時は幹事が再実行 → スイッチャーへ **差分1行** DM
5. **2画面** — Zoom + 班表（スマホ可）

詳細: `GROUP_SPLIT_SWITCHER_PREP.md` · 製品側は **M13 対応表出力**（未実装 · 記事では「幹事がスプシで作る手順」で可）

### 各組必須が人数不足のとき（v1.2.3）

- 例: 営業サポート22名 · 23組 · 各組1名以上 → **数学的に1組は空く**
- 厳格停止 vs **「可能な限り実行」**（不足組を結果に表示）— **幹事の判断**
- 記事では「公平＝全制約を100%満たすことだけではない」と書ける（煽りすぎない）

## H2 案（Gemini §1–§4 + 実体験）

1. **班分けあるある3選** — 新入社員ブレイクアウト · ハッカソン職能 · Slack 共有（§1）
2. **【NEW】同じ班分けでも現場が違う — 納会・アイディアソン・ハッカソン**（上表 · 提督体験）
3. **Excel RAND() が正本を失う理由** — 再計算 · 手動整形 · シードの話
4. **ルーレットサイトと役割が違う** — 演出 vs 事務 · §2 比較表
5. **【NEW】ハッカソン名簿の職能列** — ツールの外にある Pain（非エンジニアへの役割）
6. **2次利用 — TSV を Excel に · Slack 用見出し** — §3
7. **【NEW】オンラインならスイッチャーが持つもの** — BR対応表 · 差分DM（Zoom APIなし）
8. **group-split の設計** — 非送信 · 250 cap · 属性条件 · 欠席タップ再構成 · 緩和モード
9. **やらないこと** — Zoom ブレイクアウト API · 日程 · AI 相性最適 · ルーレット演出（§4）
10. CTA → `/group-split`
