### 本文

「このツール、データ送ってない？」と聞かれるたびに、私はまず **何を送らないのか** を切り分けます。SUGUDASU（[ツール一覧](https://sugudasu.com/?utm_source=zenn&utm_medium=social&utm_campaign=article_06_privacy_design)）は、請求書・班分け・正規化・抽選など **幹事向けの実務ツール集** です。設計の芯は **登録不要 · 静的配信 · ブラウザ内処理 · 入力データを POST しない** です。配信は Cloudflare Pages 上の静的 HTML + JavaScript、処理の中心はクライアント側です。

この記事では、マーケ文句の「プライバシーファースト」ではなく、**「ブラウザ内」が実は何層にも分かれること** と、**オンラインでしかできないこと / オフラインでもできること** を切り分けて説明します。エンジニア向けの実装メモと、幹事が自分で確かめる手順を同じ記事に載せます。詳細の読み物は [SUGUDASU の約束](https://sugudasu.com/statements?utm_source=zenn&utm_medium=social&utm_campaign=article_06_privacy_design) にまとめています。

## 「ブラウザ内」は一言で片付けられない

「ブラウザで処理」と言うと、1つの箱に見えます。実際は **別々の機能の束** です。

| 層 | ざっくり何をするか | 例 |
|----|-------------------|-----|
| **取得（fetch）** | ネットから HTML / JS / 画像を取る | 初回アクセス · CDN · 広告タグ |
| **キャッシュ** | 取ったファイルを端末に残す | HTTP キャッシュ · 戻るボタン（bfcache） |
| **実行（JS）** | 貼ったテキストを変換 · 抽選 · 計算 | normalize · group-split の本体 |
| **出力** | 結果を外に渡す | Clipboard API · `Blob` ダウンロード · 印刷 PDF |
| **保存（任意）** | 端末に残ることがある | localStorage · 下書き JSON ファイル |

日常のたとえで言うと、**ニュースサイトをオンラインで読む** のと、**記事を PDF として保存してオフラインで読む** のは別の話です。どちらも「ブラウザ」ですが、前者は **取得レイヤ**、後者は **保存したファイルを別アプリ的に開く** に近い。Web アプリの非送信設計も、同じように **どの層の話か** を分けないと誤解が生まれます。

### オンラインでできること / オフラインでもできること

SUGUDASU コアを例にすると、次のように分けられます。

| | だいたい **オンラインが必要** | **一度読込後はオフラインでも動きうる** |
|---|------------------------------|----------------------------------------|
| 典型 | ページ初回表示 · 広告 · 更新版 JS の取得 | 名簿の貼付 · 班分け · 正規化 · コピー |
| 理由 | サーバー（CDN）からファイルを取る | キャッシュされた JS が端末で動く |
| 注意 | 機内モードテストは **キャッシュ状態依存** | 100% オフライン保証ではない |

ここでエンジニアでも混同しやすい点を3つだけ。

1. **「POST しない」≠「通信しない」** — ページ取得・広告の GET は通信します。見るべきは **貼付内容がボディに入る POST かどうか** です。  
2. **PDF はサーバーから配っていない** — 請求書の PDF は `window.print()` 相当で **ブラウザが組み立てたものを保存** しています。サーバーに「PDF を生成して返す」API はありません。  
3. **JSON 下書きはダウンロード** — `Blob` + ダウンロードリンクで **手元ファイル** になります。クラウド DB への自動アップロードではありません。

「ブラウザ完結」を謳うサービスほど、この **層の切り分け** を先に書いた方が、読者（と将来の自分）の理解コストが下がります。

### ブラウザごとに得意不得意がある（内蔵 AI は除く）

さらに言うと、「ブラウザ」とも **製品ごとに実装が違います**。最近話題の **ブラウザ内蔵 AI** は別機能なので本記事では触れません。ここでは **Web 標準の実装差** と、非送信ツールで効いてくる違いだけ拾います。

大きくは3系統です（Chromium 系 · Firefox（Gecko）· Safari（WebKit））。

| 観点 | Chromium 系（Chrome · Edge 等） | Firefox | Safari（macOS / iOS） |
|------|----------------------------------|---------|------------------------|
| **DevTools** | Network 手順の **事実上の共通語** | 同等だが UI が異なる | macOS は Web Inspector · iOS は制限多め |
| **Clipboard** | デスクトップは比較的素直 | おおむね同等 | **iOS はユーザー操作・権限の縛りが強い** |
| **印刷 → PDF** | OS 印刷ダイアログ経由 | 同左 · レイアウト差が出やすい | macOS/iOS の **PDF 保存** は慣れが必要 |
| **キャッシュ / オフライン** | キャッシュが効きやすい一方、挙動が端末依存 | 追跡防止の既定が強め | **ITP** 等でストレージ周りが厳しめ |
| **拡張機能** | 豊富（**通信を増やす第三者もいる**） | Add-ons | iOS はほぼ不可 |

エンジニアでも見落としがちなのは次の2点です。

1. **「同じ Web アプリ」でも、出力レイヤの体感が違う** — 請求書の PDF はサーバー生成ではなく印刷ダイアログ依存なので、**Chrome と Safari で余白・改ページが変わる** ことは普通にあります。SUGUDASU でも PC 確認を推奨しているのはここです。  
2. **非送信の検証は、自分が使うブラウザでやる** — 「POST が出ない」は Chromium の Network で見るのが手軽ですが、幹事が iPhone だけ使うなら **Safari でコピーと PDF** を試すのが実務的です。

設計側の教訓はシンプルで、**最新 API 1本に賭けない** ことです。Clipboard が失敗したらトーストで知らせる、印刷はブラウザ既定に任せる、名簿はサーバーに送らない — **共通の下限（textarea · Blob · print）** で組みます。ブラウザの「得意分野」を活かしつつ、**不得意を UI と FAQ で吸収する** のが、個人開発の現実的な落とし所です。

## 非送信と言うとき、何を送らないのか

ここで言う「非送信」は、**あなたが貼り付けた名簿 · 金額 · 抽選リストを、処理目的で当社サーバーへ POST しない** という意味です。

送らない対象と、送る（または発生しうる）ものは別です。

| 区分 | 例 | 名簿・入力データ |
|------|-----|------------------|
| **送らない（設計）** | 貼付 · 抽選 · 班分け · 文字列変換 | **端末内のみ** |
| **発生しうる（別レイヤ）** | HTML/JS の取得 · 広告 · 集計用アクセス解析 | 対象外 |
| **インフラ** | CDN · ホスティング事業者のログ | 対象外 |

「ブラウザ完結」と言っても、**ページを開くこと自体** はインターネット経由です。信頼は「通信ゼロ」ではなく、**レイヤを隠さない** ことから始まります。

### 補足: 静的配信とは？

**静的配信** は、ユーザーごとにサーバーで HTML を組み立てず、あらかじめビルドしたファイルを CDN から配る方式です。SUGUDASU コアは **ユーザー用 DB · 常時接続の同期サーバー · WebSocket ダッシュボード** をスコープ外にしています。月数枚の書類や当日の班分けのように、**短時間・低頻度・説明責任が個人に寄る** 作業に合わせた選択です。

## なぜ SaaS DB ではなく、この形か

クラウド SaaS は、アカウント · 共同編集 · 履歴 · 連携 API が強みです。SUGUDASU コアが狙うのは別のPainです。

- 月に数回しか使わないのに **登録と月額** が重い  
- 名簿1列を **外部 Converter に貼りたくない**  
- 結果は Slack / Excel に **コピーして終わり** でよい  

だから DB に預けず、**Copy-first**（TSV · Slack 用テキスト · JSON 下書き · 印刷 PDF）を正とします。API 自動投稿や OAuth 連携は、権限審査と名簿送信の境界が曖昧になるため、コアではしません。

[見積→請求の記事](https://sugudasu.com/invoice?utm_source=zenn&utm_medium=social&utm_campaign=article_06_privacy_design) で書いた JSON 下書きも同じ思想です。サーバーに保存しない代わりに、**手元ファイルで再開** します。

## 通信レイヤ — 5段で見る

`statements` ページでは、次の5レイヤで整理しています。

1. **① 入力処理** — 貼付 · 抽選 · 分割 · 変換（**端末内のみ**）  
2. **② ページ配信** — HTML / JS / CSS（CDN）  
3. **③ 広告** — 配信ネットワーク（閲覧に伴うリクエスト）  
4. **④ 計測** — アクセス解析（集計用 · 貼付テキストは含めない設計）  
5. **⑤ 外部例示** — 説明文内の「Excel に貼る」等の**例**（提携ではない）

「プライバシーファースト」と名乗るサービスでも、入力をサーバーに送るものはあります。**用途で選ぶ** のが正解で、幹事は自分のブラウザで確認できると安心です。

## 3分で確かめる — DevTools 手順

技術に詳しい方は、次の3ステップで確認できます。

1. **Network** — `F12` → Network を開き、名簿を貼って抽選・分割を実行。**当社ドメイン以外への POST**（ボディに貼付内容が含まれるもの）が出ないか見る。ページ読込 · 広告 · 解析用 GET は別レイヤ。  
2. **Application** — Local Storage / Session Storage に **名簿全文が残っていないか** 確認（環境依存 · 拡張機能に注意）。  
3. **機内モード** — 一度読込後、オフラインで core 機能が動くか試す。**キャッシュされていれば動く** — これは「オフライン対応アプリ」ではなく「取得済み JS が残っている」状態。正本は①。

## ツールごとの当てはめ（コア）

| ツール | 非送信の意味 |
|--------|-------------|
| 請求書 | 下書きは JSON を **ダウンロード** · サーバー DB なし |
| 班分け | 名簿はブラウザ内 · TSV/Slack をコピー |
| 正規化 | 500行 cap · ブラウザ内変換 |
| 抽選 | 証跡 PDF をローカル生成 |

いずれも **「画面で見せて終わり」ではなく、次の1クリック（コピー・保存）まで** が責務です。

## 正直な但し書き — Sync は別ライン

**SUGUDASU Sync**（`sync.sugudasu.com`）は、イベント当日の **短期バッファ** 用の別製品です。コアの「名簿を預けない」と同じ言葉で書くと誤解されます。Sync はアカウントとクラウド保存があり、**コアと Sync は別の約束** です（詳細は Sync 側の statements 予定）。

コア記事では、**コア = 非送信 · Sync = 同期が必要な場面** と線を引きます。「全部オフライン」ではありません。

## やらないこと

- 「100%安全」「完全に送信されない」の断定  
- 入力ゼロ通信の暗示（CDN · 広告は発生しうる）  
- 景表法 · 税務の「問題なし」宣言（判断は現場へ）  
- 競合 SaaS の名指し批判  

## まとめ

非送信設計は、魔法ではなく **ブラウザの層を意図的に選ぶ設計** です。取得は CDN に任せ、入力処理は JS 実行層に置き、成果物は Clipboard / ファイル / 印刷 PDF で渡す — SUGUDASU コアの背骨はそこにあります。

SaaS が悪いわけではありません。**失敗の仕方と運用コストの置き場所** が違うだけです。幹事が「貼る前に確認したい」ときに、DevTools で自分の目を信じられること。それがこの設計のゴールです。
