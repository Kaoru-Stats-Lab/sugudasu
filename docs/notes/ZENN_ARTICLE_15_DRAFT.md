### 本文

イベント本番で、受付と会場が同じ進行表に同時に行を足したあと、保存は成功しているのに **読めなくなった** 経験はありませんか。私は幹事経験と個人開発の両方から、同時編集で本当に怖いのは「保存できないこと」ではなく **「誰の作業塊か分からなくなること」** だと考え、**混ざらない同期・現場向けマージ** を意識した進行表ツール [SUGUDASU Sync](https://sync.sugudasu.com/timeline?utm_source=zenn&utm_medium=social&utm_campaign=article_15_sync_crdt_field) を設計しています。コア版（非同期・手元完結）は [sugudasu.com/timeline](https://sugudasu.com/timeline?utm_source=zenn&utm_medium=social&utm_campaign=article_15_sync_crdt_field) にあります。

この記事では、流行語としての `CRDT` ではなく、**なぜ行が混ざると現場が止まるか** と、**編集者少数・閲覧者多数** の運用に合わせた同期の考え方を書きます。幹事向けの痛みの言語化と、エンジニア向けの設計判断を同じ記事に載せます。なお、行単位の CRDT マージはロードマップ上 **S4** の予定であり、現行（S1）は `revision` による楽観ロックです。本文は **設計メモ** として読んでください。

## 保存は成功するのに、現場が止まる理由

進行表でよくあるのは、受付担当が「誘導A/B/C」を追記し、会場担当が同じ位置に「音響A/B/C」を追記する、というパターンです。クラウド表なら両方とも保存できます。しかし行が交互に並ぶと、**役割ごとの塊** が崩れ、司会・スイッチャーが読めなくなります。

技術記事では、同じ位置に `mom` と `dad` を挿入すると `"HI mdoamd!"` のように文字が交互に混ざる例（interleaving）がよく出ます。これは文字列の遊びではなく、**進行表の行混在** への比喩として使えます。文字が壊れると読めない。行が混ざっても同じく読めない。現場の痛みは同じです。

![](https://static.zenn.studio/user-upload/PLACEHOLDER-20260701.png)
*混在前（受付3行＋会場3行が交互）と混在後のイメージ — 投稿前に差し替え*

ここで必要なのは「全員が同時に書けること」そのものではなく、**意味の塊を守ること** です。

## CRDT は名前ではなく、下回りの設計判断

記事で伝えたい主語は、次の3つです。

1. **混ざっても読めること** — 誰がどの塊を足したかが追える  
2. **あとから説明できること** — 競合後も「何が起きたか」を説明できる  
3. **現場で止められること** — 自動反映に任せ切らない  

「最先端の用語を知っているか」ではなく、**事故を1件減らせるか** で判断したい。ここが、キーワードで釣るのではなく本質を取る線引きです。

### 補足: CRDT とは？

**CRDT（Conflict-free Replicated Data Type）** は、離れた場所での編集操作を、あとで **同じ結果に収束** させるためのデータ型の総称です。オフライン復帰後に壊れない、編集が混ざらない、と説明されることが多いですが、万能ではありません。SUGUDASU では **進行表の行順** にスコープを絞り、文字単位の共同編集そのものは目指しません。

## これは新宗教ではなく、昔から続く議論の延長

Zenn 読者の中心像を決め打ちはできませんが、このテーマは世代をまたいで通じます。分散か中央集約か、即時性か管理性か、自由度か監査性か。名前は変わっても、議論の芯はずっと同じです。

たとえば開発者向けには `Git vs SVN`、少し広い比喩なら `Napster/Gnutella vs iTunes` が近いです。どちらも「勝者と敗者」の話ではなく、**どこで失敗し、どこに運用コストを置くか** の設計判断です。Excel とスプレッドシートを宗教戦争にしないのと同じで、CRDT も **向き不向き** の話です。

最近は Git の代替や再設計をうたうサービスも出てきていますが、論点は変わりません。結局は、**何を中央で握り、何を分散で持つか** の再配分です。

ネットワークが前提の社会では、「つながること」自体より、**つながったあとに何を守るか** が設計の本丸になります。攻殻機動隊的な世界観と言ってよいかは人それぞれですが、少なくとも私にとっては、**接続そのものより、接続後の責任の置き場所** が設計の中心です。

## なぜ「全員フル編集」より「現場向けマージ」か

イベント運営では、編集者と閲覧者の人数が非対称です。編集するのは数人、見る人は多い。全員フル編集を前提にすると、更新は速くても **事故面積** が広がります。

SUGUDASU Sync で目指している運用は次のとおりです。

1. **編集者**（幹事・受付・会場など少数）が更新する  
2. **閲覧者**（司会・スタッフ多数）は「新しい版があります」バナーで気づく  
3. 反映は **手動で確定** する — 本番中に勝手に上書きされない  

![](https://static.zenn.studio/user-upload/PLACEHOLDER-20260701-2.png)
*新版バナーと手動反映の導線 — 投稿前に差し替え*

「自動で全部解決」より、**人が止める境界を残す** ほうが、本番運用では安全です。便利さを下げるのではなく、**壊れ方を制御する** 設計です。データは短期バッファとして扱い、export を前提にします（永久クラウド保管ではありません）。

| | 一般的なフル共同編集 | SUGUDASU Sync（方針） |
|---|---|---|
| 編集者 | 全員フル編集 | **編集者少数 · 閲覧者多** |
| 反映 | 自動上書き | **新版バナー + 手動反映** |
| 守る単位 | セル・文字 | **行の塊（順序）** |
| データ | 永久クラウド | **短期バッファ · export 前提** |

## 技術の核は 1 点 — 行の塊を守る（S4 予定）

現行の S1 では、ルーム状態を `payload` + 単調増加の `revision` で保存しています。編集者が同時に書いた場合は、後から書いた版が勝つ（LWW）形です。S4 では、編集者2人が **行の insert / reorder** を同時に行っても、同一編集者の連続挿入が塊のまま保たれるよう、NonInterleaving LSeq 系のマージを検討しています。

要点だけ書くと、分数インデックスに **peer（誰が）+ seq_no（何番目の操作か）** を持たせ、同じ人の連続操作が後から合流しても交互に割り込まれない、という発想です。実装の細部はこれから詰めます。

```text
# 混在してほしくない例（進行表）
受付: 誘導A, 誘導B, 誘導C
会場: 音響A, 音響B, 音響C

# interleaving すると読めない
誘導A → 音響A → 誘導B → 音響B → …

# 守りたいのは「塊」
[誘導A,B,C] + [音響A,B,C]  （順序はマージ規則で決める）
```

文字単位 CRDT や Yjs の丸ごと導入はスコープ外です。進行表・班分けなど **リスト型** に絞る判断です。工程管理（schedule）への横展開は、当日系（timeline/group）とは別判断にします。

## 刺さる人には、なぜ刺さるのか

この設計が刺さるのは、次のどれかを経験した人です。

- 「誰の更新が消えた？」に **答える責任** を持ったことがある  
- 本番中に表が読めなくなり、手で復旧したことがある  
- 便利機能より、**再現性と説明責任** を優先したことがある  

キーワードだけでは刺さりません。Before/After の具体例、守る単位の明示（文字ではなく行）、運用ガード（手動反映・新版通知）をセットで示す必要があります。

## やらないこと（誇張しない）

- 「CRDT だから競合ゼロ」とは言いません  
- 本番中の Checkout · 永久保存 · Zoom 連携はスコープ外です  
- 請求書・抽選など、リスト型以外への CRDT 拡張はしません  
- S4 未実装の機能を **実装済み** と書きません  

## まとめ

同時編集で本当に困るのは、保存の成否ではなく **読めなくなること** です。CRDT は流行語として売るものではなく、**混ざっても読める・説明できる・止められる** 同期を実現するための下回りの一つです。Git と SVN、Napster と iTunes、Excel とスプシと同じで、優劣ではなく **失敗の仕方と運用コストの置き場所** の話です。

SUGUDASU Sync は、編集者少数・閲覧者多数の現場に合わせ、新版バナーと手動反映で事故面積を抑えつつ、S4 で行の塊を守るマージを足していく — その設計メモがこの記事です。
