## §1 Executive Summary

SUGUDASU Syncのβ版におけるアカウントページの設計方針。提督の「見せすぎない」という初期仮説を、認知負荷の低減とイベント現場のコンテキストファーストの観点から全面的に支持・補強する。短期バッファ・ルーム単位管理という製品特性上、アカウント全体に依存する統計や容量メーターはノイズであり排除すべきである。本稿では、不要な重複情報を削ぎ落とした「4要素＋安全弁」に絞った最小構成の設定画面を確定し、インフラ逼迫や制限などのコンテキスト依存情報は、ユーザーが最も必要とする別の適切なチャネルへ分離・再配置する設計を提案する。

## §2 業界パターン — アカウントページに「載せる/載せない」の境界

| 情報カテゴリ | 典型 SaaS の挙動 | Sync への示唆・採用判断 |
| --- | --- | --- |
| **アクセス数・利用統計** | **載せない**（Notion, Linear等）。HubSpot等のアナリティクス製品のみがダッシュボードに配置。 | **非採用（提督仮説を支持）**。年数回しか使わないイベント幹事にとって、過去のアクセス数やログイン回数は当日のオペレーションを妨げる完全なノイズである。 |
| **内部 ID（UUID）** | **絶対に載せない**。サポート用にも隠蔽され、URLのハッシュ等にのみ出現。 | **非採用（提督仮説を支持）**。エンドユーザーに見せても混乱を招くだけである。問い合わせはすべて「ログインメールアドレス」を正本として処理する。 |
| **quota / grace の内訳** | **載せない**。利用規約の別紙、またはプラン購入モーダルの「詳細」リンクの裏に隠す。 | **非採用（提督仮説を支持）**。アカウントページに仕様説教を長文で載せるのはNotionの「静かなUI」に反する。事実（期日、作成ロック）のみを配置する。 |
| **セッション一覧（端末名等）** | **載せる傾向（中〜大規模）**。Securityタブ配下に配置。 | **β版では非採用**。個人開発のβフェーズにおいて、複数端末のセッション管理は開発コスト過多。S3以降（チーム編集）のロードマップへ退避。 |
| **利用規約・ビルド番号** | **最下部に小さく載せる**。控えめなグレー文字で配置。 | **採用**。プロダクトの信頼性と法務リスクの防波堤として、目立たないフッター領域にリンクとバージョンを格納する。 |

## §3 インフラ逼迫・制限の見せ方（深掘り）

* **パターン A：アカウント内メーター（Dropbox 型）**
  * アカウント設定画面の目立つ位置に「使用容量：45%（256KiB / 512KiB）」のようなバーを出す。

* **パターン B：リソース画面のみ（ルーム行にインライン配置）**
  * ルーム一覧のカードや、進行表の編集画面（テーブル）のステータスバーに直接残量を出す。

* **パターン C：グローバルバナーのみ（障害・メンテ）**
  * アプリケーションのヘッダー最上部に、赤または黄色の全幅バナーを割り込ませる。

* **パターン D：外部 Status ページのみ**
  * アプリ内には何も出さず、`status.sugudasu.com` などの外部サービスに障害情報を集約する。

### Sync β 推奨パターン：【パターン B と C のハイブリッド配置】

**理由および他パターンの却下理由：**

Syncのデータ制約である「512KiB上限」や「保持期限（`retain_until`）」は、アカウント全体ではなく「ルーム（イベント）ごと」に独立して機能する事実である。そのため、アカウントページに一括メーターを載せるパターン Aは、マルチイベント運用時に整合性が崩れ、幹事に無用なストレスを与えるため却下する。

* **容量逼迫・保持期限（制限）：** ルーム一覧の各行、および進行表の編集画面内（**パターン B**）に配置する。幹事が「今まさにテーブルをいじっている現場」で初めて意味を持つ情報だからである。
* **プラットフォーム障害・同期遅延（インフラ）：** アプリ上部のトースト/グローバルバナー（**パターン C**）および外部Status（**パターン D**）を併用する。設定画面を開かないと障害が分からない設計は、サポート問い合わせの爆発を招くため却下する。

## §6 アカウントに絶対載せないリスト

提督方針の「事実中心・見せすぎない」を徹底するため、以下の項目はアカウント設定のUIコードから物理的に排除（または別画面へ完全隔離）する。

1. **`auth.users.id` (UUIDの文字列)**：開発者用デバッグ情報であり、ユーザーには意味不明なため。
2. **`stripe_customer_id`**：漏洩リスクを高めるだけで、画面に載せるUX上の動機がない。
3. **累計PV数 / アクセスヒットカウンター**：幹事はタイムラインの同期成否のみに関心があり、統計は不要。
4. **過去のログインIPアドレス履歴**：Enterprise向けセキュリティ製品ではないため、過剰情報。
5. **「無料（trial）プラン」という文言の強調バナー**：アップセルを急ぐ安易なデザインを排除、静けさを維持。
6. **Stripeのダミー請求書一覧 / 領収書ボタン**：β期間中は決済が走らないため、UIの枠すら作らない。
7. **「データ保持期間は最大7日間です」という仕様の解説テキスト**：内規を説教しない。日付だけをルームに出す。
8. **アカウント全体の「合計使用容量」メーター（0.5MB / 10MB等）**：容量はルーム単一に帰属するため。
9. **「チームメンバーを招待」ボタン（未実装機能のグレーアウト）**：S4+の機能をβから見せて期待値を無駄に上げない。
10. **「API制限まであと○○回」という開発者向けレートリミット表示**：インフラの都合をユーザーに押し付けない。
11. **パスワードの暗号化方式やソルトの技術的説明**：セキュリティアピールはノイズとなる。
