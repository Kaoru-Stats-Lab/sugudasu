## H2 構成

### 1. AI の使い分け — 企画と実装を混ぜない

**書くこと**

| 役割 | 担当 | やること | やらないこと |
|------|------|----------|--------------|
| 企画・表 | Gemini 等 | テーマ一覧・スケジュール・比較表 | **本文・コードの遂行** |
| 実装・突合 | Cursor / Claude | HTML/JS · changelog 照合 · テスト | 誇大コピー |
| 短文化・批判 | 別セッション | タイトル案・冗長削除 | 事実の追加 |

- Zenn #1（invoice）は **Cursor 側で執筆**、企画表だけ Gemini — 具体例として1段落
- 「全部 AI に書かせる」とハルシネーション・未実装の宣伝が増える
- **Vibe Coding** は使わない（読者に伝える必要なし · 自戒）

**避ける:** 各社の内部プロンプト・リーク URL · モデル名の盛り

---

### 2. Cloudflare Pages を選んだ理由 — サーバー代より「触らない」

**書くこと**

- 静的 HTML + `build-pages.mjs` で `dist/` 生成 · Git push で配信
- 見積ツールは **ブラウザ内 JSON** · 会員 DB なし（#1 と接続可 · 繰り返しは短く）
- 月額を抑えるより、**ユーザーデータを預けない** 設計が先
- Pages Functions は使わない（現状）— 複雑さとコストを避けた

**技術用語（Zenn 向け · 補足付き）**

- **静的サイト生成（SSG）** — ビルド時に HTML を焼く。リクエストごとにサーバーで組み立てない
- **クライアントサイド** — ブラウザの JavaScript が入力を処理

**スクショ候補:** hub · `updates.html` · Cloudflare ダッシュボード（ドメインのみ · 秘密なし）

---

### 3. Form → changelog の速度 — フィードバックを製品に戻す

**書くこと**

- `updates.html` + フォーム → GAS → `data/changelog.json` → 再ビルド
- 例: 見積→請求転用の案内追加（2026-06-17）— **ユーザー要望が changelog に載る** 流れ
- PR TIMES は見送り、**自分で記事（Zenn）+ X** に寄せた判断（1〜2 文 · 誇らない）
- 修正のたびに長いデプロイ手順を踏まないよう、ビルドスクリプトを1本化

**読者への価値:** 個人開発でも「改善ログを公開する」だけで信頼が積み上がる

## まとめ（約 150 字）

AI は代筆者ではなく、**役割が違う複数の編集者** だと捉えると使いやすい。静的配信とブラウザ内処理でコストとプライバシーを守り、フォーム経由で changelog を回す。SUGUDASU は完成品ではなく、**実務で使いながら直している** ポータルです。
