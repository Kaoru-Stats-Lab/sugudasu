## 現状（事実 · 調査前に固定）

| 項目 | 現状 |
|------|------|
| ホスト | **sugudasu.com（core）** · 登録不要 · 画像/見積をサーバーへ送らない宣伝 |
| 状態 | **1 ブラウザ内メモリのみ** |
| 投票 | UI は **`あなたの名前` 1 名分**だけカード投票可能 |
| Reveal | ローカル Map の隠匿→一括公開。**端末横断はない** |
| コピー | 「Revealは同時公開のみ」— **思想は同時公開、実装は単機** |

**ユーザー痛:** 同時公開の価値は「アンカリング回避」。それには **各人が伏せたまま点数を寄せ、ホスト（またはルーム）が同時に開く** 必要がある。

## Cursor 向け短縮ガードレール（調査後の実装フェーズ用 · 暫定）

調査 AI の出力5が正本になったら差し替える。それまでの **仮ルール**:

### DO

- 単機モード（現行）を壊さない。多人は明示的オプトイン（「ルームを作る」等）。
- 同期ロジックは `planning-poker-engine.js` の外側（例: `planning-poker-sync.js`）に隔離する。
- Reveal 前: UI は「投票済み / 未投票」のみ。カード数値はローカル本人以外に描画しない。
- ユーザー向け文案は実装と一致させる（未配線なら「同時公開」を多人と誤解させない）。

### DON'T

- service role / Supabase 秘密を core `assets/` や Pages 公開物に入れない。
- 「E2Eだから絶対読めない」と書かない（鍵共有・ホスト改ざん・デブツールの限界を無視しない）。
- Sticky Room の盤面モデルを無理に Planning Poker に流用し、見積セマンティクス（Story 単位・一斉 Reveal）を壊さない。
- URL に生の全票を平文クエリで載せない（案1採用時は短寿命・断片・難読化の最低ラインを仕様に書く）。

### ファイル境界（想定）

| 触ってよい | 触ってはいけない（調査完了前） |
|------------|--------------------------------|
| `assets/planning-poker-*.js` · `tools/planning-poker.html` · FAQ | Sync 課金・`sync_rooms` schema の場当たり変更 |
| `data/roadmap.json` · 本プロンプト | 「登録不要」グローバル約束の勝手な撤回 |
