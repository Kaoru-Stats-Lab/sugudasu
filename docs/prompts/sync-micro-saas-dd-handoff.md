## A. デューデリ結論（前チャットで途中まで整理した内容の完成版）

### A-1. 元案の構造評価

元案は「海外マイクロSaaS × 日本ローカル摩擦 × Sync 勝ち筋」の型は良い。ただし次が弱い。

| 問題 | 内容 |
|------|------|
| **旗艦との衝突** | Sync 正本は **Schedule / timeline-sync**。5ドメイン全部を並列旗艦にすると S1 未完のまま散漫になる |
| **「非送信」の誤用** | Sync は**送信する**ライン。コアの信頼を「非送信のまま API ホスティング」と書くと **STATEMENTS と矛盾** |
| **競合の取り違え** | Pastebin ≠ 社内テンプレ同期。Invoice Ninja ≠ 電帳法ストレージ。1Password ≠ 抽選名簿 |
| **法務・コンプラ過大** | 電帳法「検索要件の受け皿」は **税理士・スキャナ保存・真実性**まで踏み込むとミニストレージでは足りない |
| **コモディティ警告の放置** | Text ハブ · API モックは AI / 既存無料で代替されやすい — 課金フックが弱い |
| **コア接続の危険** | invoice 最下部に「クラウド保存」は良いが、**コア UI を汚さない**・**オプトイン明示**が必須（既方針と整合） |

### A-2. ドメイン別ジャッジ（要約）

| # | 案 | 判定 | 一言 |
|---|-----|------|------|
| 1 Text ハブ | **PARK / Tier C** | AI 代替高 · Pastebin 系は差別化薄い。コア `report`/`sns` のコピーで足りる場面が多い |
| 2 セキュア API モック | **HOLD · 縮小のみ可** | 「非送信ベースの API」は矛盾。やるなら **有期限・暗号化・日本リージョン明示の共有ビン**に落とす。Mockaroo 代替は情シス向けだが開発コスト大 |
| 3 電帳法ミニストレージ | **条件付き GO（最有力だが法務ゲート）** | 日本摩擦は本物。ただし「電帳法対応」断定は NG。**検索可能な請求履歴アーカイブ（参考・専門家確認）**に言い換え必須。無期限保存は Sync 方針と衝突 → **有期限 + エクスポート義務** |
| 4 画像アセットドライブ | **PARK** | Dropbox / Google Drive / LINE WORKS と正面衝突。stamp 連携だけでは弱い |
| 5 名簿・鍵箱 | **縮小 GO（イベント Room 拡張）** | パスワード管理は 1Password 領域で OUT。**fair-draw / group-split の当日 Room 同期**だけなら timeline-sync と同型で勝ち筋あり |

### A-3. 優先順位の修正案（元案 Top2 への反論込み）

| 順位 | 元案 | 修正案 | 理由 |
|------|------|--------|------|
| **P0（今）** | — | **timeline-sync / Schedule の製品 E2E 完了** | 旗艦未完了のまま新ドメインは禁止 |
| **P1（次の Sync 楔）** | 電帳法ストレージ | **請求・領収の「履歴ルーム」**（電帳法**断定なし** · 日付/取引先/金額インデックス · 有期限 · PDF+メタ JSON） | 日本摩擦は強いが法務表現を制御 |
| **P2** | API モック | **イベント当日 Room（名簿・進行・受付ステータス）** | 既存 Sync アーキテクチャ再利用 · fair-draw 接続 |
| **PARK** | Text / Assets / 汎用鍵箱 | 同上 | コモディティ or 既存 SaaS |

**元案第2優先（API モック）を落とす理由:**  
コアの売りは「送らない」。Sync で API 公開すると信頼ストーリーが分裂する。情シスが欲しいのは「海外に送らない」であり、それは **日本リージョンの有期限共有**で足り、Mockaroo 互換 API は後回し。
