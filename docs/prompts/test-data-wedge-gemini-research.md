# Gemini依頼: 閉域テストデータ — 楔（Wedge）選定リサーチ

**用途:** SUGUDASU `test-data` の **最初の1楔** を決める（業種横断の絞り込み）  
**前提ドキュメント:** [`docs/notes/TEST_DATA_CLOSED_ENV_RESEARCH.md`](../notes/TEST_DATA_CLOSED_ENV_RESEARCH.md)

## 選定方針（提督メモ）

- **金融だけに固定しない。** 閉域でテストデータが欲しい業種は幅広い。
- ただし **SUGUDASUのような新興・無料・ブラウザ単体サービスを受け入れられそうな楔** から入る。
- エンタープライズTDM（SnowFakery / Delphix / 社内基盤）が既に入っている層は **優先度を下げる**。

## Gemini への依頼文（コピペ用）

```text
あなたは、日本のB2Bプロダクト立ち上げに詳しいUXリサーチャー兼セキュリティ寄りのITコンサルタントです。
目的は、新興Webツール「SUGUDASU テストデータ」（ブラウザ内だけでダミーCSVを生成・外部送信なし）の
**最初の市場楔（Wedge）を1つに絞る** ための判断材料を作ることです。

【プロダクト概要】
- 名称: SUGUDASU テストデータ（test-data）
- 形態: 無料 · 登録不要 · 静的HTML + ブラウザJS · データは端末内のみ
- 現状MVP: 顧客マスタ（10列）· 取引明細（6列）· 100/500/5000件 · シード再現 · 地雷データ5% · CSV
- 差別化の芯: 「外部サイトからCSVを落とさない」「情シスに非送信を説明しやすい」「Excel手打ち卒業」
- 既存ツール群: 請求書 · 全角半角整え · マスク · シフト · 班分け 等（事務・幹事・非エンジニア寄り）
- やらない: 本番DBマスキング · 100万件負荷 · 実マイナンバー/実口座 · 50型フィールドビルダー全面競合

【競合（参考）】
- Handy Dev Tools: ブラウザ完結・50型・日本語・最大1万件・SQL出力あり
- SnowFakery: YAML · CLI · SQL/JSON · エンジニア向け
- Mockaroo 等: クラウド · 無料枠制限

【リサーチの主軸 — 最重要】
「閉域でテストデータが欲しい」業種は多いが、**SUGUDASUのような新興サービスを実際に使ってもらえそうな楔**は限られる。
以下の **受け入れやすさ（Adoption Readiness）** を軸に、候補業種・セグメントを評価・ランキングせよ。

### Adoption Readiness の評価軸（各5点満点 · 合計25点）

1. **導入障壁の低さ**
   - 個人・チーム単位で試せるか（情シス承認・ベンダー白名单・入札が不要か）
   - インストール不要・ブラウザだけで足りるか
   - 無料で試せるか

2. **意思決定者と利用者の近さ**
   - 困っている人（テスター・マニュアル担当・受託SI）が自分でURLを開けるか
   - 情シスが「外部送信なし」で通しやすいか

3. **痛みの切実さ（今日困っているか）**
   - Excel手打ち · 怪しいCSV DL · 本番コピー違反 のどれが日常か
   - 件数は100〜5000で足りる現場か

4. **SUGUDASU適合（既存ポジションとの一致）**
   - 非IT・事務・幹事・マニュアル文化と重なるか
   - mask / normalize / invoice 等への導線が自然か

5. **競合空白（Handy/SnowFakeryで十分でない理由）**
   - 「非送信の説明」「地雷」「シード」「業種プリセット」のどれが刺さるか
   - エンタープライズTDMが届いていない下位レイヤーか

【評価対象セグメント（最低これらを含め、追加可）】
- 金融: 受託SIの開発・テスト · マニュアル担当（行内本部は除外寄りで評価）
- 人材・給与・派遣・労務: 社内情シス少数 · 受託ベンダー
- 医療: 病院ではなく **医療機器・SIベンダー** の受託開発
- 官公庁・自治体: ベンダー側 / 自治体の小規模情シス
- 製造・物流: 工場LAN · 生産管理の社内担当
- 小売・FC: レガシーPOS周り · 本部少数情シス
- 建設: 現場監督 · 受託SI · マニュアル（工程表文化）
- 法律・会計事務所: 小規模 · 守秘義務強
- 教育: 塾 · 社内LMS · 学校ICT担当
- 汎用: フリーランスエンジニア · 個人開発者 · スタートアップ（閉域ではないが比較用）

【制約・分類コード（回答で使うこと）】
- C1 インターネット遮断 / C2 外部ファイル持ち込み禁止 / C3 本番コピー禁止
- C4 クラウドSaaS禁止 / C5 再現性要求 / C6 顧客閉域の中のベンダー

【依頼アウトプット — 必須】

## 1) エグゼクティブサマリー（5行以内）
- 推奨楔 **1位のみ** を明示（2位3位は補足）
- なぜ新興SUGUDASUが受け入れられそうか一言

## 2) セグメント比較表
| 順位 | セグメント | Adoption合計 /25 | 閉域痛 | 必須データ（列イメージ） | 最大のネック | 新興ツール受け入れ | GO/HOLD/OUT |

## 3) 推奨楔1位の深掘り（1セグメントのみ）
- ペルソナ2名（具体: 役割 · 組織規模 · ITリテラシー）
- あるあるシーン3つ（「いつ · 誰が · 何に困る」）
- 今の代替手段（Excel / ネットDL / 本番コピー / 社内ツール）と不満
- **MVPに足すプリセット1種** の提案（列名15個以内 · 日本語ヘッダー）
- 情シスへの説明で通る一言 / 通らない一言
- note・社内チャットで刺さる見出し案3つ

## 4) 捨てる楔（OUT）— 最大3つ
- 閉域ニーズはあっても **新興SUGUDASUが入れない理由** を明記

## 5) 不確実性と追加インタビュー
- 仮説の弱点3つ
- 現場に聞くべき質問10（はい/いいえで答えられるもの多め）

## 6) v0.2機能の優先（楔1位向け）
| 機能 | Must/Should/Could | 理由 |
（例: SQL出力 · 顧客+取引関連 · 社員マスタ 等）

【禁止】
- 「全部の業種に対応すべき」という結論
- 実在の個人情報・口座・マイナンバーの生成推奨
- SUGUDASUをエンタープライズTDMの代替と断定
- 根拠のない市場規模の断定（推定するなら仮説と信頼度を分ける）

【トーン】
- 日本の実務寄り · 表多用
- 各セグメントに **具体的な現場シーン** を必ず1つ以上
- 最後に「提督が明日から取るべき1アクション」1つ

【文字数】
本文は 4,000〜6,000字程度（表は別カウント可）
```
