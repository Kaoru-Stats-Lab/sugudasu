## あなたの作業手順（Thinking 推奨）

### Phase 1 — 監査（出力に含める）

1. INV ごとに「この変更で危ない点」を列挙
2. 変更候補の **呼び出し元** をすべて洗い出す（import · export · HTML onclick · test）
3. 公開 API（export 関数・シグネチャ）を変えずに済むか判定
4. 50行超・複数ファイル必要なら **タスク分割案** を先に提示

### Phase 2 — 設計図

- テンプレ `IMPLEMENTATION_PACKET.example.md` と **同じ見出し構造** で出力
- パケット ID: `PKG-{REFタスクID}-{nn}`
- **変更種別** を必ず指定: A/B/C/D/E/F（Playbook §5）

| 種別 | 意味 |
|------|------|
| A | エンジン JS のみ |
| B | エンジン + app UI |
| C | tools/*.html |
| D | tool-registry.json / nav |
| E | sugudasu-shell.js · sugudasu.css 等共有 |
| F | デプロイ設定 |

### Phase 3 — 各変更の必須記載

- **変更前（関数全文）** — 現行コードと文字列一致すること
- **変更後（関数全文）** — Cursor がコピペ実装できる完成形
- **呼び出し元表** — 影響「なし」も明記
- **実行後ゲート** — コピペ可能な bash ブロック

```bash
cd C:\asl_dev\sugudasu
npm run test:{tool}
# 種別に応じて追加
npm run validate:tool-naming
npm run build:pages
```

## 禁止（設計図に入れてはいけない）

- リネーム · 横断共通化 · 依存パッケージ追加
- 「ついでに」別ファイルの改善
- 行番号だけの Diff（関数全文必須）
- テスト期待値の変更（INV 更新が先 · 別タスク）
- INV に無い挙動変更

---

## BLOCKED — Gemini / 提督へ返す条件

次のいずれかなら **実装パケットを出さず** BLOCKED 報告のみ:

- INV を満たしながら安全にリファクタできない
- ソースが不足し呼び出し元が特定できない
- 1タスクに収まらず、分割案も提督承認が必要
- 解釈が2通り以上ある

```markdown
## BLOCKED — {タスクID}（GLM）

- **理由:**
- **必要な追加情報:**
- **推奨:** {タスク縮小 / テスト追加先行 / INV 見直し}
```
