**用途:** `docs/PR_TIMES_LAUNCH_2026.md` §2 入稿原稿の **戦略・フック・リスクの構造分析のみ**  
**やらない:** 原稿リライト · 礼賛 · 新機能の追加 · 入稿確定（Claude/Cursor + 提督）

**前提:** Grok Pass1〜4 済（推敲ログは `PR_TIMES_LAUNCH_2026.md` §6）

## 使い方

1. 下記 **「Gemini への依頼文」** を丸ごとコピー
2. 続けて **`PR_TIMES_LAUNCH_2026.md` の §2 本文**（■リリース概要〜問い合わせまで）を貼る
3. （任意）Grok Pass1 の記者スキップ表を末尾に貼る — 重複分析の抑制用
4. 出力は **指定フォーマットのみ** を Claude に渡して採否判断

## Gemini への依頼文（コピペ用）

```text
あなたは日本のIT・ガジェット・フリーランス系メディアに精通したメディア戦略アナリストです。
礼賛・応援・総評の前置きは不要です。「素晴らしい」「画期的」「革命的」は禁止です。

【タスク】
添付の PR TIMES 入稿原稿（Grok推敲反映済）について、**メタ解析のみ**行ってください。
原稿の書き直し・推敲・リライトは一切しないでください。

【分析の目的】
PR TIMES 配信後、テックライター・ガジェットブロガーが記事化するかどうかを、
構造（フック・ポジション・リスク）から冷徹に評価する。

---

### 1. 五メタフック採点

以下5項目について、各 **1〜5点** と **根拠1行**（原稿のどの要素か）を表にする。

| フック | 定義 |
|--------|------|
| 社会的背景・時流 | インボイス・フリーランス増・クラウド疲れ等との接続 |
| 逆張り・カウンター | 登録不要・ローカル処理・非送信という「常識への逆」 |
| 数字・タイパの明快さ | 読者ベネフィットがタイトル・リードで伝わるか |
| 開発者ストーリー | 非エンジニア×AI×個人開発×低コスト運用の新鮮さ |
| すぐ試せる拡散性 | URL1本で体験・Xシェア・記事読者の2次拡散 |

**採点基準（厳しめ）**
- 5 = 記事化の主軸にできる
- 3 = 補助材料程度
- 1 = ほぼ効かない / 飽和 / 根拠薄弱

---

### 2. ポジショニング

- **奪うべきカテゴリ名**（1行・造語可。例: No-Account Fast Tool）
- **記事化しやすい読者像**（1行。例: 月数枚しか請求書を出さないフリーランス）
- **記事化しにくい読者像**（1行）

---

### 3. 文脈案の適合度

次の3シナリオそれぞれについて、**現原稿との適合度 1〜5** と **1行理由** のみ。

- 案① 時流×カウンター（インボイス・登録不要・非送信）
- 案② タイパ×幹事・店長（割り勘・シフト・名もなき事務）
- 案③ 開発者ストーリー×UX偏愛（個人開発・AI・静的配信）

**第一弾PR TIMESに最適なのはどれか** — 1つ選び、理由3行以内（礼賛禁止）。

---

### 4. 記者向け見出し

記者がそのまま転用しうる見出しを **3案**（各 **全角40字以内**）。
PR原稿のタイトルと **言い回しを変えて** 出すこと。

---

### 5. 弱点と補強（表のみ）

| 弱点（記者・読者の疑念） | 深刻度 高/中/低 | 原稿に既にある対処 | まだ足りない1行（新事実の捏造禁止） |

Grokが指摘しがちな観点を必ず1行以上含めること:
- 個人開発の小規模感
- 類似無料ツールとの差
- 利用実績・数字の欠如
- 非送信主張の検証方法
- インボイス・税計算の責任範囲

---

### 6. タイトル評価（現行1案のみ）

現タイトル:
「登録不要・ローカル処理のインボイス対応請求書ツール集「SUGUDASU（すぐだす）」を個人開発で正式公開」

- HEAT 1〜5
- 想定メディア（ガジェット / フリーランス / 個人開発 から最大2つ）
- スルーされやすい弱点1行
- **文言変更案は1行まで**（全面リライト禁止）

---

### 7. 入稿前チェック（Yes/No）

各項目を Yes / No / 要確認 で答える。理由は各1行まで。

- [ ] 誇大表現（商用品質・100%・完全無料 forever）がない
- [ ] インボイス・税務の責任範囲が適切に限定されている
- [ ] 非送信・ローカル処理の説明が検証可能な言い方になっている
- [ ] 個人開発・小規模でも記事化の「角度」が1つ以上ある
- [ ] 競合名を出さずに差別化が伝わる
- [ ] PR色（宣伝文）が強すぎずニュース調に近い

---

### 8. 出力形式（厳守）

1. 五メタフック採点表
2. ポジショニング（3行）
3. 文脈案適合度表 + 第一弾推奨
4. 見出し3案
5. 弱点と補強表
6. タイトル評価
7. 入稿前チェック表

**禁止出力:** 原稿全文の推敲版 · 段落の書き直し · 挨拶 · 総評の礼賛 · 新機能・新URLの追加

---
```

## 出力の受け渡し

Geminiの表を `PR_TIMES_LAUNCH_2026.md` §6 末尾に「Gemini §6 ログ」として追記するか、チャットで Claude に渡す。

**採用ルール**

| Gemini出力 | 扱い |
|------------|------|
| 弱点表の「まだ足りない」 | 原稿に1行追記できるものだけ採用 |
| 見出し3案 | §1 候補と突合。良ければ1つ追加 |
| タイトル変更案1行 | HEAT↑かつ誇大でなければ検討 |
| 書き直し段落 | **すべて破棄** |
