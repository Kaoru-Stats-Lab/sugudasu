# 紙媒体デザイナー向けLP — リサーチ評価シート（採用/保留/却下）

**用途:** Gemini等のリサーチ出力を、実装判断に落とすための判定テンプレート  
**対象:** 紙媒体寄りデザイン会社向け「機能美 × スケジュール管理」仮説  
**判定:** `採用` / `保留` / `却下`（必ず理由を1行添える）

---

## 0) 判定ルール（先に固定）

- `採用`: 現場語で刺さり、MVP実装に直結し、誇大表現がない
- `保留`: 仮説として有望だが、追加インタビューか定量確認が必要
- `却下`: ペルソナ不一致、IT語過多、高機能化前提、または根拠不足

**重要:** 「面白い」より「現場で使われる」を優先する。

## 6) MVP機能判定（Must/Should/Could）

| 機能候補 | 区分 | 判定 | 理由 |
|----------|------|------|------|

| 引き継ぎメモ | Should | 採用/保留/却下 |  |
| 通知連携（LINE/メール） | Should | 採用/保留/却下 |  |
| 高度分析ダッシュボード | Could | 採用/保留/却下 |  |
| 複雑な権限設計 | Could | 採用/保留/却下 |  |

**重くなる境界（ここを超えると却下候補）**

- 設定項目が多く初期セットアップに30分以上かかる
- 主要操作が3手以上必要（登録→更新→共有の導線が長い）
- 用語説明がないと使えない
