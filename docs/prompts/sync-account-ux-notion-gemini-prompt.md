## 1. なぜ Notion か（リサーチの文脈）

### 1-A. SUGUDASU Sync のペルソナ（Notion とのシナジー）

| 像 | 行動 | Notion との接点 |
|----|------|-----------------|
| **イベント幹事** | 進行表 · 当日差し込み · 複数関係者への共有 | ドキュメント感 · 表編集 · 共有リンクに慣れている |
| **研修・社内イベント担当** | 稟議 · 事前準備 · 当日オペ | Notion / Excel からの移行候補 |
| **個人 PM・フリーランス** | 工程表 · タイムライン | Notion データベース・カレンダー利用者 |

**仮説:** Sync の **アカウント周り UI** は、幹事が既に慣れた **Notion の「静かな設定画面・余白・階層」** に寄せると学習コストが下がる。

### 1-B. SUGUDASU Sync の制約（変更しない · リサーチ結果はこの枠内で適用）

| 項目 | 方針 |
|------|------|
| **製品** | `sync.sugudasu.com` — イベント進行のクラウド同期（コア `sugudasu.com` は **登録不要** · 別ライン） |
| **ログイン** | **メール + パスワード**（マジックリンク単独ログインは **不採用**） |
| **OAuth** | Google 等は **後追い可**（β では未必須） |
| **内部 ID** | `auth.users.id`（UUID）— **エンドユーザーに UUID は見せない** |
| **メール** | ログイン ID · 表示 · 将来 Stripe 連絡先 — **課金キーではない** |
| **β期間** | **決済 API（Stripe 本番）は導入しない** — アカウント画面に **請求・プラン・カード UI は載せない** |
| **アカウント設定（予定）** | メール表示/変更 · パスワード変更 · ログアウト · **退会（削除）** |
| **デザイン** | Notion の **余白・ヘアライン・設定の階層感** は借りる · **色は Notion HEX をそのまま使わない**（`#0075de` 等禁止）· マップは `--sg-primary` = blue-600 系 |
| **表示方針** | grace 日数・quota 内訳など **運営内規を FAQ 化しない** — 事実（メール・保持期限日付）中心 |

**実装予定画面（マップ対象）**

- 未ログイン: ログイン · 新規登録 · パスワード忘れ · リカバリ（`/timeline/app/` 内パネル）
- ログイン後: ルーム一覧 · **アカウント** パネル（同一 SPA 内 or `/timeline/app/account`）
- **β に無いもの:** ワークスペース切替 · メンバー招待 · 請求 · プラン変更

## 2. リサーチ依頼（Notion 中心）

### 2-1. Notion — 調査してほしい画面（各画面を独立セクションで）

以下 **すべて** について、**画面上の UI 要素を箇条書き**（上から下の順）で列挙し、**primary CTA のラベル・配置**・**secondary リンク**・**エラー表示の位置** も書くこと。

1. **新規登録（Sign up）**
   - メール入力 · パスワード · 「Continue with Google」等 OAuth の有無と配置
   - 利用規約チェックボックスの有無
   - 「すでにアカウントがあります」導線

2. **ログイン（Log in）**
   - メール/パスワード vs SSO の優先順位
   - 「パスワードを忘れた」リンクの位置
   - ログイン状態保持（Remember me）の有無

3. **メール確認（Verify email）** — 初回登録後
   - 待機画面のコピー · 再送ボタン · アプリ内ブロックの有無

4. **パスワードリセット（Forgot password）**
   - 入力フィールド数 · 送信後の確認画面 · 戻る導線

5. **パスワード再設定（Reset password）** — メールリンク着地

6. **設定の情報設計（Settings IA）**
   - 左ナビ or モーダルか · **My account** / **Workspace** / **Members** / **Security** の分割
   - **個人アカウント** と **ワークスペース** の ID が UI 上どう分かれているか

7. **プロフィール / アカウント（個人）**
   - 表示名 · アイコン · メールアドレスの編集 UX（インライン vs 別画面）
   - メール変更時の **再認証** や **確認メール** の見せ方

8. **セキュリティ（Security）**
   - パスワード変更 · 2FA · セッション一覧 · ログアウト全端末
   - 各項目の行 UI（リスト + 右 chevron 等）

9. **退会・データ削除**
   - Notion に相当するフローがあるか · 確認ステップ数 · タイプ入力（DELETE 等）

10. **エラー・バリデーション**
    - メール形式エラー · パスワード弱い · 列挙対策（「送信しました」統一）の見え方

### 2-2. Notion について答えてほしい問い

- Notion はエンドユーザーに **ユーザー ID / UUID** を見せるか？ 見せるならどこか？
- **ワークスペース ID** と **ユーザー ID** をユーザーが意識する場面はあるか？
- 幹事1人製品の Sync に **ワークスペース切替 UI を借りるべきか** — 賛否と理由
- Notion の **「Settings & members」** から **個人アカウントだけ** 抜き出すと、最小構成は何要素か？
