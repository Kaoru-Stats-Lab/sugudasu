# ADR-004

Badge System（Hub カードバッジ）

| 項目 | 値 |
|------|-----|
| **Status** | Accepted |
| **Date** | 2026-07-20 |
| **Scope** | `hub-cards.json` の badges（status / spec / popular）の **意味と表示優先度** |
| **Related** | [ADR-001](ADR-001-top-page-ia.md) · [`../design/badge-guideline.md`](../design/badge-guideline.md) · `HUB_IA_REFRESH_V2.md` |

---

## Background

なぜ議論したか

カード上に NEW / Beta / 正式 / 完全ローカル / PC推奨 / 人気 が並ぶと、**すべてが同重要に見え、情報階層が消える**。

「人気」をバッジで強調すると、検索主導（ADR-001 / 003）と衝突し、押し売りになる。

Top IA で **何を強く見せ、何を弱く見せ、何を出さないか**を決めた。

---

## Problem

何が問題だったか

1. status（成熟度）と spec（実行条件）と popular（編集部おすすめ）が **同じ視覚重量**になりがち。
2. 「人気」バッジは **誘導**であり、ユーザーの課題マッチより先に目に入る。
3. バッジを増やしたくなる（無料 · 登録不要 · 新機能…）が、カードがステッカーボードになる。

---

## Options

検討案

| ID | 案 | 概要 |
|----|-----|------|
| A | **階層化** — status 強調 · spec 弱 · カード上 popular 非表示 | 人気は「実務で人気」セクション側 |
| B | **全部同等表示** | 情報量最大 |
| C | **人気だけ強調** | 誘導最大化 |
| D | **バッジ全廃** | テキストのみ |

---

## Decision

採用案

**案 A — 情報階層付きバッジ**。

### 目的

- ユーザーが **信頼・前提条件**を一瞬で読む。
- **売り込み**をしない。
- カードサイズを変えずに階層だけ整える。

### 情報階層（強い → 弱い → 出さない）

| 優先 | 種類 | 値 | 表示ラベル例 | 扱い |
|------|------|-----|--------------|------|
| **1（強調）** | **status** | `new` / `beta` / `ga` | NEW / Beta / 正式 | 成熟度。主バッジ |
| **2（弱）** | **spec** | `local` / `pc` | 完全ローカル / PC推奨 | 実行前提。弱表示（小さく · 低コントラスト可） |
| **出さない（カード上）** | **popular** | boolean | （人気） | カード上バッジは **非表示**。人気は一覧セクション「実務で人気」等で表現 |

### 採用バッジの意味

#### status

| 値 | 意味（要約） |
|----|----------------|
| NEW | 最近公開 · まだ慣れていないユーザー向けの注意も含む |
| Beta | 試験公開 · 壊れる・変わる可能性 |
| 正式（ga） | 通常利用の想定 |

#### spec

| 値 | 意味（要約） |
|----|----------------|
| 完全ローカル | 当該ツールの処理がブラウザ完結であることの表示（思想の可視化） |
| PC推奨 | 重い処理等でスマホ非推奨の明示 |

### 理由

- status は「使ってよいか」の判断に直結する。
- spec は必要だが、毎回叫ぶとノイズ。弱表示で十分。
- 人気をカードに貼ると検索・JTBD より先に目が行く → **非採用（カード上）**。

---

## Rejected

却下案 · 理由

| 案 | 理由 |
|----|------|
| **カード上「人気」バッジ表示** | 押し売り。検索主導と矛盾。一覧性を壊す |
| **案 B（全部同等）** | 階層が消え、何も読まれない |
| **案 C（人気強調）** | Discover 化。Constitution 違反 |
| **「無料」「登録不要」を全カードにバッジ化** | サイト全体の約束であり、カードごと繰り返すとノイズ（サイトクロム / statements 側） |
| **色だけで意味を混在**（status と spec を同色） | 誤読。modifier を分ける（実装は CSS · 本 ADR は意味層） |
| **案 D（全廃）** | Beta / PC推奨が伝わらず、期待値ミスマッチが増える |

---

## Future Review

再検討条件

1. **status 値の追加**（例: deprecated）が必要になったとき — 本 ADR を更新してから UI に足す
2. **100 ツール超**でカードがさらに密になったとき — spec をアイコンのみ等へさらに弱めてよいか再評価
3. **解析で「人気セクション」自体が害**と分かったとき — セクション側の人気提示も見直し（カードバッジ復活は原則しない）
4. **アクセシビリティ要件**で色以外の区別が不足と分かったとき — テキスト併記ルールを強化

---

## Constitution

SUGUDASUが絶対守る思想

| 思想 | バッジでの意味 |
|------|----------------|
| **引き算** | バッジを増やさない。人気ステッカーを貼らない |
| **3分で終わる** | 読むコストの高いバッジ群にしない |
| **検索中心** | 人気バッジで検索結果の意味を歪めない |
| **ブラウザ完結** | 「完全ローカル」を嘘のマーケバッジにしない（事実と一致するときだけ） |
| **完全ローカル思想** | spec は約束の可視化であり、誇張の道具ではない |
