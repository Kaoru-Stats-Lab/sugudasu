# ADR-002

Card Writing（Hub カード説明文）

| 項目 | 値 |
|------|-----|
| **Status** | Accepted |
| **Date** | 2026-07-20 |
| **Scope** | Hub カードの `conceptName` · `hub-cards.json` の `blurb`（description） |
| **Related** | [ADR-001](ADR-001-top-page-ia.md) · [ADR-003](ADR-003-search-ux.md) · [`TOOL_CARD_WRITING_GUIDELINE.md`](../notes/TOOL_CARD_WRITING_GUIDELINE.md) · [`../design/card-guideline.md`](../design/card-guideline.md) |

---

## Background

なぜ議論したか

Hub カードは SEO 記事ではなく、**一覧を見た瞬間に自分の課題と一致する**ための UI である。

複数 AI・複数人が文言を書くと、次が混ざる。

- 機能カタログ調（「〜できます」）
- 誇張・性能アピール（「高速」「AIで」）
- SEO キーワード羅列
- UI 操作説明（「ドラッグして」）

Top IA と同時に、**JTBD（Jobs To Be Done）2文固定**を意思決定として残す。

実装時の詳細チェック表の正本は `docs/notes/TOOL_CARD_WRITING_GUIDELINE.md`。本 ADR は「なぜその型か」の意思決定層。

---

## Problem

何が問題だったか

1. カードが「何ができるか」ばかり書き、**いつ使うか**が伝わらない。
2. SEO のために conceptName / blurb を検索語だらけにすると、**ブランド用語と命名 3 層が崩れる**。
3. 誇張・保証できない仕様語が混ざると、SUGUDASU の約束（引き算・ブラウザ完結）と矛盾する。
4. AI ごとに文体がブレると、Hub 一覧の信頼感が落ちる。

---

## Options

検討案

| ID | 案 | 概要 |
|----|-----|------|
| A | **JTBD 2文固定** | 「○○したい時に。」「○○します。」· SEO は検索辞書側 |
| B | **機能一文 + キーワード** | 「PDF画像抽出・埋め込み画像・ZIP」型 |
| C | **マーケティング短コピー** | 「かんたん！速い！無料！」型 |
| D | **長文説明** | カード内に使い方・注意事項まで書く |

---

## Decision

採用案

**案 A — JTBD を最優先した 2文固定**。

### JTBD を最優先

カードは検索キーワード UI ではない。**利用シーン（いつ使うか）→ 行為（何をするか）**の順で書く。

### SEO は検索辞書で吸収

- Hub 表示の conceptName / blurb は **ユーザー向け一般名称と JTBD** に最適化する。
- 言い回し・誤変換・英語別名は `synonyms.json` · `search-dictionary/{toolId}.json` · 検索バンドル側で吸収する（[ADR-003](ADR-003-search-ux.md)）。
- **表示名を SEO 用に書き換えない。**

### ブランド用語統一

| NG | OK |
|----|-----|
| 写真 | 画像 |
| 文書・文章（テキスト用途なら） | テキスト |
| 二次元コード | QRコード |
| スクショ（カード本文） | 画像（シーンに応じて） |
| HP | Webサイト |

命名 3 層（id / conceptName / productName）は `TOOL_NAMING_AGENT_PLAYBOOK` に従う。カード見出しは **conceptName**（だいたい 2〜8 字の一般名称）。

### 説明文構造（必須）

```
○○したい時に。
○○します。
```

| 文 | 役割 | 書いてよいこと | 書いてはいけないこと |
|----|------|----------------|----------------------|
| 1文目 | いつ使うか | ユーザーの状況 | 機能自慢・性能 |
| 2文目 | 何をするか | ツールの行為 | UI操作・実装技術・誇張 |

長さの目安: description 全体で **40〜70 文字程度**。常体禁止。必ず上記の結び。

### 具体例

**良い**

```
PDFの中から画像だけを取り出したい時に。
画質を落とさず抽出します。
```

```
画像形式を変更したい時に。
WebPをJPGへ変換します。
```

```
QRコードをパソコンで読み取りたい時に。
カメラなしでファイルから読み取ります。
```

**悪い（採用しない型）**

```
AIで高速に議事録を自動作成。
```

→ 利用シーンなし · AI / 高速 / 自動。

```
ドラッグ＆ドロップで画像をアップロード。
```

→ UI 説明。

```
CSV JSON Markdown HTML XML 対応の万能変換。
```

→ SEO 羅列。

### 禁止事項（要約）

1. 誇張（高速 · 最強 · 簡単 · 一瞬 · 最高 · 誰でも）
2. 保証できない仕様語（AIで · 自動で · おすすめ · 公平に · 完全に · 高精度 · 正確に）※仕様として保証できる場合のみ例外検討
3. UI 説明（クリックして · ドラッグして · ボタンを押して）
4. 実装説明（JSON · Canvas · WASM · ブラウザAPI）
5. SEO キーワードの羅列
6. 商標依存（シャチハタ · Photoshop · Excel互換 等）→ 一般名称へ

詳細チェックリスト → [`../design/card-guideline.md`](../design/card-guideline.md) · 実装正本 → `docs/notes/TOOL_CARD_WRITING_GUIDELINE.md`。

---

## Rejected

却下案 · 理由

| 案 | 理由 |
|----|------|
| **案 B（機能一文 + SEO 羅列）** | 一覧で「自分ごと」にならない。検索は辞書の仕事 |
| **案 C（マーケ短コピー）** | 引き算・信頼・3分思想と矛盾。誇張が入りやすい |
| **案 D（長文）** | カードは 2 行固定想定。詳細は Product / FAQ / ガイドへ |
| **conceptName を英単語やシステム名にする** | 一瞬で理解できない。命名 Playbook 違反 |
| **blurb に「人気」「おすすめ」を書く** | 押し売り。バッジ・人気リストの責務と重複し、検索主導を壊す |

## Future Review

再検討条件

1. **カードの CTR / 選択率が著しく低い**（解析方針内）とき — 文型は維持しつつ例文・長さを調整してよいか再評価
2. **多言語 Hub** を本気でやるとき — 2文型の他言語写像を別 ADR で定義
3. **検証スクリプトが文型を機械検査**できるようになったとき — ガイドラインと validator の責務境界を更新
4. **100 ツール超**でカード密度が変わったとき — 1文短縮案を検討してよいが、JTBD 優先は壊さない

---

## Constitution

SUGUDASUが絶対守る思想

| 思想 | カード文言での意味 |
|------|-------------------|
| **引き算** | 長く説明しない。誇張しない。UI 操作を書かない |
| **3分で終わる** | 「今の自分の用事」が 1 秒で分かること |
| **検索中心** | カードは検索結果の理解を助ける。SEO のゴミ捨て場にしない |
| **ブラウザ完結** | 「アップロードしてクラウドで」等の誤解を誘う表現をしない |
| **完全ローカル思想** | 保証できない「AI / 自動 / 完全」で安心感を売らない |
